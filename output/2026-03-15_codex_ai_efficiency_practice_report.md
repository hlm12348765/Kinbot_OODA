## 1. 汇报摘要

2026 年 3 月 8 日到 2026 年 3 月 15 日，我们围绕 `Kinbot` 家庭室内移动交互机器人，开展了一次以 `Codex` 为核心生产力工具的多线程架构设计实践。目标不是“写几份概念文档”，而是把一个原本容易在多人讨论中发散的机器人产品，收敛成可进入 `IPD / PDCP` 评审的系统架构基线、总体方案基线和技术研判基线。

本次实践的直接结果是：

- 形成了覆盖 `需求输入 -> 架构原则 -> P1 系统架构 -> P2 技术可行性 -> 研究专题 -> 团队规划` 的文档体系。
- 在当前分支上沉淀了近一周约 `50` 次提交。
- 当前 `docs/` 目录下已有 `56` 份 Markdown 文档，其中：
  - `02_p1_architecture` 下 `14` 份
  - `03_p2_feasibility` 下 `9` 份
- 形成了“主架构线程 + 专项技术线程 + 外部评审线程 + Linear 协作”的工作方式。

需要强调的是，这次实践尚未进入编码阶段，但已经把“后续研发如何组织、模块如何并行、哪些接口已经稳定、哪些技术路线需要继续验证”这些关键问题提前压实。这正是 AI 在系统设计阶段最有价值的地方。

## 2. 案例说明

### 2.1 应用开发目标

本项目面向家庭室内场景，目标产品是 `Kinbot` 家庭机器人系统。其核心不是单机玩具，也不是单点算法验证平台，而是一个完整的产品系统，包括：

- 机器人本体
- 穿戴设备协同
- 智能家居协同
- 家属 App
- 云服务
- 后台人工服务
- 外部生态接入

项目的核心目标有三条：

1. 为用户提供多模态交互陪伴服务
2. 为用户提供健康管理服务
3. 为用户提供安全保障服务

从阶段定位上看，本次工作聚焦在：

- 产品需求基本完成后的系统架构设计
- `PDCP` 节点的系统架构评审准备
- 总体方案和模块方案下发基线形成
- 技术路线、成本、功耗、服务闭环等关键约束的前置研判

### 2.2 核心功能与架构范围

围绕上述目标，本次通过 Codex 线程群逐步冻结的核心功能与架构范围包括：

- 多尺度动态 `OODA` 架构
- `World State` 统一状态平面
- 决策状态机与异常/故障枚举
- 安全、合规、授权前置门控
- 健康事件管线与升级链路
- 陪伴交互策略与长期记忆治理
- 家属 App、云服务、后台运营坐席最小闭环
- 储药与室内递送要求
- 软硬件选型矩阵、成本结构、功耗预算
- `VLN`、`UWB`、纯视觉夜间闭环等专项研究输入
- 基于架构反推的开发团队规划

### 2.3 完成情况

当前完成情况可以概括为“系统架构已完整建立，总体方案已形成主线基线，技术研判已覆盖关键高风险项”。

具体完成结果如下：

1. 已形成正式的系统架构主线
   - 产出总体架构、`PDCP` 系统架构评审包、多尺度动态 `OODA`、模块边界、世界状态、状态机、安全授权接口等主文档。

2. 已完成从架构到技术可行性的第一轮转译
   - 产出总体方案与模块方案下发基线、样机到量产预备能力缺口、工程化与 `NPI` 基线、软硬件选型、成本与功耗基线等文档。

3. 已建立专项技术研究与多线程协作模式
   - `VLN` 技术专家线程负责语义导航相关前瞻判断。
   - 主架构线程负责系统边界、接口、阶段门和主线文档冻结。
   - 外部 `Claude` 评审文档被纳入架构复核输入。

4. 已形成面向后续研发组织的团队规划基线
   - 团队不再按“算法组 / 前端组 / 后端组 / 硬件组”粗切，而是按系统工作包和跨域闭环来组织。

5. 已建立结构化协作机制
   - 用户需求输入在 `input/00_requirements/00_user_requirements_input.md` 中滚动维护。
   - 架构决策、开放问题、工作流、文档索引保持持续回写。
   - 关键架构工作项同步到 `Linear`。

当前尚未完成的是：

- 各模块的详细架构与方案设计尚未全部下发并评审完成
- 编码、联调、系统验证、试点等后续环节尚未开始
- 部分高风险专项仍处于“研究输入 / 观察线 / 待样品验证”状态

因此，本次 AI 实践的准确定位应当是：

> 用 Codex 把原本高度发散、跨专业、跨角色的系统设计问题，压缩成了一个可以进入 `PDCP` 评审并支撑后续模块并行设计的完整架构资产库。

## 3. 对话记录：关键交互过程节选

以下内容不是完整聊天记录，而是基于实际交互整理出的关键过程，用于体现问题拆解与解决逻辑。

### 3.1 阶段一：从“分析 codebase”进入架构主线

**用户起点**

- 先要求分析现有 codebase 的设计意图，不修改文件。
- 随后明确提出最终目标：规划整机软硬件工作流、持续提问澄清、记录关键信息、必要时联网了解技术进展。

**Codex 的拆解方式**

- 先识别这不是普通代码仓，而是“架构定义仓”。
- 把目标转成一套可持续推进机制：`WORKFLOW`、`DECISION_LOG`、阶段性问题澄清。

**形成的结果**

- 不再把工作理解为“写一份总体架构就结束”，而是开始建立持续演进的架构治理流程。

### 3.2 阶段二：把发散需求压成结构化输入

**用户交互方式**

- 用户不再零散回复，而是把回答持续写入 `REQUIREMENTS` 的 `Step1 ~ Step35`。
- 每轮要求 Codex 读取对应 `Step`，继续推进并回写文档。

**Codex 的拆解方式**

- 把需求澄清分成高价值批次，而不是一次性问完。
- 将问题按高风险主线组织：
  - 健康管理
  - 陪伴交互
  - 安全保障
  - 硬件与成本
  - 生命周期与团队

**形成的结果**

- 用户输入逐轮结构化，Codex 输出也逐轮冻结，避免了架构讨论反复来回推倒。

### 3.3 阶段三：从单环 OODA 升级为多尺度动态 OODA

**关键分歧**

- 经典单环 `OODA` 不足以描述机器人在家庭中同时面对安全、陪伴、健康和服务协同的多尺度行为。

**Codex 的拆解方式**

- 提出多尺度、并发、可中断、可动态调度的 `OODA` 基线。
- 引入 `R1-R4` 子环和 `OODA Scale Scheduler`。
- 将 `Orient` 升级为“情境理解 + 尺度选择”层。

**用户审阅结果**

- 接受 `R1-R4`
- 接受 `R4` 进入一代正式架构
- 接受 `OODA Scale Scheduler` 升级为一级能力

**形成的结果**

- `OODA` 不再是口号，而变成了后续状态机、接口、验证项都要遵守的正式方法论。

### 3.4 阶段四：从软件架构视角补齐产品实体架构视角

**用户指出的问题**

- 原来的系统架构更像运行时软件架构，不足以代表软硬一体机器人产品。

**Codex 的拆解方式**

- 把 `PDCP` 系统架构评审包改成“双视角基线”：
  - 产品实体架构视图
  - 运行时功能架构视图
- 显式加入机器人本体实体域和 `Body Capability Contract`

**形成的结果**

- 架构不再只讲算法和模块，而是把头部、躯干、底盘、本体能力边界一并纳入评审基线。

### 3.5 阶段五：技术可行性不再只看算法，而是看 BOM、功耗与产品感

**关键问题**

- 用户明确要求整机物料成本 `6000 到 8000 元`
- 同时要求整机与服务组合仍然要显得“聪明、温暖、精致”，能支撑高端价格带

**Codex 的拆解方式**

- 将技术评估从单纯模型/芯片选择，升级为联合约束：
  - `BOM`
  - 售价
  - 功耗
  - 高端产品感
- 形成软硬件选型矩阵、成本结构、功耗预算三份联动文档

**形成的结果**

- 技术路线讨论从“哪个芯片更强”变成“哪个方案在体验、成本、热设计、交付和产品感之间更平衡”。

### 3.6 阶段六：多线程协作开始显著提升效率

**协作模式**

- 主架构线程：负责系统边界、主文档、评审节奏
- `VLN` 专家线程：负责语义导航、前瞻技术判断
- 候选人评估线程：基于团队规划文档评估候选人与岗位匹配
- `Claude` 评审线程：提供替代架构提案与复核意见

**Codex 的拆解方式**

- 允许专项线程独立推进，但要求所有影响主线的结论必须回写：
  - 主线文档索引
  - 决策记录
  - `README`
  - `CHANGELOG`
  - `Linear`

**形成的结果**

- AI 协作不再只是“多个模型同时写东西”，而是形成了有主线、有分工、有回写规则的协同系统。

### 3.7 阶段七：从功能闭环进一步前推到夜间视觉闭环

**新增场景**

- 在纯视觉路线下，夜间不开灯场景成为关键风险点。

**Codex 的拆解方式**

- 单独建立夜间闭环专项文档，讨论：
  - 低照度下的定位稳态问题
  - 家庭空间锚点增强
  - 端侧自学习
  - 授权高质量重建
  - 匿名代理世界

**形成的结果**

- 这说明架构工作已经不是抽象原则层，而是开始直达影响量产可行性的难点场景。

### 3.8 阶段八：仓库治理本身也成为提效对象

**用户发现的问题**

- 文档命名混乱
- 索引不一致
- 分支 / worktree / Obsidian 使用冲突
- Linear issue comment 中旧路径残留

**Codex 的拆解方式**

- 重构目录为按全生命周期分层的结构
- 统一文档头信息
- 修正文档索引、Linear 索引与 comment
- 清理分支与 worktree 使用方式

**形成的结果**

- AI 的价值不仅在于“生成内容”，还在于把知识库、协作链路和版本管理维护成可持续使用的工程资产。

## 4. 问题拆解逻辑总结

本次实践中，Codex 并不是简单“回答问题”，而是持续采用了如下问题拆解逻辑：

1. 先把用户模糊目标转成阶段目标
2. 再把阶段目标拆成高风险问题包
3. 对高风险问题逐轮澄清
4. 每轮形成正式文档，而不是停留在聊天结论
5. 文档进入审阅
6. 审阅结论回写到主线
7. 再进入下一层级问题

换句话说，本次 AI 提效的本质不是“自动写文档”，而是：

> 把原本容易发散的人机讨论过程，强制组织成“输入 -> 分析 -> 提案 -> 审阅 -> 回写 -> 冻结”的工程流程。

## 5. 个人心得

以下部分采用第一人称写法，可直接用于汇报发言。

### 5.1 我对这次实践的最大感受

过去我使用 AI 更多是让它回答问题、补代码、写总结。但这次 Kinbot 的实践让我更清楚地看到，AI 真正大的价值不只是“生成内容”，而是“帮助我持续组织复杂问题”。

这次项目里，Kinbot 不是一个单点功能，而是一个跨产品、算法、硬件、服务、工程、运营、合规的复杂系统。如果只靠人工口头讨论，很容易反复、遗忘、漂移、失焦。Codex 的价值，是把这些讨论持续压成结构化资产，并且逼着整个过程留下可审阅、可追溯、可回写的文档证据。

### 5.2 我认为最有效的使用方式

我认为最有效的方式，不是一次提一个大而空的问题，而是：

- 先明确阶段目标
- 每轮只推进一小批高价值问题
- 要求 AI 产出正式提案，而不是只给口头建议
- 再由我审阅、纠偏、冻结

这种方式的好处是，AI 不再只是“灵感来源”，而是变成了架构推进过程中的持续协作者。

### 5.3 我得到的三个关键经验

**第一，AI 最适合做“高频、连续、需要回写”的脑力工作。**

像需求澄清、架构分解、评审问题整理、索引维护、跨文档一致性检查、工作流编排，这些事情单次看都不难，但持续做很耗人，而且特别容易漏。AI 在这种场景里提效非常明显。

**第二，AI 不是自动正确，必须建立“审阅 + 冻结”机制。**

如果没有阶段审阅，AI 会把很多看似合理但未被确认的内容继续往下游传播。我们这次之所以能比较稳定推进，是因为每一阶段都显式地要求“提交我审查”。这其实和正规研发流程是统一的。

**第三，多线程协作只有在有主线和回写规则时才有效。**

多个 Codex 线程确实能并行提速，但前提是：

- 谁是主线
- 谁是专项
- 哪些结论必须回写
- 哪些文档谁不能改

这些边界必须先明确，否则多线程只会带来更多混乱。

### 5.4 对未来研发工作的启发

我认为未来研发工作的变化，不会只是“工程师多了一个问答工具”，而会是以下三点：

1. 架构设计、需求澄清、评审材料准备这类原本高度依赖资深人员经验的工作，会越来越适合由 AI 参与并提效。
2. 研发组织需要尽早形成“人负责判断，AI 负责组织、展开、回写、检查”的新协作模式。
3. 文档、Issue、代码、评审、研究输入之间的边界会越来越模糊，未来高效团队一定会把这些东西看成一个整体工作流，而不是几个割裂系统。

### 5.5 我的最终判断

这次 Kinbot 的实践说明，AI 在系统架构设计阶段已经可以带来非常实在的效率提升，但前提不是“让 AI 自己跑”，而是“把 AI 纳入严谨的工程流程”。

如果让我用一句话总结这次实践，我会说：

> AI 最有价值的地方，不是替我们做决定，而是让复杂系统设计这件事第一次具备了低摩擦、可追溯、可并行、可持续推进的工作方式。

## 7. 后续建议

如果要把这次实践进一步产品化、制度化，我建议下一步做三件事：

1. 把“四轴评估框架”“阶段审阅模板”“模块方案模板”固化成标准模板。
2. 为跨文档一致性、索引回写、`Linear` 同步建立半自动检查脚本。
3. 在后续编码阶段继续保持同样的协作方式，让 AI 不只服务于架构设计，也服务于实现、验证和试点复盘。
