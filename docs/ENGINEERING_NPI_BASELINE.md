# Kinbot_OODA 工程化与 NPI 准备基线

## 1. 文档目的

本文档用于回答 `KBT-24` 的核心问题：

在 `2026-03-31` 架构冻结之后，Kinbot 要以什么工程化和 `NPI` 基线进入 `P2`，才能在后续 9 个月内收敛到 `2026-12-31` 的量产预备状态。

本文档不是完整项目计划，而是为 `P2 关键技术收敛与工程化方案` 提供第一版硬约束、阶段门和交付物清单。

## 2. 当前输入

本轮基于以下已确认输入：

1. 目标系统是机器人整机产品及服务，穿戴、智能家居、App、云服务和后台运营属于伴生系统。
2. 当前已有真实样机和粗放 Demo，已验证导航、语音、多模态交互、问诊和紧急给药链路的可行性。
3. 项目目标是在 `2026-12-31` 达到量产预备状态，并以 `2027-01` 作为 `MVP` 验证窗口。
4. `KBT-18` 已将样机到量产预备的主要差距收敛为 7 个能力缺口域。
5. 当前正式方法论为 `R1` 到 `R4` 四类多尺度动态 `OODA` 子环，`OODA Scale Scheduler` 已进入一级架构能力。

## 3. `P2` 阶段要完成的 3 个跃迁

`P2` 不是继续证明“功能能做”，而是完成以下 3 个跃迁：

1. 从 `架构冻结` 跃迁到 `工程冻结`

说明：把架构文档转成硬件、软件、伴生系统、验证和组织协同的执行基线。

1. 从 `验证平台` 跃迁到 `Alpha 平台`

说明：不再以“先堆配置再验证”为默认方式，而是开始围绕量产导向的结构、成本、功耗、可靠性和装配性收敛。

1. 从 `零散 Demo` 跃迁到 `可验证交付包`

说明：每条关键链路都要有接口、版本、验证项、责任人和阶段门，而不是只靠口头共识推进。

## 4. 七个工程化与 `NPI` 基线包

为避免工程化清单继续碎片化，当前建议按 7 个基线包推进。


| 基线包                | 要冻结什么                            | 本阶段核心输出                  | 如果不冻结会怎样                |
| ------------------ | -------------------------------- | ------------------------ | ----------------------- |
| `E1 产品配置与版本基线`     | 一代 SKU 边界、功能配置、版本切分、默认开关         | `PRD` 冻结面、版本矩阵、功能开关表     | 后续硬件、软件和试点目标持续漂移        |
| `E2 整机平台与关键器件基线`   | 底盘、传感器、算力、屏幕、麦阵、储物仓、供电与热设计边界     | 整机配置表、`BOM` 边界、关键器件风险表   | 重量、成本、功耗和装配风险失控         |
| `E3 机器人软件与运行栈基线`   | 端侧运行栈、`R1-R4` 映射、关键接口、日志与故障策略    | 软件分层图、接口清单、版本策略、故障分级     | Alpha 联调和验证缺乏稳定载体       |
| `E4 伴生系统与服务运营基线`   | 家属 App、云服务、人工坐席、第三方网关的最小交付面      | 伴生系统 `MVP` 包、角色权限表、服务链路表 | 试点、售后、人工接力和异常升级无法闭环     |
| `E5 验证与数据闭环基线`     | 功能验证、稳定性验证、故障注入、试点数据回流规则         | 验证矩阵、试验台架、数据口径和缺陷闭环机制    | 后续 `EVT/DVT` 只能凭感觉推进    |
| `E6 供应链、制造与测试导入基线` | 关键供应商、打样节奏、`DFM/DFT`、工装夹具和测试治具方向 | 供应风险表、试产前置项、制造测试清单       | 到 `NPI` 阶段才暴露装配、测试和交付问题 |
| `E7 项目节奏与阶段门基线`    | `P2` 到 `P5` 的阶段门、负责人、评审节奏和退出条件   | 阶段门清单、里程碑定义、跨团队协作面       | 研发活动很多，但没有真正收敛到同一个目标    |


## 5. 七个基线包的详细要求

### 5.1 `E1 产品配置与版本基线`

必须冻结：

1. 一代产品的核心价值边界
2. `R1` 到 `R4` 在首版的最低交付面
3. 机器人本体与伴生系统的功能归属
4. 默认授权开关和默认策略
5. `Alpha / Beta / 量产预备` 三个阶段的功能差分

当前建议：

- 先冻结“必须有、应该有、可以预留”的版本矩阵。
- 所有“以后可能会做”的能力都必须挂在明确阶段，不允许悬空。

### 5.2 `E2 整机平台与关键器件基线`

必须冻结：

1. 传感器最小集合与可裁剪项
2. 端侧算力平台候选和板级获取策略
3. 底盘、屏幕、麦阵和储物仓的结构边界
4. 电池、功耗、热设计和续航预算
5. 重量、尺寸和 `BOM` 目标

当前建议：

- 把“储物仓能力保留、机构实现重构”写入硬件收敛基线。
- 把“整机外观形态重构”视为既定工程动作，而不是视觉设计附属项。
- 所有关键器件都要有 `A/B` 路线，不把项目卡死在单点供应上。

### 5.3 `E3 机器人软件与运行栈基线`

必须冻结：

1. `R1` 反射环的底盘安全栈边界
2. `R2` 执行环的身份、导航执行和交互监督边界
3. `R3` 任务环的健康、用药和升级编排边界
4. `R4` 关系与服务环的长期记忆与服务编排边界
5. `OODA Scale Scheduler` 的输入、输出和调度责任
6. 统一日志、审计和故障分级

当前建议：

- 不追求一开始统一所有实现框架，但必须统一关键接口和日志口径。
- `R1` 与底盘安全有关的路径不得依赖云和高层模型兜底。

### 5.4 `E4 伴生系统与服务运营基线`

必须冻结：

1. 家属 App 的首版最小交付面
2. 云服务的账号、配置、审计和第三方网关能力
3. 后台运营坐席的接入、转接和审计能力
4. 高风险场景的远程确认与人工接力链
5. `OTA`、售后和试点运营的基础支撑面

当前建议：

- 把 `E4` 视为和本体同级的工程化对象，而不是“后面再补”的互联网配套。
- 任何试点方案都必须回答：异常发生后，App、云、坐席分别如何承接。
- `E4` 的优先级体现在“先冻结接口、职责和交付面”，不体现在“让伴生系统反向决定本体器件和结构实现”。
- 本体硬件与伴生系统之间要以稳定接口解耦，避免为了某个 App 或云侧实现细节反向锁死整机路线。

### 5.5 `E5 验证与数据闭环基线`

必须冻结：

1. 功能正确性验证项
2. 稳定性与长稳验证项
3. 故障注入和降级验证项
4. 场景回放、日志抓取和问题复现机制
5. 试点数据回流和缺陷优先级机制

当前建议：

- 从 `P2` 开始就建立“问题单 -> 复现 -> 修复 -> 回归 -> 关闭”的固定闭环。
- 所有高风险链路都必须有故障注入验证，而不是只做 happy path。

### 5.6 `E6 供应链、制造与测试导入基线`

必须冻结：

1. 关键器件和供应商风险分级
2. 结构装配风险、工艺风险和调试风险
3. `DFM / DFA / DFT` 的第一版约束
4. 工装、夹具、校准和测试治具方向
5. 小批量试点前的制造与测试前置项

当前建议：

- 先抓“装不装得起来、测不测得出来、修不修得回去”这三件事。
- 不把制造测试问题拖到 `EVT` 之后再集中暴露。

### 5.7 `E7 项目节奏与阶段门基线`

必须冻结：

1. `G2` 技术路线门的判定条件
2. `Alpha / EVT` 进入条件
3. `Beta / DVT` 进入条件
4. `量产预备` 的过程性指标
5. 各基线包的主责人与评审频率

当前建议：

- `P2` 不是“继续研究”，而是“把研究结果转成阶段门”。
- 每个阶段门都必须能回答“冻结什么、允许漂移什么、用什么事实放行”。

## 6. `G2` 技术路线门的第一版判定条件

当前建议把 `G2` 收敛为 7 个放行条件：

1. 一代功能配置和版本矩阵已冻结
2. 整机平台主路线和关键器件备选路线已冻结
3. 伴生系统最小交付面已冻结
4. `R1-R4` 与关键接口映射已冻结
5. 验证矩阵和试验台架计划已冻结
6. 供应链与制造风险表已冻结
7. `Alpha / EVT` 进入条件和责任人已冻结

说明：

- 只要上述 7 项里有任意一项长期漂移，`P2` 就不算真正完成。

## 7. 当前默认的优先级顺序

在 `KBT-18` 已形成初版结论后，当前建议按以下顺序推进 `KBT-24`：

1. `E4 伴生系统与服务运营基线`
2. `E2 整机平台与关键器件基线`
3. `E5 验证与数据闭环基线`
4. `E6 供应链、制造与测试导入基线`
5. `E3 机器人软件与运行栈基线`
6. `E1 产品配置与版本基线`
7. `E7 项目节奏与阶段门基线`

说明：

- 这里把 `E4` 放在最前，不是因为它比本体“更重要”，而是因为它最容易被忽视，但一旦缺失就会直接拖垮试点、售后、授权治理和人工接力。
- `E2` 和 `E5/E6` 紧随其后，是因为重量、成本、结构、验证和制造问题都具有明显的后置爆炸特征。

## 8. 对后续工作的直接输入

这份文档当前直接给后续 4 个工作项提供输入：

1. `KBT-11` 软硬件选型矩阵
2. `KBT-12` 量产预备与 `MVP` 验证计划
3. `Alpha / EVT` 进入条件定义
4. 供应链、制造、测试和试点协作面设计

## 9. 当前待你审核的 4 点

当前这份文档是 `KBT-24` 的第一版基线草案，最值得你审核的是：

1. 是否接受按 7 个工程化与 `NPI` 基线包推进，而不是按部门职能推进。
2. 是否接受把 `E4 伴生系统与服务运营基线` 放在 `KBT-24` 的第一优先级。
3. 是否接受把“储物仓机构重构”和“整机外观形态重构”明确列入 `E2` 的既定动作。
4. 是否接受当前给出的 `G2` 七项放行条件。
