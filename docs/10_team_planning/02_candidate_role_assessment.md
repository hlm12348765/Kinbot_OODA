## 3. 候选人评估

### 3.1 黄洁

#### 结论

黄洁与团队规划 **有条件匹配**。建议录用，但不建议按“通用算法工程师”或“当前阶段一级域负责人”引入。

更合适的落位是：

`S4 世界状态与决策团队` 的 **记忆与状态建模专题 Owner / 高级算法工程师**，
并承担与 `S7` 治理观测与数据团队协同的 **行为评测与可解释性规则设计** 职责。

#### 与团队规划岗位的匹配判断

- 与当前最急缺岗位的直接匹配度：`中`
- 与 `S4` 域团队职责边界的匹配度：`高`
- 与 `S7` 评测治理协同职责的匹配度：`中高`
- 是否适合作为当前阶段一级负责人：`否`

判断理由：

1. 她最强的能力是把模糊体验目标压成结构化问题，并建立中间表示、规则链和评价口径，这与 `S4` 的 world state、状态机、任务编排、OODA 调度前置建模逻辑高度一致。
2. 她做过子系统级重构，说明她不是单点调参型工程师，而是能承担专题 owner 职责的人。
3. 她对规则、知识、结构与模型混合治理有成熟倾向，适合补 `S4` 中“记忆 / 事件 / 状态 / 决策解释”这类容易失控但又必须工程化的部分。
4. 她对评测体系、客观指标、可解释性有明显优势，能够为 `S4` 与 `S7` 的验收接口提供统一口径。
5. 但她并不匹配当前最急缺的 `S5`、`S6`、系统集成测试、`NPI` 等 owner 岗；对机器人业务和整机交付链路的理解也不足以支撑她在当前阶段直接担任 `S4` 一级负责人。

#### 拟录用岗位说明

岗位名称建议：

`S4 世界状态与决策团队 - 记忆与状态建模专题 Owner`

岗位定位：

- 角色级别建议为 `高级工程师 / 专题 Owner / 关键二号位`
- 汇报对象应为 `S4 世界状态与决策负责人`
- 工作接口重点连接 `S3` 多模态交互与陪伴团队、`S7` 治理观测与数据团队

核心职责建议：

1. 负责家庭机器人长期记忆、事件、状态、中间表示的结构化建模。
2. 负责规则 + 模型联合决策链路设计，支撑行为选择、上下文解释和部分 OODA 调度输入。
3. 负责记忆质量、行为一致性、决策可解释性相关指标定义，并与 `S7` 协同建立评测方法。
4. 输出本专题的模块架构、接口草案、验收指标与验证方案，满足团队基线对域团队“架构 + 接口 + 验证”的统一要求。
5. 在大模型输出进入系统行为链路前，承担一部分前后处理治理、规则约束和结果解释设计。

不建议赋予的职责：

1. 不建议让她承担当前阶段 `S4` 一级负责人职责。
2. 不建议将她投放为“通用算法补位”或“哪里缺人去哪里”的机动人选。
3. 不建议让她承担以数据采集、设备联调、现场杂务为主的岗位。
4. 不建议把她放到职责漂移严重、长期缺乏验收边界的探索型岗位。

#### 录用价值

如果团队当前缺的是以下能力，她的价值明确：

1. 把“更自然、更像人、更懂你”转成可定义、可评测、可迭代的结构化目标。
2. 补强 `S4` 中记忆、事件、状态、规则-模型混合决策这类系统性薄弱环节。
3. 为 `S4` 与 `S7` 之间建立行为评测、解释口径和验收接口。

#### 主要风险

1. 她对机器人业务、硬件约束和整机交付链路认知仍浅，短期内不适合承担域负责人角色。
2. 她对“问题是否值得做、目标是否清楚”有较高要求，岗位边界不清时容易快速失去投入感。
3. 她存在过度建模风险，必须由强 manager 或强系统 owner 压住收敛节奏和交付边界。
4. 她不适合承担大量脏数据、设备前置、接口救火类工作。

#### 录用建议

建议结论：`建议录用`

前提条件：

1. offer 岗位必须明确写成 `S4` 方向的专题 owner，而不是泛算法岗。
2. 入职后首个课题必须能在 `2-3` 个月内完成边界定义、指标建立和阶段验收。
3. 管理上必须明确她的上游 owner、接口团队和验收方式，避免职责漂移。
