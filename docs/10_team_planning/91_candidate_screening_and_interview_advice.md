## 3. 候选人逐一判断

### 3.1 王世轩

对应岗位：

1. `01` 架构师（高级开发人员）
2. 实质上对应应用后端 / AI 融合架构岗位

简历快照：

1. 本硕背景，硕士来自天津大学。
2. 2021 年起在华为任软件开发高级工程师，主要经历集中在数据工程、BI、自研平台、规则引擎和架构重构。
3. 当前有模块 leader / MDE 经验，具备一定带小团队和架构演进经验。

岗位匹配判断：

1. 与岗位对口的部分：
   - Java 后端和分布式能力较强。
   - 有系统性能优化、模块架构演进、数据平台能力建设经验。
   - 有一定 AI 工程能力，尤其偏数据工程、训练管理、特征工程、作业流编排。
2. 不足的部分：
   - 工作年限略短，严格看仍偏年轻型候选人。
   - 简历里对“大模型应用后端、Agent 编排、多模态 / 语音落地、智能终端”支撑证据不够强。
   - 更像平台 / 数据 / BI 架构型背景，不完全等于家庭机器人应用后端架构 owner。
   - 机器人、智能硬件、多模态交互系统的直接经验缺失。

筛选建议：

1. `B 档，可推进`
2. 建议进入专家一面或 Q1 深聊，不建议直接跳过一面进入终局判断。
3. 如果岗位目标是“能立即独立承担应用后端架构 owner”，他当前证据还不够硬。
4. 如果岗位可以接受“强后端底子 + AI 工程化潜力 + 需要 3 到 6 个月完成场景迁移”，则值得继续看。

主要优势：

1. 后端基础扎实，能讲清系统性能、模块分层、数据架构和效率提升。
2. 有把一次项目沉淀成平台能力的意识。
3. 有团队协作、带小组和推动重构的经历。

主要风险：

1. 与岗位 JD 要求的“AI 能力 + 业务后端 + 多模态 / 语音 / 智能终端”之间仍有距离。
2. 目前更像企业软件 / 数据平台骨干，而不是家庭机器人应用后端架构 owner。
3. CTO 二面要重点防止被“华为 + 架构 + AI 工程化”标签抬高预期。

建议面试链：

1. `Q1 / 初筛`：确认其对机器人、终端、语音、多模态产品的兴趣和迁移动机。
2. `一面`：由应用后端 / 平台负责人重点验证其是否真的具备架构 owner 水平。
3. `CTO 二面`：只在一面结论明确偏正向时再安排。

CTO 二面候选问题（`10` 道供选择）：

1. 技术能力：你过去做过最复杂的后端 / 平台系统是什么？其中你亲自设计和拍板的核心模块有哪些？
2. 技术能力：你在 AI 工程化里，真正落过哪些 `LLM / Agent / 多模态` 能力？哪些是直接面向业务，哪些只是支撑训练和数据流？
3. 系统工程：如果一个“语音请求 -> Agent 编排 -> 云服务 -> App 回执”链路在现场频繁失败，你会怎样设计日志、监控、追踪与排障入口？
4. 架构思考：你过去做的平台架构，如何迁移到一个“机器人本体 + App + 云服务 + AI 能力”的产品系统？哪些思路可以直接复用，哪些必须重做？
5. 架构思考：如果让你从 `0` 到 `1` 设计 Kinbot 应用后端，你会如何拆核心服务边界？你会先固化哪些能力，后延哪些能力？
6. 行业认识：你认为企业数据 / 平台系统和家庭机器人应用后端最大的本质差异是什么？为什么这会影响架构决策？
7. 产品思维：如果让你在 Kinbot 入职 `3` 个月内打通一条“语音 / Agent / App / 云”的主链，你会怎样拆阶段目标？你会优先保证哪一段体验？
8. 协作能力：讲一个你和产品、算法、客户端或运维团队发生明显分歧的例子。你最终怎样推动一个可交付的共同方案？
9. owner 意识：你更像平台型架构师，还是业务型架构师？请拿一个项目证明你不是只负责协调，而是对结果真正负责。
10. 风险与自我认知：如果你来做这个岗位，你目前最不充分的能力是什么？你准备怎样在前 `90` 天补齐它？

综合建议：

1. `建议推进到专家一面`
2. `不建议现在就作为最优先候选人`
3. 如果一面发现他实际更适合平台 / 数据工程岗位，也可以转岗池保留

### 3.2 陈明华

对应岗位：

1. `09` 资深硬件工程师-本体SE

简历快照：

1. 超过 20 年硬件与系统工程经验。
2. 近年在消费级 3D 打印、机器视觉、达闼机器人、优必选等公司担任系统架构专家、硬件高级经理、硬件系统高级经理、硬件专家。
3. 明确做过机器人本体、轮式人形、配送、巡检、双足、核心器件选型、供应商管理、系统方案设计、跨团队拉通。
4. 同时有大公司和机器人创业公司经历，既有技术深度，也有组织和流程经验。

岗位匹配判断：

1. 与岗位强匹配的部分：
   - 做过机器人整机级系统方案设计和硬件架构。
   - 有跨电子、结构、驱动、关节、测试、导航、视觉、语音、运动控制的拉通经验。
   - 有系统经理 / 架构专家背景，符合本体 `SE` 角色定义。
   - 有供应商选择、合作管理、量产导入与流程建设经验。
2. 需要验证的部分：
   - 当前是否仍愿意以“强 owner + 强落地 + 真正下场”的方式承担本体 `SE`，而不是更偏管理与高层架构。
   - 对 Kinbot 当前“轻高端家庭机器人”的成本、形态、组织节奏是否认可。
   - 是否愿意从大公司 / 管理岗位心态切换到更贴近真机落地与早期阶段的打法。

筛选建议：

1. `A 档，强推进`
2. 建议作为本体 `SE` 方向的优先候选人快速推进。
3. 可以直接进入高质量专家深面或与 CTO 的组合面试。

主要优势：

1. 行业相关度高，且不只是单板硬件，而是真做过机器人整机与系统方案。
2. 具备典型本体 `SE` 需要的跨专业拉通能力。
3. 同时具备系统架构、硬件深度、量产经验和团队组织能力。

主要风险：

1. 可能偏高阶，需要确认是否愿意真正扛当前阶段的细节和节奏。
2. 可能对岗位级别、组织位置和授权边界有更高要求。
3. 要验证他是否会成为“高位判断者”而不是“当前阶段的结果 owner”。

建议面试链：

1. `一面`：由本体负责人 / 硬件负责人深挖其近 5 年真实下场程度和技术颗粒度。
2. `CTO 二面`：重点判断角色适配、阶段适配和上手方式。
3. `必要补面`：如需要，可加结构 / 控制接口人补面，确认协作风格。

CTO 二面候选问题（`10` 道供选择）：

1. 技术能力：如果让你担任 Kinbot 本体 `SE`，你会如何定义“本体方案成功”的判据？哪些指标必须在 `EVT` 前就盯住？
2. 技术能力：你过去做过最典型的跨电路、结构、电气、控制冲突是什么？你最终如何裁决，代价是什么？
3. 系统工程：当传感、驱动、电源、热、结构和控制同时互相牵制时，你通常用什么方法把问题拆开，而不是把整机拖进反复返工？
4. 架构思考：你会如何定义本体 `SE` 与硬件负责人、结构负责人、系统软件负责人之间的接口边界，避免多人都负责、最后没人负责？
5. 架构思考：如果当前资源有限，你会优先在哪些模块做“过设计”，在哪些模块强行控成本？你的判断依据是什么？
6. 行业认识：你怎么看家庭机器人和你过去做过的工业 / 商用 / 特种机器人在本体设计上的本质差异？哪些经验不能直接照搬？
7. 产品思维：你如何在“高端产品感、量产收敛、成本约束”三者之间做取舍？如果三者冲突，你默认的优先顺序是什么？
8. 协作能力：当产品、`ID` 或业务侧坚持一个会显著恶化稳定性、装配或成本的方案时，你会怎样推动他们接受工程现实？
9. owner 意识：你愿意下沉到什么颗粒度？你会亲自抓哪些事，不抓哪些事？怎样避免自己变成只做高位判断的人？
10. 风险与自我认知：如果前三个月只允许你抓 `3` 个问题，你会抓什么？你最担心自己在哪类组织或授权条件下发挥不出来？

综合建议：

1. `建议快速推进`
2. `候选人明显符合本体 SE 高优先级补位方向`
3. 是否录用的关键，不是技术真假，而是“角色定位 + 组织适配 + 下场深度”

3. 具备一定运动学仿真能力，对悬挂和机身运动仿真有实际经验。

主要风险：

1. 经验边界明显偏轮式底盘，关节、身体和整机级家用机器人结构经验不足。
2. 历史项目大多到 `EVT / DVT`，量产后段和长期家居体验收敛证据偏弱。
3. 目前更像“底盘结构专项骨干”，而不是“高级结构工程师-传动”所要求的泛结构高级位。

建议面试链：

1. `结构专家一面`：已完成，结论为 `B 档，刚过及格线不多`。
2. `CTO 二面`：仅在岗位当前明确需要轮式底盘 / AGV 结构专项能力时再安排。
3. `后续补面`：如继续推进，建议由本体负责人补问其向家庭机器人整机结构迁移的边界与成本。

2. 技术能力：你在越障、爬坡、重心稳定性这些指标上，过去实际怎么做方案权衡和验证？
3. 系统工程：你提到做过悬挂仿真和机身运动仿真，具体是怎样把仿真结果转成可执行结构改动的？
4. 架构思考：如果让你从底盘结构继续往上接整机结构，你认为最容易出问题的接口在哪里？为什么？
5. 架构思考：过去项目大多到 `EVT / DVT`，如果让你继续向量产收敛推进，你认为自己最需要补的能力是什么？

8. 协作能力：讲一个你和硬件、电气、测试或 `ID` 之间发生明显矛盾的案例。你是怎么把一个局部最优问题拉回整机最优的？
9. owner 意识：如果当前岗位只让你主抓轮式底盘 / `AGV` 结构专项，你会怎样定义自己的结果边界？什么算你交付完成，什么不算？
10. 风险与自我认知：如果入职 Kinbot，你前三个月最该先收敛哪类结构风险？同时你最担心自己在哪类整机结构问题上存在迁移盲区？

综合建议：

1. `建议保留推进，但不作为高优先级泛结构高级位`
2. `如果当前缺口是轮式底盘 / AGV 结构专项，可以继续进入 CTO 二面`
3. `如果当前缺口是家庭机器人整机高级结构位，建议谨慎推进或转入后备池`
