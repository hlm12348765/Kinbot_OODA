## 1. 文档目的

本文用于解决一个现实问题：

当前 CTO 需要对 Kinbot 多个技术方向的候选人做统一筛选与二轮面试，但不可能在结构、电气、操作系统、机器人控制、应用后端、具身智能等所有方向都成为一线专家。

因此，本方案的目标不是让 CTO 取代领域专家，而是建立一套可复用的统一机制，用来回答以下 4 个问题：

1. 这份简历是否值得进入正式面试流程。
2. 领域专家一面已经通过后，CTO 二面还需要判断什么。
3. 不同技术岗位之间，怎样用一套统一标尺比较“是否适合当前 Kinbot 阶段”。
4. 领域专家需要向 CTO 提供什么样的一页纸信息，才能让 CTO 做高质量判断。

## 3. CTO 在招聘流程中的角色定义

CTO 在面试链里的角色，不是“最后一个更懂技术的专家”，而是“最终判断这个人能否提升 Kinbot 产品竞争力的人”。

CTO 重点判断 5 件事：

1. 候选人做过的问题，是否属于产品关键链路，而不是外围执行工作。
2. 候选人是否具备系统观、取舍能力和跨团队翻译能力。
3. 候选人是否有 owner 意识，能对结果负责，而不只是对任务负责。
4. 候选人的真实技术深度，是否和简历包装、岗位级别一致。
5. 候选人是否适配 Kinbot 当前阶段，而不是只在另一个更成熟、更大团队的环境里成立。

CTO 不应该做的事：

1. 不替代结构、电气、操作系统等领域专家做细节技术考核。
2. 不靠自己临场猜测去判断某个细分技术点是否正确。
3. 不因为候选人大厂背景或术语密度高，就降低对结果与闭环的要求。

## 4. 标准招聘流程

建议统一采用以下流程：

1. `HR / 用人经理初筛`：核对基本条件、岗位编号、年限、地点、薪酬、到岗意愿。
2. `Q1 筛选`：根据岗位基线中的 `Q1筛选` 字段决定是否做简短电话筛选。
3. `领域专家一面`：重点验证基础功底、真实做过什么、技术深度、是否能上手。
4. `CTO 二面`：重点验证系统观、owner 意识、取舍能力、阶段适配度和组织上限。
5. `必要时补面`：由相关领域专家或跨团队接口人做针对性校验。
6. `最终评审`：形成“是否推进、适合级别、入职后定位、风险点”四项结论。

推荐分工如下：

| 环节 | 主要负责人 | 重点输出 |
| --- | --- | --- |
| 初筛 | HR / 用人经理 | 是否满足硬门槛 |
| Q1 | HR / 用人经理 / 指定一面官 | 是否值得进入正式一面 |
| 领域专家一面 | 该领域一线负责人 | 技术深度、真实经历、上手风险 |
| CTO 二面 | CTO | 系统观、owner 潜力、岗位与阶段匹配 |
| 最终评审 | CTO + 用人经理 + 领域专家 | 录用建议、级别建议、主要风险 |

## 5. 统一简历筛选逻辑

### 5.1 简历初筛看什么

先看证据，不先看名词。

必看 6 项：

1. 是否做过真实复杂系统，而不是单点模块。
2. 是否在关键链路上承担过明确责任。
3. 是否经历过一个完整闭环，例如方案、样机、联调、试产、量产、上线或大规模运营。
4. 是否能说出量化结果，例如成本、稳定性、效率、良率、时延、功耗、返修率。
5. 是否有跨团队协同证据，而不是只在单点技术内自洽。
6. 是否与目标岗位的“阶段诉求”一致，例如当前 Kinbot 更需要 owner、平台型人才、系统型人才，而不是只会局部实现的人。

### 5.2 简历分级

统一分 4 档：

1. `A 档`：强匹配。核心经历、行业背景、岗位责任、阶段适配度都明显符合，可优先推进。
2. `B 档`：可推进。总体匹配，但有 1 到 2 个关键验证点需要在一面或二面重点确认。
3. `C 档`：保留。存在潜力，但与当前岗位级别或阶段诉求不完全匹配，不建议优先推进。
4. `D 档`：终止。硬门槛不满足，或与岗位真实诉求明显错位。

### 5.3 统一红旗

出现以下情况时，原则上不建议推进：

1. 简历只有职责，没有结果。
2. 只做过大团队内局部执行，没有 owner 证据。
3. 项目经验全部停留在 Demo、论文、原型，没有交付闭环。
4. 只会复述术语，无法说明自己做了什么、为什么这样做。
5. 与目标岗位要求的技术方向或级别明显不匹配。

## 6. CTO 二面统一框架

### 6.1 面试前准备

每位候选人在 CTO 二面前，必须准备 3 份输入：

1. 候选人简历
2. 对应岗位 JD 和必备技能条目
3. 领域专家一页纸标尺

如果没有第 3 份，CTO 原则上不进入正式二面。

### 6.2 建议时长

建议 `45 到 60` 分钟。

### 6.3 建议结构

1. `5 分钟`：请候选人讲“最能代表自己的一段经历”。
2. `10 到 15 分钟`：深挖一个最复杂、最有代表性的项目。
3. `10 到 15 分钟`：追问关键取舍、失败、复盘和跨团队协同。
4. `10 到 15 分钟`：给出 Kinbot 当前场景，判断其阶段适配度和优先级意识。
5. `5 分钟`：确认动机、预期和岗位定位。

### 6.4 CTO 通用问题库

所有技术岗位都可以使用以下问题：

1. 你做过最复杂、最影响产品结果的一个问题是什么？你具体负责什么？
2. 那个问题里最难的约束是什么？你是怎么取舍的？
3. 这个问题更难的是技术本身，还是跨团队协同？为什么？
4. 如果今天重做一次，你会怎么改方案或改架构？
5. 你做的这个东西，最后怎样影响了产品结果？请给量化或可感知结果。
6. 如果把你放到 Kinbot 当前阶段，你前三个月最该先解决什么问题？
7. 你更像一个 specialist，还是一个 owner？请拿过去经历证明。
8. 你过去做过的系统，哪些经验可以迁移到机器人，哪些不能？

## 7. 跨领域共用评分尺

### 7.1 核心评分维度

统一采用 `5` 个核心维度，每项 `1 到 5` 分，总分 `25` 分。

| 维度 | 要看什么 |
| --- | --- |
| 技术真实度与深度 | 是否真做过、是否有硬深度、是否经得起追问 |
| 系统观与取舍能力 | 是否理解上下游约束，是否能做系统级取舍 |
| owner 意识与闭环推进 | 是否能对结果负责，是否做过闭环 |
| 跨团队翻译与协同能力 | 是否能把本领域语言翻译成其他团队可执行约束 |
| 岗位与阶段匹配度 | 是否适合 Kinbot 当前阶段、当前岗位和当前组织缺口 |

### 7.2 分值锚点

每项建议按以下锚点打分：

1. `1 分`：明显不满足，或只能停留在术语层。
2. `2 分`：部分满足，但需要强依赖他人补位。
3. `3 分`：基本满足，可以作为执行骨干使用。
4. `4 分`：明显满足，能作为关键岗位推进。
5. `5 分`：显著超出预期，能拉高团队上限或承担 owner 角色。

### 7.3 总分对应建议

1. `22 到 25`：建议录用，优先推进，适合关键位或 owner 位。
2. `18 到 21`：建议推进，适合骨干位，但需明确风险与补位。
3. `14 到 17`：可保留或降级使用，不建议作为当前最优先岗位。
4. `13` 分及以下：不建议推进。

## 8. 领域专家一页纸标尺

### 8.1 必须包含的内容

每位进入 CTO 二面的候选人，对应领域专家必须提供一页纸，内容不超过一页 A4：

1. 岗位一句话使命
2. 该岗位最重要的 `3` 个 must-have
3. 该岗位最关键的 `3` 个红旗
4. 候选人技术初评结论
5. 候选人最大优势 `2` 条
6. 候选人最大风险 `2` 条
7. 希望 CTO 重点验证的 `3` 个问题

### 8.2 结构领域一页纸标尺

岗位本质：

1. 不只是画图，而是把传动、空间、工艺、装配和试产收敛成整机结构质量。

must-have：

1. 做过机器人或复杂智能硬件结构设计，并经历过样机到量产闭环。
2. 能讲清空间布局、容差、DFM/DFA、装配和可靠性，不只会讲建模。
3. 能与硬件、电气、ID、测试协同收敛问题。

红旗：

1. 只会讲零件，不会讲整机。
2. 只会讲图纸和建模，不会讲试产、装配和问题闭环。
3. 只做过工业设备，不理解消费级产品的精致感、静音和轻量化。

建议 CTO 重点追问：

1. 你做过最难的结构冲突是什么？你如何在空间、强度、工艺和成本之间取舍？
2. 你处理过哪些量产结构问题？例如异响、松动、变形、装配不良、密封不稳。
3. 你如何确保结构方案不只是“能装上”，而是“有高级感、可量产、可靠”？

### 8.3 电气 / 本体 SE 领域一页纸标尺

岗位本质：

1. 不是单板设计，而是从系统角度对本体硬件结果负责。

must-have：

1. 做过整机级或底盘级系统方案 owner。
2. 能处理电源、电机驱动、控制板、接口、电气约束与结构协同。
3. 具备供应商判断、风险识别和工程落地能力。

红旗：

1. 只会做板卡，不会做系统取舍。
2. 只有技术深度，没有跨团队推进能力。
3. 只有概念方案经验，没有样机、试产或量产闭环。

建议 CTO 重点追问：

1. 你怎么定义一个“整机上站得住”的本体方案？
2. 遇到结构、电气、控制目标冲突时，你怎么做决策？
3. 你过去如何缩短验证周期、降低售后风险或提升供应链质量？

### 8.4 操作系统 / 系统软件领域一页纸标尺

岗位本质：

1. 不是写几个驱动，而是把端侧平台稳定性、可观测性和上层承载能力建起来。

must-have：

1. 做过 BSP、驱动、系统优化、故障恢复、升级、日志等平台工作中的至少两类。
2. 能解释系统级疑难问题如何定位和闭环。
3. 能把平台能力和上层算法、应用、设备协同串起来。

红旗：

1. 只会调 API，不理解平台边界。
2. 只讲模块，不讲系统行为和稳定性。
3. 没有 root cause 级别的问题处理经验。

建议 CTO 重点追问：

1. 你处理过最难的系统级稳定性问题是什么？怎么定位？
2. 你做的平台能力怎样直接支撑了上层应用或算法？
3. 你如何建立可观测、可回归、可持续演进的平台基础？

## 9. 面试结论模板

建议所有 CTO 二面都统一输出以下结论：

1. `是否推进`：推进 / 保留 / 终止
2. `建议级别`：与当前岗位级别一致 / 可降一级 / 可升一级
3. `建议定位`：owner / 核心骨干 / 执行骨干 / 暂不匹配
4. `最大优势`：不超过 `2` 条
5. `最大风险`：不超过 `2` 条
6. `若录用，前三个月最该让他解决什么问题`：不超过 `3` 条

## 10. 使用建议

1. CTO 不要追求“自己懂所有方向”，而要追求“自己能稳定判断关键问题是否会被做成结果”。
2. 一面专家负责判断“他会不会”，CTO 二面负责判断“他做的是不是关键问题、能不能把关键问题做成结果”。
3. 当领域明显不熟时，CTO 不能跳过专家一页纸。
4. 如果候选人很强，但岗位不对，不要硬塞到当前岗位；可以转为后续潜在岗位池。
