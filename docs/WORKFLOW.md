# Kinbot_OODA 推进工作流

## 1. 目的

本工作流用于把当前仓库从“总体架构定义”持续推进到“可落地的软硬件方案、MVP 原型和实现计划”。

核心原则：

- 先冻结高代价架构决策，再进入实现细节
- 先定义稳定接口，再绑定具体算法、模型和供应商
- 所有关键假设都要记录为“已确认”或“待确认”，不能隐式漂移
- 对时间敏感、高风险或高采购成本的信息，必须在需要时联网核实

## 2. 协作方式

我作为架构师，按以下节奏与你协作：

1. 我先基于现有信息形成候选方案、问题清单和建议。
2. 只向你提一批高价值问题，避免一次性发散过多问题。
3. 你的回答会被沉淀到 `docs/DECISION_LOG.md`。
4. 每完成一轮关键澄清，我会补充下一层文档，而不是停留在讨论层。
5. 涉及最新芯片、传感器、模型、法规、供应链价格时，我会联网确认后再给建议。

## 3. 工作流阶段

### 阶段 0：目标和边界冻结

目标：

- 明确首发用户是谁
- 明确家庭场景优先级
- 明确健康管理和安全保障的边界
- 明确首发市场和合规范围

输出物：

- 更新后的需求边界
- 利益相关者清单
- 第一版成功标准
- 第一批架构决策记录

### 阶段 1：系统分解

目标：

- 把总体架构拆成可开发的模块边界
- 固定软硬件分层
- 固定端云职责边界

输出物：

- 模块分层与模块边界文档
- 顶层软硬件框图
- 核心运行闭环和时序关系
- 初版接口清单

### 阶段 2：核心状态和接口冻结

目标：

- 定义 `World State Schema`
- 定义 `Task / Action Contract`
- 定义 `Safety / Compliance / Authorization API`
- 定义关键运行时状态

输出物：

- 世界状态与核心数据结构文档
- 决策状态机文档
- 关键接口契约文档
- 审计与隐私治理边界

### 阶段 3：技术路线评估

目标：

- 评估底盘、算力、相机、麦克风、电池、显示、通信等硬件路线
- 评估导航、感知、语音、VLM/LLM、调度框架等软件路线
- 形成买件、自研、集成的边界

输出物：

- 技术选型矩阵
- 关键器件 shortlist
- 风险清单
- 初版 BOM 和算力预算

说明：

- 这一阶段会大量涉及时效性信息，届时我会联网核实最新技术进展、供货状态、价格带和生态成熟度。

### 阶段 4：MVP 定义

目标：

- 冻结 2027 年 1 月 MVP 的必须能力
- 明确哪些能力进入 MVP，哪些延后
- 固定验证指标和验收条件

输出物：

- MVP 功能清单
- MVP 非功能指标
- MVP 验证计划
- MVP 实施里程碑

### 阶段 5：原型实现规划

目标：

- 把架构转成可执行的工程计划
- 固定 bring-up 顺序和集成顺序
- 固定 repo 结构和测试策略

输出物：

- 软件 repo 规划
- 硬件 bring-up 顺序
- 集成测试计划
- 家庭场景试验计划

### 阶段 6：集成与迭代

目标：

- 在真实家庭环境中验证导航、交互、安全和稳定性
- 用观察结果反推架构和产品边界

输出物：

- 试验记录
- 缺陷闭环
- 版本里程碑
- 下一代架构修订建议

## 4. 并行工作流

为了避免“先做完软件再看硬件”或“先买完硬件再想架构”的错误顺序，后续工作按六条并行流推进：

### A. 产品与场景流

- 用户画像
- 家庭场景切片
- 首发功能优先级
- 使用频率和失败容忍度

### B. 安全与合规流

- 伤害场景分析
- 数据隐私边界
- 用户授权机制
- 首发市场监管约束

### C. 硬件平台流

- 底盘和运动形态
- 传感器布局
- 端侧算力
- 电池、散热和结构

### D. 端侧智能流

- 导航与避障
- 感知与身份识别
- 语音交互
- 世界状态更新

### E. 云与应用流

- 联网问答
- App 与远程控制
- 配置中心
- 审计、日志与运维

### F. 验证与量产准备流

- 指标定义
- 测试台架
- 家庭试验
- 供应链风险

## 5. 文档产物顺序

从当前状态出发，建议按以下顺序继续补文档：

1. `docs/DECISION_LOG.md`
2. 模块分层与模块边界
3. 世界状态与核心数据结构
4. 决策状态机
5. 端云职责拆分
6. 技术选型矩阵
7. MVP 定义与验证计划

## 6. 需要联网核实的内容

以下内容天然具有时间敏感性，不能只靠静态经验判断：

- 端侧算力平台的最新代际、价格和功耗
- 深度相机、RGB 相机、麦克风阵列和雷达的供应状态
- 语音、VLM、SLAM、导航和多模态模型的最新可用路线
- 首发市场的隐私、安全和无线通信相关要求
- 第三方云服务、模型服务和医疗相关服务的接入边界

## 7. 当前建议的推进节奏

第一轮：

- 冻结用户、场景、合规和健康边界
- 完成 `DECISION_LOG`
- 输出模块分层与模块边界

第二轮：

- 输出世界状态和决策状态机
- 拉出第一版接口契约

第三轮：

- 在联网调研基础上做技术选型矩阵
- 冻结 MVP 技术路线

## 8. 每轮沟通的标准输出

每轮我都会尽量保持固定结构：

1. 已确认事实
2. 本轮新增判断
3. 需要你回答的问题
4. 我会补充的文档
5. 如果需要，附上联网核实结论和来源
