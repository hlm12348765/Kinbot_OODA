# Kinbot_OODA 推进工作流

## 1. 目的

本工作流用于把当前仓库从“总体架构定义”持续推进到“可落地的软硬件方案、MVP 原型和实现计划”。

全生命周期视角见 [docs/LIFECYCLE_WORKFLOW.md](docs/LIFECYCLE_WORKFLOW.md)。

当前生命周期管理明确参考 `IPD`（集成产品开发）思想。

核心原则：

- 先冻结高代价架构决策，再进入实现细节
- 先定义稳定接口，再绑定具体算法、模型和供应商
- 所有关键假设都要记录为“已确认”或“待确认”，不能隐式漂移
- 对时间敏感、高风险或高采购成本的信息，必须在需要时联网核实
- 项目时间线按 2026 年 1 月 1 日起算，其中 2026 年 1 到 2 月已用于需求收集与 Demo 概念验证
- 当前总目标是 2026 年 12 月 31 日达到量产预备状态，2027 年 1 月完成 MVP 验证窗口
- 产品定义与架构冻结争取在 2026 年 3 月 31 日完成
- 每一阶段都要检查系统各实体元素的组合，判断整机与服务是否仍然呈现“聪明、温暖、精致”的高端产品感，并能支撑 `20000 到 30000 元` 售价区间

## 2. 协作方式

我作为架构师，按以下节奏与你协作：

1. 每轮开始前，我先检查 `docs/REQUIREMENTS.MD` 顶部“对 Codex 的要求”是否有变化。
2. 我会基于现有信息形成候选方案、问题清单和建议。
3. 只向你提一批高价值问题，避免一次性发散过多问题。
4. 你的回答会被沉淀到 `docs/DECISION_LOG.md`。
5. 每完成一轮关键澄清，我会补充下一层文档，而不是停留在讨论层。
6. 涉及最新芯片、传感器、模型、法规、供应链价格时，我会联网确认后再给建议。
7. 每一轮本地文档中的待办和 issue，会同步到 Linear 项目中，保持 repo 文档和协作系统一致。
8. 本地 Markdown 文档和 Linear 项目、里程碑、issue、文档统一用中文维护。
9. 涉及 VLN 路线、导航推理和相关前瞻技术判断时，我会通过独立 Linear issue 与另一个负责 VLN 技术评估的 Codex 线程交叉校验；当前 `VLN` 专项线程的产物已纳入版本控制，后续由当前线程继续作为正式技术输入引用，并在需要时维护 `docs/VLN_ROLE_AND_PLAN.md`。
10. `docs/REQUIREMENTS.MD` 作为用户维护的需求事实源，只由用户修改；我只读取该文件并据此推进其他文档。
11. 当关键信息未澄清但推进不能停时，我会先形成“架构师设想包”，以 `provisional` 方式写入本地文档并同步到 Linear，等待你审核。
12. 当前总体方法论继续以 OODA 为主，并且已经冻结为“多尺度、并发、可中断、可动态调度”的正式基线；后续文档需要显式体现 `R1` 到 `R4` 子环、`R4` 的一代正式地位，以及 `OODA Scale Scheduler` 这一一级能力。
13. 当前置评审 issue 未关闭时，我必须先强提醒你；后续 issue 可以并行起草，但不能在前置 issue 未关闭时被当成最终冻结结果。
14. 对存在明确结构分解的文档，我会优先同步补上直观图示，并把图示视为正式架构产物，而不是可有可无的排版装饰。
15. 每一轮涉及成本、功耗、结构、交互或伴生系统裁剪的评审，我都要同步检查其是否损伤“聪明、温暖、精致”的高端产品感，并记录是否还能支撑 `20000 到 30000 元` 售价区间。
16. 每一轮都要复查 Linear 中仍处于 `In Review` 的 issue 是否与本地文档状态一致；若文档已形成正式基线，则必须及时关闭或重定向，避免工作流滞留。
17. 当我暂停并等待你输入时，必须提出明确的审阅需求，至少指出“请审阅哪个 issue 或哪份文档的哪一段提案”，不能只给模糊的“下一步继续推进”。
18. 允许为后置里程碑提前起草“逆向约束型文档”，用于把量产、发布、运营或验证约束提前反压到前置设计；但这不代表已经越过前置里程碑，当前主优先级必须始终回到尚未关闭的前置里程碑。

## 2.1 前置评审清账规则

为避免“后续文档写了很多，但前置边界还没关掉”导致返工，当前明确采用以下规则：

1. `KBT-5 模块边界` 是 `KBT-6 世界状态结构` 的前置评审项。
2. `KBT-6 世界状态结构` 是 `KBT-7 决策状态机` 的前置评审项。
3. `KBT-7 决策状态机` 是后续接口冻结、工程化基线和技术选型冻结的重要前置项。
4. 后续 issue 可以先起草，但只要前置 issue 未关闭，我就必须把其结论标成 `provisional` 或“待前置关闭后正式冻结”。
5. 每一轮开始时，我都要先检查仍处于 `In Review` 的前置 issue，并强提醒你当前未清账项。
6. 如果后置里程碑中已有提前起草的约束型文档，它们只作为输入存在，不能抢占当前主评审顺序。

当前评审顺序强制收敛为：

1. `KBT-5` 模块边界
2. `KBT-6` 世界状态结构
3. `KBT-7` 决策状态机
4. `KBT-8` 安全 / 合规 / 授权接口
5. 其余功能包、工程化和选型项

## 3. 工作流阶段

### 3.0 当前主线补充：`PDCP` 系统架构评审与总体方案下发

基于最新要求，当前主线进一步明确为：

1. 当前阶段不是量产导入准备阶段，而是“产品需求基本完成后的系统架构设计与技术研判阶段”

## 4. 并行工作流

为了避免“先做完软件再看硬件”或“先买完硬件再想架构”的错误顺序，后续工作按六条并行流推进：

### A. 产品与场景流

- 用户画像
- 家庭场景切片
- 首发功能优先级
- 使用频率和失败容忍度

### B. 安全与合规流

- 伤害场景分析
- 数据隐私边界
- 用户授权机制
- 首发市场监管约束

### C. 硬件平台流

- 底盘和运动形态
- 传感器布局
- 端侧算力
- 电池、散热和结构

### D. 端侧智能流

- 导航与避障
- 感知与身份识别
- 语音交互
- 世界状态更新

### E. 云与应用流

- 联网问答
- App 与远程控制
- 配置中心
- 审计、日志与运维

### F. 验证与量产准备流

- 指标定义
- 测试台架
- 家庭试验
- 供应链风险

### 功能澄清主线

在上述六条并行流之内，当前阶段的功能澄清不再按单一健康主线推进，而按三条功能主线并行推进：

- 健康管理：重点冻结事件管线、数据时效、人工服务升级和外部平台责任边界
- 陪伴交互：重点冻结人设边界、主动发起规则、长期记忆和多角色共享边界
- 安全保障：重点冻结家庭危险源清单、空间分级、降级停机矩阵和异常升级规则

## 9. Linear 同步规则

为了形成持续协作，后续每轮按以下规则同步到 Linear：

1. `docs/WORKFLOW.md` 中的阶段性待办，会映射为项目级 issue 或 milestone。
