## 1. 文档目的

本文档用于把已经明确的系统架构，转成各团队可直接开工的当前唯一开发入口。

它回答的是：

1. 哪些工作包需要被正式承接；
2. 每个工作包负责哪些模块、对象、接口和验证事项；
3. 各工作包按什么顺序评审、在什么条件下启动。

它不替代以下主线文档：

1. [01_overall_architecture.md](../02_p1_architecture/01_overall_architecture.md)：总体架构总图；
2. [03_execution_paradigms_runtime_baseline.md](../02_p1_architecture/03_execution_paradigms_runtime_baseline.md)：运行时语法；
3. [05_world_state_schema.md](../02_p1_architecture/05_world_state_schema.md)：共享状态平面；
4. [07_safety_compliance_authorization_api.md](../02_p1_architecture/07_safety_compliance_authorization_api.md)：安全、合规、授权边界。

## 2. 使用方式

`S1-S7` 是对当前已明确架构的开发承接工作包，不是另一套并列顶层架构。

本文聚焦工作包分工、接口协同和交付要求；需要回看架构事实源时，应返回 `01 / 03 / 05 / 07` 对应主文档。

使用顺序固定为：

1. 先读取总体架构、运行时、状态平面和安全边界；
2. 再按本文找到对应工作包；
3. 每个工作包只回答“谁负责什么、依赖什么、要交什么”，不重复解释整套系统架构。

## 3. 当前已明确的输入

当前各工作包必须共同继承以下输入：

1. 总体架构采用双视角基线：本体 `6` 个实体域 + `头部 / 躯干 / 底盘` 空间承载子视图，以及运行时 `9` 个一级模块。
2. 运行时认知仍保留 `4` 类执行范式：离散决策、连续流式、事件驱动、长周期演化；但当前开发承接只把离散决策与事件驱动作为必交付主链，连续流式与长周期演化只保留研究线 / 接口位。
3. `World State` 采用 `7` 实体目标模型：`Person / CareRelationship / Household / Place / Object / Task / CareEvent`。
4. 动作执行前必须经过安全、合规、授权边界，且高风险动作不能绕过端侧门控。
5. 一代本体主线继续采用“主控 `SoC` + 实时控制 `MCU`”、“`1` 组双目 + `2` 个单目”、“头部多自由度、单麦阵优先、两轮差速稳健底盘”。
6. 一代默认资源线按 `8GB RAM + 64GB Flash` 组织；`12GB + 64GB` 作为边界验证线；`16GB + 64GB` 及以上只作为前瞻验证线或未来 `Pro SKU` 候选。
7. 产品交付对象仍是“机器人本体 + 伴生系统 + 服务”的完整产品系统，而不是单一机器人本体；但当前 `V1` 伴生系统主链只按 `家属 App + 最小云` 承接，人工服务与第三方服务先保留后续适配位。
8. 量产目标、成本约束、隐私边界和“聪明、温暖、精致”的产品感要求，对所有工作包同时生效。

## 4. 开发承接总览

### 4.1 `9` 模块到 `S1-S7` 的映射

| 工作包 | 负责模块 | 说明 |
| --- | --- | --- |
| `S1` | `platform_runtime` + `mobility_navigation` | 本体平台、计算控制、运动与近场安全基础面 |
| `S2` | `human_health_sensing` | 人体感知、健康事件候选与补采基础面 |
| `S3` | `multimodal_interaction` | 多模态交互、陪伴表达与交互体验面 |
| `S4` | `world_state_memory` + `decision_orchestration` | 共享状态平面与离散业务编排面 |
| `S5` | `safety_compliance_authorization` | 安全、合规、授权和动作门控面 |
| `S6` | `companion_service_system` | `家属 App + 最小云` 主链，以及后续服务适配位协同面 |
| `S7` | `observability_data_governance` | 横切观测、审计、数据治理与验证支持面 |

说明：

1. `S7` 是横切工作包，不是新的一级模块。
2. `4` 类执行范式不单独映射为工作包；它们通过 `S1-S7` 的协作落地。

### 4.2 本体实体域 / 空间承载到 `S1-S7` 的映射

| 本体实体域 / 空间承载 | 主责工作包 | 协同工作包 |
| --- | --- | --- |
| 计算与控制核心 | `S1` | `S4` `S5` |
| 移动底盘与运动执行 | `S1` | `S5` |
| 环境与人体感知组件 | `S2` | `S1` `S4` |
| 交互表达组件 | `S3` | `S1` `S4` |
| 电池电源、热与本体安全 | `S1` | `S5` `S7` |
| 储物仓、结构与外观 | `S1` | `S3` `S5` |
| 头部模块 | `S1` `S2` `S3` | `S4` |
| 躯干模块 | `S1` | `S3` `S5` |
| 底盘模块 | `S1` | `S5` |

### 4.3 状态对象、当前主承接范式与关键边界

| 工作包 | 主要对象 | 当前主承接范式 | 关键边界 |
| --- | --- | --- | --- |
| `S1` | `Place` `Object` `Task` 相关执行状态 | 离散决策、事件驱动 | 运动安全、本体资源、端侧实时性；局部连续控制只作为实现约束，不单列为当前交付面 |
| `S2` | `Person` `CareEvent` | 事件驱动、离散决策 | 候选事件只到结构化判断，不直接越过动作门 |
| `S3` | `Person` `Task` 相关交互状态 | 离散决策 | 交互体验不直接替代授权和编排判断；长期个性化只保留接口位 |
| `S4` | `7` 实体全量状态平面 | 离散决策、事件驱动 | 不绕过 `S5`，不代替 `S1/S2/S3/S6` 的实现职责；连续流式与长周期演化只保留兼容接口 |
| `S5` | `CareRelationship` `Task` `CareEvent` 审批相关字段 | 离散决策、事件驱动 | 统一动作门、审计、回退 |
| `S6` | `Household` `CareRelationship` 外部协同字段 | 事件驱动 | 当前只承接 `家属 App + 最小云` 的远程确认与结构化同步；人工服务与第三方只保留后续适配位 |
| `S7` | 审计记录、指标、验证证据 | 横切支持 | 只负责观测和治理，不直接做业务判断 |

## 5. 七个工作包定义

### 5.1 `S1` 本体平台与运动方案

**主责**

- 端侧计算平台、板级资源、运动执行、安全执行链和本体资源预算。

**涉及模块**

- `platform_runtime`
- `mobility_navigation`

**主要对象**

- 本体实体域中的计算与控制核心、移动底盘与运动执行、电池电源与热
- 空间承载中的头部、躯干、底盘

**主要接口**

- 本体能力接口
- 导航与执行状态接口
- 资源状态与故障状态接口

**依赖与边界**

- 读取 `S4` 提供的任务目标和状态摘要
- 所有高风险动作必须经过 `S5`
- 不负责健康事件判断、交互内容生成和外部服务调用

**必须交付物**

1. 平台分层图与板级资源表
2. 端侧资源预算与默认/验证线差异表
3. 本体布局与关键器件落位图
4. 运动与近场安全方案
5. 关键接口说明
6. 风险与验证清单

### 5.2 `S2` 人体感知与健康方案

**主责**

- 人体识别、姿态、生命体征接入、健康事件候选和补采策略。

**涉及模块**

- `human_health_sensing`

**主要对象**

- `Person`
- `CareEvent`
- 与健康相关的 `Object`

**主要接口**

- 感知结果接口
- 健康事件候选接口
- 外设接入与数据新鲜度接口

**依赖与边界**

- 写入 `S4` 的共享状态平面
- 受 `S5` 的隐私与授权规则约束
- 不直接给出最终医疗结论，不直接触发危险动作

**必须交付物**

1. 感知能力范围说明
2. 健康事件候选流图
3. 外设接入和补采方案
4. 数据质量与降级规则
5. 关键接口说明
6. 风险与验证清单

### 5.3 `S3` 多模态交互与陪伴方案

**主责**

- 语音、屏幕、灯光、提示音、交互节奏和陪伴体验表达。

**涉及模块**

- `multimodal_interaction`

**主要对象**

- `Person`
- 与交互相关的 `Task`

**主要接口**

- 交互输入输出接口
- 提示与确认反馈接口
- 交互状态回写接口

**依赖与边界**

- 依赖 `S4` 提供上下文和任务信息
- 依赖 `S5` 提供可自动执行与必须确认的边界
- 不负责最终授权判断，不负责长期记忆主表设计

**必须交付物**

1. 交互方式说明
2. 陪伴体验与低打扰规则
3. 头部一体化约束说明
4. 交互状态与反馈接口说明
5. 风险与验证清单

### 5.4 `S4` 世界状态与决策方案

**主责**

- 统一状态平面、离散业务编排以及跨范式协调输入的组织。

**涉及模块**

- `world_state_memory`
- `decision_orchestration`

**二级责任面**

- `S4A` 状态平面：`7` 实体、快照、状态同步、对象关系
- `S4B` 编排与任务执行：任务推进、离散业务状态、事件触发与执行范式协调输入

**主要对象**

- `Person`
- `CareRelationship`
- `Household`
- `Place`
- `Object`
- `Task`
- `CareEvent`

**主要接口**

- 状态读写接口
- 任务与事件关联接口
- 动作提议与审批前接口

**依赖与边界**

- 从 `S1/S2/S3/S6` 接收结构化结果
- 输出给 `S5` 的是动作提议和上下文，不是绕过门控的直接执行命令
- 不替代 `S5` 的审批判断，不替代 `S1/S2/S3/S6` 的具体实现

**必须交付物**

1. `7` 实体在模块内的表示方式
2. `S4A / S4B` 二级分层图
3. 任务、事件、状态关系图
4. 关键接口说明
5. 状态版本与迁移说明
6. 风险与验证清单

### 5.5 `S5` 安全合规与授权方案

**主责**

- 安全、合规、授权、动作门控、回退和审计规则。

**涉及模块**

- `safety_compliance_authorization`

**主要对象**

- `CareRelationship`
- `Task`
- `CareEvent`

**主要接口**

- 动作评估接口
- 远程确认结果接口
- 审计与回退接口

**依赖与边界**

- 接收 `S4` 的动作提议
- 对 `S1/S3/S6` 形成统一动作边界
- 不负责业务目标规划，不直接执行动作

**必须交付物**

1. 动作门规则表
2. 授权与角色边界说明
3. 降级、回退与异常保护方案
4. 审计字段说明
5. 风险与验证清单

### 5.6 `S6` 伴生系统与服务方案

**主责**

- 当前 `V1` 下的 `家属 App + 最小云` 主链，以及后续服务适配位的受控预留。

**涉及模块**

- `companion_service_system`

**主要对象**

- `Household`
- `CareRelationship`
- 外部协同相关的 `Task / CareEvent`

**主要接口**

- 远程确认接口
- 配置与结构化同步接口
- 后续服务适配位预留接口
- 记忆治理与审计导出接口

**依赖与边界**

- 接收 `S4` 的结构化事件和任务结果
- 接收 `S5` 的远程确认与授权边界
- 不能直接绕过端侧安全链控制本体动作
- 人工服务、在线问诊和第三方履约当前只保留适配位，不写入当前主链验收条件

**必须交付物**

1. `家属 App + 最小云` 最小闭环说明
2. 远程确认、配置同步与结构化结果回写说明
3. 后续服务适配位边界说明
4. 日志与审计字段说明
5. 风险与验证清单

### 5.7 `S7` 治理观测与技术研判方案

**主责**

- 观测、审计、数据治理、验证支持和横切追踪。

**涉及模块**

- `observability_data_governance`

**横切角色**

- `S7` 不是新的架构模块，而是所有工作包共用的观测与验证支持面。

**主要对象**

- 指标、日志、审计记录、验证证据

**主要接口**

- 指标采集接口
- 审计导出接口
- 数据分级与留存规则接口

**依赖与边界**

- 从 `S1-S6` 接收结构化运行信息
- 不直接做业务判断，也不单独决定技术路线
- 技术研判在这里作为输入整理职责存在，不反向替代主线架构判断

**必须交付物**

1. 指标与日志清单
2. 数据分级与留存规则
3. 阶段门证据清单
4. 问题归因与验证看板
5. 风险与验证清单

## 6. 所有工作包共同遵守的边界

1. 默认量产线按 `8GB RAM + 64GB Flash` 组织，不得把前瞻验证线默认写成量产线。
2. 原始敏感数据默认端侧处理；外发只允许最小必要结构化信息。
3. 所有涉及运动、上报、外部下单、医疗联动的动作，都必须经过 `S5`。
4. 夜间、卫生间、身份不确定和低照环境中的主动性增强，必须明确说明边界与降级规则。
5. 任何工作包都不得把“机器人 + App + 云 + 服务”的完整交付对象重新简化成单一机器人本体。

## 7. 各工作包必须提交的产物

1. 责任边界说明
2. 内部结构或分层图
3. 对外接口清单
4. 关键对象或状态说明
5. 风险与依赖清单
6. 验证方案与阶段门证据

## 8. 评审顺序

为减少接口漂移，建议按以下顺序评审：

1. `S4` 世界状态与决策方案
2. `S5` 安全合规与授权方案
3. `S1` 本体平台与运动方案
4. `S2` 人体感知与健康方案
5. `S3` 多模态交互与陪伴方案
6. `S6` 伴生系统与服务方案
7. `S7` 治理观测与验证支持方案

## 9. 启动条件

工作包正式启动前，至少应满足以下条件：

1. 已明确对应的主线文档和上游输入；
2. 已明确本包不负责什么；
3. 已明确关键接口与依赖方；
4. 已明确验证范围与阶段门证据；
5. 已明确是否涉及需要进一步评审的 `provisional` 内容。

## 10. 当前结论

当前 `S1-S7` 继续保留为开发承接入口。

后续如果再新增复杂度，必须先回答两个问题：

1. 它属于哪个工作包；
2. 它影响哪些模块、对象、接口和验证事项。

如果答不上来，就不应直接进入主线正文。
