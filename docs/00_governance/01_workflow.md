## 1. 目的

本工作流用于把当前仓库从“总体架构定义”持续推进到“可落地的软硬件方案、MVP 原型和实现计划”。

全生命周期视角见 [docs/00_governance/02_lifecycle_workflow_and_gates.md](02_lifecycle_workflow_and_gates.md)。

当前生命周期管理明确参考 `IPD`（集成产品开发）思想。

核心原则：

- 先冻结高代价架构决策，再进入实现细节
- 先定义稳定接口，再绑定具体算法、模型和供应商
- 所有关键假设都要记录为“已确认”或“待确认”，不能隐式漂移
- 对时间敏感、高风险或高采购成本的信息，必须在需要时联网核实
- 项目时间线按 2026 年 1 月 1 日起算，其中 2026 年 1 到 2 月已用于需求收集与 Demo 概念验证
- 当前总目标是 2026 年 12 月 31 日达到量产预备状态，2027 年 1 月完成 MVP 验证窗口
- 产品定义与架构冻结争取在 2026 年 3 月 31 日完成
- 每一阶段都要检查系统各实体元素的组合，判断整机与服务是否仍然呈现“聪明、温暖、精致”的高端产品感，并能支撑 `20000 到 30000 元` 售价区间

## 2. 协作方式

我作为架构师，按以下节奏与你协作：

1. 每轮开始前，我先检查 `input/00_requirements/00_user_requirements_input.md` 顶部“对 Codex 的要求”是否有变化。
2. 我会基于现有信息形成候选方案、问题清单和建议。
3. 只向你提一批高价值问题，避免一次性发散过多问题。
4. 你的回答会被沉淀到 `docs/00_governance/03_decision_log.md`。
5. 每完成一轮关键澄清，我会补充下一层文档，而不是停留在讨论层。
6. 涉及最新芯片、传感器、模型、法规、供应链价格时，我会联网确认后再给建议。
7. 每一轮本地文档中的待办和 issue，会同步到 Linear 项目中，保持 repo 文档和协作系统一致。
8. 本地 Markdown 文档和 Linear 项目、里程碑、issue、文档统一用中文维护。
9. 涉及 VLN 路线、导航推理和相关前瞻技术判断时，我会通过独立 Linear issue 与另一个负责 VLN 技术评估的 Codex 线程交叉校验；当前 `VLN` 专项线程的产物已纳入版本控制，后续由当前线程继续作为正式技术输入引用，并在需要时维护 `docs/09_research/01_vln_role_analysis_and_technical_plan.md`。
10. `input/00_requirements/00_user_requirements_input.md` 作为用户维护的需求事实源，只由用户修改；我只读取该文件并据此推进其他文档。
11. 当关键信息未澄清但推进不能停时，我会先形成“架构师设想包”，以 `provisional` 方式写入本地文档并同步到 Linear，等待你审核。
12. 当前总体架构主线已切换为“家庭共居智能体 + 多执行范式”；`OODA` 只保留为运行时层的离散决策范式。后续文档应优先体现执行范式轴、共享状态平面与跨范式审批硬边界，不再把旧 `OODA` 总法、`R1-R4` 或旧调度器写成当前一级主线。
13. 当前置评审 issue 未关闭时，我必须先强提醒你；后续 issue 可以并行起草，但不能在前置 issue 未关闭时被当成最终冻结结果。
14. 对存在明确结构分解的文档，我会优先同步补上直观图示，并把图示视为正式架构产物，而不是可有可无的排版装饰。
15. 每一轮涉及成本、功耗、结构、交互或伴生系统裁剪的评审，我都要同步检查其是否损伤“聪明、温暖、精致”的高端产品感，并记录是否还能支撑 `20000 到 30000 元` 售价区间。
16. 每一轮都要复查 Linear 中仍处于 `In Review` 的 issue 是否与本地文档状态一致；若文档已形成正式基线，则必须及时关闭或重定向，避免工作流滞留。
17. 当我暂停并等待你输入时，必须提出明确的审阅需求，至少指出“请审阅哪个 issue 或哪份文档的哪一段提案”，不能只给模糊的“下一步继续推进”。
18. 允许为后置里程碑提前起草“逆向约束型文档”，用于把量产、发布、运营或验证约束提前反压到前置设计；但这不代表已经越过前置里程碑，当前主优先级必须始终回到尚未关闭的前置里程碑。
19. 根 `README.md` 只维护“当前状态、当前有效入口、当前阶段门入口和历史资料指针”，不再平铺全部历史评审与长阅读清单。
20. 当前主线事实源默认按“原则层 -> 总体架构 -> 运行时基线 -> 状态 / 接口 / 专题层 -> 总体方案下发基线”读取；`14_family_co_living_agent_paradigm.md` 只保留背景/决策来路角色，`02_pdcp_system_architecture_review_package.md` 只保留阶段评审包角色。
21. `docs/08_reviews/` 当前只保留少量活跃入口，其余文档进入 `archive/`；`docs/superpowers/` 采用 active-only + archive 结构，已被主线吸收的工作文档不再停留在活跃入口。
22. 详细 `provisional` 内容只允许保留在受控载体中：原则附录、活跃评审总包、受控附录、`docs/09_research/` 与 `docs/superpowers/` 工作文档；其他主线文档只允许保留摘要与指针，不再展开候选子表。
23. 当前主线默认把 `01_overall_architecture.md` 作为唯一总图文档、`03_execution_paradigms_runtime_baseline.md` 作为唯一运行时语法文档、`06_decision_state_machine.md` 作为唯一离散决策状态机文档；进入开发承接时，默认从 `docs/03_p2_feasibility/01_overall_solution_and_module_design_baseline.md` 开始。
24. 活跃主线文档默认目标控制在 `500` 行以内；若超过 `600` 行，必须在文档定位、索引或治理文档中说明继续保留为单文件的理由。
25. `docs/08_reviews/` 不再随手新增活跃入口；复杂度复盘、阶段后总结与类似“总结型评审”默认进入 `archive/`。
26. 主线文档不得展开详细 `provisional` 子表，只允许保留摘要与指向 Linear issue 的指针；每个活跃 `provisional` 都必须绑定明确承接项，并至少写清假设内容、文档位置、为何未冻结、冻结条件、目标阶段门与 owner。
27. `docs/00_governance/03_decision_log.md` 与 archive 文档中的历史 `provisional` 留痕默认只用于追溯，不直接计入活跃复杂度 KPI；当前治理不采用“全仓总数”这类僵硬 KPI。

## 2.1 跨线程 Codex 协作规范

为避免“其他线程只改单文档、不补工作流”的情况，当前明确增加以下 `6` 条跨线程协作规范：

1. 其他线程只能把本线程负责的专项结论写成输入，不得绕过当前主线评审顺序直接改写系统边界。
2. 其他线程如修改已关闭 issue 对应文档，必须同步回写 `docs/00_governance/03_decision_log.md`、`README.md`、`CHANGELOG.md` 和对应 Linear issue / 评论。
3. 其他线程如新增下游约束或派生任务，必须明确由哪个当前主线 issue 承接、由哪个协作 issue 承接，并在 Linear 中建立关联。
4. 其他线程如引入会影响模块下发的技术判断，必须同步回写 `docs/03_p2_feasibility/01_overall_solution_and_module_design_baseline.md` 或同级承接文档，不能只停留在专项文档。
5. 其他线程不得修改 `input/00_requirements/00_user_requirements_input.md`，不得绕过用户审阅节点直接宣布冻结结论。
6. 其他线程提交前必须执行一次文档一致性检查，至少检查文件名引用、索引、旧路线残留、Linear 状态和本地评审入口是否一致。

## 2.2 前置评审清账规则

为避免“后续文档写了很多，但前置边界还没关掉”导致返工，当前明确采用以下规则：

1. `KBT-5 模块边界` 是 `KBT-6 世界状态结构` 的前置评审项。
2. `KBT-6 世界状态结构` 是 `KBT-7 决策状态机` 的前置评审项。
3. `KBT-7 决策状态机` 是后续接口冻结、工程化基线和技术选型冻结的重要前置项。
4. 后续 issue 可以先起草，但只要前置 issue 未关闭，我就必须把其结论标成 `provisional` 或“待前置关闭后正式冻结”。
5. 每一轮开始时，我都要先检查仍处于 `In Review` 的前置 issue，并强提醒你当前未清账项。
6. 如果后置里程碑中已有提前起草的约束型文档，它们只作为输入存在，不能抢占当前主评审顺序。

当前评审顺序强制收敛为：

1. `KBT-5` 模块边界
2. `KBT-6` 世界状态结构
3. `KBT-7` 决策状态机
4. `KBT-8` 安全 / 合规 / 授权接口
5. 其余功能包、工程化和选型项

## 4. 并行工作流

为了避免“先做完软件再看硬件”或“先买完硬件再想架构”的错误顺序，后续工作按六条并行流推进：

### A. 产品与场景流

- 用户画像
- 家庭场景切片
- 首发功能优先级
- 使用频率和失败容忍度

### B. 安全与合规流

- 伤害场景分析
- 数据隐私边界
- 用户授权机制
- 首发市场监管约束

### C. 硬件平台流

- 底盘和运动形态
- 传感器布局
- 端侧算力
- 电池、散热和结构

### D. 端侧智能流

- 导航与避障
- 感知与身份识别
- 语音交互
- 世界状态更新

### E. 云与应用流

- 联网问答
- App 与远程控制
- 配置中心
- 审计、日志与运维

### F. 验证与量产准备流

- 指标定义
- 测试台架
- 家庭试验
- 供应链风险

### 功能澄清主线

在上述六条并行流之内，当前阶段的功能澄清不再按单一健康主线推进，而按三条功能主线并行推进：

- 健康管理：重点冻结事件管线、数据时效、人工服务升级和外部平台责任边界
- 陪伴交互：重点冻结人设边界、主动发起规则、长期记忆和多角色共享边界
- 安全保障：重点冻结家庭危险源清单、空间分级、降级停机矩阵和异常升级规则
