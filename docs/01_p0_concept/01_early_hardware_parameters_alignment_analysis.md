# 早期硬件参数需求与新架构一致性分析

## 1. 文档目的

本文用于分析 [input/01_kinbot_prd_v0_2_hardware_parameters.csv](../../input/01_kinbot_prd_v0_2_hardware_parameters.csv) 这份早期版本参数需求，与当前主线架构基线是否一致，并判断：

1. 哪些参数仍然符合当前架构理念；
2. 哪些参数已经偏保守；

## 2. 总体结论

### 2.1 总判断

这份 `v0.2` 早期参数表与当前新架构 **方向基本一致，但表达方式已经明显过时**。

它对产品的理解并没有走偏，仍然能看出以下核心方向：

- 面向居家养老场景；
- 以陪伴、健康、安全为核心价值；
- 机器人本体需要具备移动、交互、储药/储物、递送、回充和主动安全能力；
- 穿戴设备、智能家居、App 等外部输入已经被纳入整体方案思考。

但它的问题在于，这份文档仍然是一个 **扁平的器件/功能/参数清单**，而当前主线架构已经升级为：

1. “机器人本体 + 伴生系统 + 外部生态”的完整产品系统边界；
2. “本体实体域 + 运行时模块”的双视角架构；
3. 多尺度 `OODA`；
4. 以 `TTFT / TPS / 热稳态持续 TPS` 定义端侧算力；
5. 以 `BOM 6000 到 8000 元`、`售价 20000 到 30000 元`、`>4h` 续航与高端产品感共同约束方案。

所以，这份 `v0.2` 文档 **不能再直接作为当前主线参数基线**，但仍然可以作为“早期本体能力种子表”和“需求来源档案”保留。

### 2.2 一句话结论

如果只看产品方向，它大体是对的；如果把它当成当前硬件冻结基线，它已经不够用了。

## 3. 与当前新架构一致的部分

以下内容与当前主线依然基本一致，应视为可以保留的方向性输入。

### 3.1 机器人本体形态约束总体合理

以下参数方向与当前“家庭空间友好、老人场景安全、非机械臂路线”的理念一致：

- 高度范围 `80-140cm`
- 最大宽度 `≤60cm`
- 最小通行宽度 `65cm`
- 原地转弯半径 `0`
- 最大速度 `0-0.6m/s`
- 最大越障高度 `2.5cm`
- 最大越缝/越沟 `4cm`

这些要求与当前主线强调的“家庭空间通过性、夜间安全、找人到人、到点递送、非复杂操作型机器人”是同向的。

### 3.2 储药/储物仓能力方向与当前主线一致

虽然早期文档使用的是“药箱”表述，但其核心思想与当前一代“储物仓能力保留、机构重构”的方向基本一致：

- 需要本体具备递送相关仓体；
- 仓体可电动开合；
- 断电或故障时应支持手动兜底；
- 需要防夹手能力；
- 需要支持日常药品与小型健康用品存放。

这与当前主线中“储物仓是整机能力的一部分，而不是外挂附件”的判断一致。

### 3.3 感知与交互主轴基本同向

早期文档提出了：

- 环视相机 + 交互相机；
- 麦阵、声源定位、唤醒、扬声器；
- 人脸、声纹、手势、物品识别；
- 自主导航、自动回充、自动跟随。

这些内容与当前主线的 `multimodal_interaction`、`mobility_navigation`、`human_health_sensing` 三条能力主轴仍是同向的。

### 3.4 安全保护意识与当前主线一致

早期文档已经把以下内容列入显式要求：

- 急停；
- 倾斜 / 倾倒 / 抬升 / 跌落保护；
- 电池 / 电机保护；
- 仓门防夹手；
- 突发疾病、跌倒、睡眠异常、地面积水、陌生人闯入等风险事件。

这说明早期版本已经具备“安全先于动作”的意识，与当前 `safety_compliance_authorization` 总门和安全闭环是一致的。

### 3.5 续航目标与当前主线一致

早期文档提出平均续航时间 `4h`，这与当前功耗和能效基线中的 `>4h` 目标保持一致。

这意味着续航目标没有方向性冲突，冲突主要出现在 **计算口径和实现余量** 上，而不是目标本身。

## 4. 与当前新架构不完全一致的地方

这里不是说“参数错了”，而是说它的组织方式已经不适合当前阶段。

### 4.1 它把“产品系统参数”和“本体参数”混在了一起

当前架构已经明确，Kinbot 不是单一本体，而是：

- 机器人本体；
- 穿戴与健康外设；
- 智能家居；
- 家属应用；
- 云服务；
- 后台人工服务；
- 外部生态接口。

但这份 `v0.2` 文档里，`APP`、`4G/5G`、睡眠异常、外出未归、地震报警、陌生人闯入、烟雾报警等内容，和底盘、相机、药箱、电池、电机混在同一张硬件参数表里。

这在当前阶段会产生两个问题：

1. 容易把“服务闭环能力”误写成本体硬件硬承诺；
2. 容易把“系统级能力”误写成单机指标。

### 4.2 它把“能力项”和“验收项”混在了一起

例如：

- `突发疾病检测与处理：有`
- `睡眠异常检测与处理：有`
- `陌生人闯入报警：有`
- `物品识别：有`

这种写法在早期讨论阶段可以接受，但在当前主线里已经不够了。当前架构更需要的是：

- 触发条件；
- 风险等级；
- 输入来源；
- 端云责任边界；
- 是否进入一期标准承诺；
- 成功率 / 误报率 / 漏报率 / 回退路径。

也就是说，它的很多条目不是“错”，而是 **颗粒度不对**。

### 4.3 它没有按当前双视角架构来表达参数

当前主线已经要求参数至少要能挂到两类对象上：

1. 本体实体域：算力、底盘、感知、交互、电池热、储物仓结构；
2. 运行时模块与闭环：导航、交互、健康、安全、服务、治理。

而 `v0.2` 文档是单层表格，因此它更像早期需求池，而不是当前可直接下发的系统级参数基线。

## 5. 参数偏保守的地方

这里的“偏保守”，主要指参数或假设会限制当前方案空间，或者明显继承了旧硬件思路。

### 5.1 电池方案过早锁到 `18650 / 7s4p / 25.2V`

早期文档已经把电池路线写成：

- 电芯型号 `18650`
- 电芯组合 `7s4p`
- 额定电压 `25.2V`

这在今天看是偏保守的。问题不是这条路线绝对不能做，而是它把“电池实现方案”过早固化了。

当前主线更强调：

- 先冻结 `C5` 功耗基线和 `W1-W4` 四类工况；
- 再由续航、热、重量、结构空间、维护性共同反推电池路线；
- 不应在系统架构阶段把电芯规格当成既定事实。

因此，这一组参数应视为“旧实现假设”，不是当前应冻结的产品级参数。

### 5.2 端侧算力定义方式偏保守

早期文档对算力的表达是：

- `基础算力：待定`
- `升级算力：待定`

这种表达方式已经落后于当前主线。

当前主线要求端侧算力必须按以下方式定义：

- 任务模板；
- `TTFT`；
- 稳态 `TPS`；
- 热稳态持续 `TPS`；
- 真实板级视觉输入与上下文长度。

因此，早期文档的算力定义不是数值偏低，而是 **方法偏保守**，仍停留在“给芯片分档”的旧方式，没有进入“按真实体验和任务负载定义算力”的新方式。

### 5.3 `IPX2` 本体防护等级可能偏保守，值得重新评估

早期文档把机器人和充电站都写成 `IPX2`。

在当前架构下，这不是立即判错项，但它显得偏保守，原因是：

- 产品明确面向居家养老；
- 一代场景里有地面积水、防滑提醒、厨房 / 阳台 / 卫生间相邻区域等真实水风险；
- 长期陪伴型产品比普通玩具型机器人更需要考虑“低强度但高频”的家庭泼溅与误水环境。

所以 `IPX2` 更像“最低可接受起点”，不应直接视为当前高端产品定位下的理想目标。

## 6. 参数偏激进的地方

这里的“偏激进”，主要指它在当前成本、功耗、架构边界和一期承诺下，目标压得过高或过宽。

### 6.1 `定位精度 ≤2cm` 过于激进

在当前主线中：

- 量产主线是双目 + 单目 `3 到 5` 相机；
- 深度相机和激光雷达退出一代量产主线；
- `UWB` 不进入主导航硬依赖。

在这样的基线上，直接把整机定位精度写成 `≤2cm`，明显过于激进。

这类要求更像实验条件下的局部定位目标，而不是当前一代量产路线的系统级指标。它会对：

- 视觉路线；
- 标定；
- 底盘闭环；
- 算力；
- 功耗；
- 测试成本

都带来不成比例的压力。

### 6.2 `0-500㎡` 家庭建图范围过于宽泛

对于当前已冻结目标用户和首发场景而言，`0-500㎡` 更像一个“理论上支持的超大范围”，而不是一代真实主场景。

问题不在于系统绝对不能支持大户型，而在于：

- 这不是当前主目标用户的典型场景；
- 它会放大地图、重定位、算力和存储预算；
- 会让原本应该为居家养老场景优化的系统，去承担大面积泛化目标。

因此，这个参数在当前阶段偏激进，也偏分散。

### 6.3 相机组合如果按早期写法全部兑现，会比当前量产主线更激进

早期文档写的是：

- 环视相机 `4`
- 交互相机 `1-2`

如果按字面全部落地，本体将达到 `5-6` 个摄像头。

而当前主线已经把量产主线收敛到：

- 双目 + 单目；
- 总数 `3 到 5`；
- 自研深度估计；
- 避免额外深度感知硬件。

所以，早期相机路线不是完全错误，但如果全盘照搬，会比当前一代量产主线更激进，尤其是在：

- `C2` 成本；
- `C5` 功耗；
- 标定复杂度；
- 结构与外观；
- 板级带宽

这些方面都会更吃紧。

### 6.4 交互相机 `60fps`、`自动对焦`、高广角同时冻结，偏激进

早期文档对交互相机提出了：

- `60fps`
- `自动对焦`
- `H ≥150° / V ≥120°`

这组目标单项看都不是不合理，但一旦组合起来，并叠加当前一代成本与能效约束，就变得偏激进。

因为它会同时推高：

- 模组成本；
- ISP 压力；
- 热与功耗；
- 带宽；
- 算法调试复杂度。

当前更合理的做法应是：先冻结关键体验需求，再反推是否真的需要把这些指标全部做成一期硬承诺。

### 6.5 睡眠异常与突发疾病能力描述明显过宽

早期文档写法包含：

- 心绞痛、急性心肌梗死、脑卒中、肺栓塞等突发疾病检测与处理；
- 睡眠呼吸暂停、不宁腿、快速眼动睡眠行为障碍等睡眠异常检测与处理。

这在当前主线下明显过激进。

当前主线对一代能力的收敛更保守也更工程化：

- 睡眠监测只承诺“夜间离床、未按习惯起床、基础睡眠节律观察”；
- 突发疾病只承诺“候选事件触发、风险分级、补采、联动和升级”，不承诺机器人自主做医学诊断。

因此，早期文档这部分更像“产品理想愿景”，不是当前一代标准能力参数。

### 6.6 `4G/5G 模块` 作为本体必选项偏激进

早期文档把 `4G/5G 模块` 直接写成“有”，并举例用于拨打紧急联系人、小区物业、救援电话。

而当前主线已经把这件事收敛为：

- 一期优先通过手机 App / 手机能力侧承接应急拨号；
- 机器人直接入网拨号只做架构预留，不进入一期预研主线。

所以，把蜂窝模块写成一期本体硬参数，已经偏激进，也偏早定。

### 6.7 `250Wh` 电池与 `≤1.5h` 快充组合偏激进

早期文档给出的电池与充电组合是：

- 平均续航时间 `4h`
- 电池电量 `250Wh`
- 充电时间 `≤1.5h`

按当前主线的 `W1-W4` 工况模型和 `>4h` 续航、热稳态、大算力端侧模型路线来看，这组组合偏激进，原因是：

1. `250Wh` 更接近当前可行区间的下沿；
2. 当前功耗预算已经把 `W4` 视为危险工况；
3. 若主线算力平台采用当前大算力候选，`250Wh` 很容易把系统推到“续航刚够、热设计紧、快充压力大”的边缘；
4. `≤1.5h` 的家庭产品快充目标，还会进一步抬高充电电流、热和电池寿命压力。

因此，这组参数不适合直接作为当前冻结值，更适合保留为早期目标假设。

### 6.8 `≤45kg` 机器人净重上限偏宽，等价于对整机方案过于激进

表面上看这是保守要求，因为它是“上限”。但对当前产品目标来说，这个上限其实偏宽。

原因是当前主线已经把以下能力同时压进一代：

- 全向观测；
- 屏幕交互；
- 储物仓能力保留；
- 大算力评估；
- `>4h` 续航；
- 高端产品感中的“轻盈感、精致感、舒适感”。

在这种情况下，如果允许本体做到 `45kg` 上限，实际上会对：

- 轻盈感；
- 重心稳定；
- 安装搬运；
- 地面适配；
- 成本与材料；
- 噪声与启停质感

带来较大压力。

所以它不是“危险错误”，但以当前理念看，这个上限偏宽。

## 7. 既不该简单保留，也不该简单删除的条目

有一类条目不属于“对/错”，而属于“需要改写表达方式”。

### 7.1 `药箱` 应统一升级为 `储物仓`

当前主线已经从“药箱”升级为“储物仓”，原因是：

- 一代不仅要放药，还要放常用健康用品；
- 仓体是整机结构和递送链的一部分；
- 当前强调通用空间，而不是为单一药品容器定制。

因此，早期文档这部分不是方向错，而是术语已经落后。

### 7.2 “有 / 没有”能力项应改成分层承诺

例如：

- 睡眠异常检测；
- 外出未归；
- 陌生人闯入；
- 地震报警；
- 物品识别；
- 手势识别。

这些能力当前更适合按：

- 必须有；
- 应该有；
- 可以有；
- 预留；
- 观察线

来分层，而不是统一写成“有”。

### 7.3 安全能力应改写为“能力 + 触发条件 + 回退路径”

比如：

- 倾倒检测；
- 防跌落；
- 仓门防夹手；
- 突发疾病候选事件；
- 水滑风险提醒。

当前更需要的是“闭环定义”，而不是单项功能名称。

## 8. 最终判断

### 8.1 是否符合新的架构理念

**大体符合，但不再适合作为当前主线基线文档。**

它符合的是产品方向，不符合的是当前参数治理方式。

### 8.2 是否有参数过于保守

有，主要体现在：

1. 电池路线过早锁到 `18650 / 7s4p / 25.2V`
2. 算力定义仍停留在“基础算力 / 升级算力”
3. `IPX2` 本体防护可能只是最低起点，不像高端长期陪伴产品目标

### 8.3 是否有参数过于激进

有，主要体现在：

1. `定位精度 ≤2cm`
2. `0-500㎡` 家庭建图
3. `5-6` 摄像头的字面兑现风险
4. 交互相机 `60fps + 自动对焦 + 超广角` 同时冻结
5. 睡眠异常与突发疾病能力描述过宽
6. `4G/5G` 作为一期本体硬要求
7. `250Wh + ≤1.5h` 快充组合
8. `≤45kg` 的本体重量上限过宽

## 9. 建议如何使用这份早期文档

当前建议对这份 `v0.2` 文档采用以下处理原则：

1. **保留**：作为早期需求来源档案和本体能力种子表。
2. **不直接冻结**：不把它当作当前系统级参数基线。
3. **重点继承方向**：空间尺寸、储物仓能力、安全意识、4h 续航目标、陪伴交互主轴。
4. **重点回收旧假设**：18650 电池、`2cm` 定位、`500㎡` 建图、4G/5G 必选、疾病检测过宽承诺。
5. **后续若重写参数基线**：应按“产品系统级 / 本体实体域 / 运行时验收 / 伴生系统 / 外部生态接口”五层结构重写，而不是继续扩写这张扁平表。

## 10. 我的建议结论

如果只问一句话，我的判断是：
