## 1. 文档定位

本文档用于承接家庭共居智能体革新路线的 `Phase 3`。

它不直接重写 `World State` 主表，也不直接宣布 `9 -> 7` 成立，而是回答以下 `4` 个问题：

1. `Phase 2` 完成后，状态模型与数据模型到底还剩什么关键决策没有完成；
2. 当前应比较哪些候选路线，而不是继续模糊讨论；
3. 每条路线的收益、代价、触发条件与风险分别是什么；
4. 用户在这一轮审阅时，真正需要拍板的最小问题集是什么。

## 2. 当前阶段判断

到 `Phase 2` 结束时，当前主线已经完成：

1. 总架构从旧 `OODA` 总中心上抬为“家庭共居智能体”；
2. 运行时层改写为多执行范式表达；
3. `World State` 被明确定位为多执行范式共享的状态平面。

但以下事项仍未冻结：

1. 现有 `9` 类一级实体是否应重组为 `7` 类；
2. `CareRelationship / CareEvent` 是否应成为正式一级实体；
3. `V1` 的实际能力闭环，是否已经需要承担足够大的结构重组成本。

因此，`Phase 3` 的核心不是“立即改表”，而是“先完成状态模型与数据模型的正式决策”。

## 3. 当前决策问题重写

当前真正要回答的问题，不是：

**“`World State 9 -> 7` 看起来是不是更优雅？”**

而是：

**“在当前 `V1` 产品边界、模块并行设计和评审节奏下，Kinbot 是否已经具备承担 `World State` 主表重组成本的必要性与时机？”**

这会直接决定：

1. `Phase 3` 是应先做扩展层治理，还是直接重组主表；
2. 当前结构变化是帮助主线降复杂度，还是反而制造新的迁移成本；
3. 下游 `PDCP / KBT-32 / KBT-33` 文档是否会被迫再次大面积改写。

## 4. 当前必须遵守的约束

本轮评审必须同时满足以下约束：

1. 不得把 `provisional` 判断伪装成已冻结事实；
2. 不得为了结构美观而提前扩大迁移范围；
3. 不得破坏 `Phase 2` 已建立的多执行范式、共享状态平面和跨范式审批硬边界；
4. 不得让当前架构重新变复杂。

针对第 `4` 点，本轮明确追问一次：

**现在的架构是不是太复杂了？**

当前判断是：**如果在 `V1` 支撑关系和验证口径都还没写清之前，就直接重组 `World State` 主表，复杂度大概率会上升。**

## 5. 候选路线

### 5.1 路线 A：保留 `9` 类骨架 + 关系 / 事件扩展层

定义：

- 当前继续保留 `Person / RoleBinding / Household / Place / Object / HealthProfile / MedicationAsset / Task / RiskEvent` 这 `9` 类一级实体；
- 通过扩展层方式引入“关系视图”和“事件视图”，而不是立刻改主表；
- 关系与事件先作为跨实体治理层、聚合层和评审层存在。

这条路线的优点：

1. 与当前 `Phase 2` 主线衔接最稳；
2. 不需要立即触发大规模文档迁移；
3. 更有利于控制复杂度；
4. 允许先验证“关系 / 事件是否真值得独立上提”。

这条路线的风险：

1. 结构上看起来不如直接重组“干净”；
2. 需要对扩展层和主表边界做清晰治理，避免形成双重事实源；
3. 若扩展层长期不收敛，后续仍可能回到主表重组。

### 5.2 路线 B：直接执行 `World State 9 -> 7`

定义：

- 当前直接启动一级实体重组；
- 同步决定 `CareRelationship / CareEvent` 的正式地位；
- 把现有若干实体做合并、抽象或下沉。

这条路线的优点：

1. 理论上更接近下一代总图的统一表达；
2. 有机会把长期关系和事件链路直接前置到主表层。

这条路线的风险：

1. 迁移成本显著更高；
2. 需要立即重写多处主线文档与评审包；
3. 若 `V1` 支撑关系仍不稳定，容易出现“先改结构，后补验证”的倒挂；
4. 更容易让当前架构再次膨胀。

## 6. 对比判断

| 维度 | 路线 A：`9` 骨架 + 扩展层 | 路线 B：直接 `9 -> 7` | 当前判断 |
| --- | --- | --- | --- |
| 与 `Phase 2` 连续性 | 高 | 中 | `A` 更稳 |
| 对当前文档迁移冲击 | 低 | 高 | `A` 更稳 |
| 对复杂度控制 | 高 | 低到中 | `A` 更稳 |
| 对下一代框架一致性 | 中 | 高 | `B` 理论更强 |
| 对 `V1` 当前支撑关系要求 | 中 | 高 | `B` 需要更多先决条件 |
| 当前可执行性 | 高 | 中低 | `A` 更现实 |

## 7. 当前推荐

当前推荐不是立即确认 `9 -> 7`，而是：

**先按路线 A 启动 `Phase 3`，把“关系 / 事件扩展层”作为当前默认候选路线；待验证口径、`V1` 支撑关系和下游迁移影响更清楚后，再决定是否进入路线 B。**

这只是当前推荐，不是已冻结结论。

当前仍保持 `provisional` 的判断包括：

1. `CareRelationship` 是否必须独立为一级实体；
2. `CareEvent` 是否必须独立为一级实体；
3. `9 -> 7` 是否真能减少复杂度，而不是把复杂度转移到别处。

## 8. 本轮建议用户只确认这三件事

1. 是否接受 `Phase 3` 先比较两条路线，而不是直接改主表；
2. 是否接受当前默认先走路线 A；
3. 是否接受把 `9 -> 7` 继续保留为后续单独冻结项。

## 9. 下一步动作

若本轮审阅接受本文框架，下一步应按以下顺序推进：

1. 在 `05_world_state_schema.md` 中继续保留 `9` 类骨架，并显式挂接“关系 / 事件扩展层”的承接位置；
2. 在下游模块方案评审中检查哪些接口已经真正依赖“关系 / 事件”成为一级实体；
3. 当且仅当这些依赖已经稳定、且确实能减少复杂度时，再进入 `9 -> 7` 的正式冻结评审。
