## 1. 文档定位

本文档承接：

1. `docs/08_reviews/24_kbt52_strategic_ambition_gap_review.md`
2. `docs/08_reviews/23_phase4_downstream_sync_closure_package.md`
3. `docs/08_reviews/13_future_home_robot_vision_charter_for_emt.md`

它不重写 `Phase 1-4` 的任何冻结结论，也不修改七实体目标模型、模块边界或已冻结接口。

它只回答一件事：

**在工程架构已经基本正确的前提下，还要补什么战略接口，才能让 Kinbot 从“能做出产品”更进一步变成“有机会赢市场”。**

## 2. 当前判断

`Phase 1-4` 已经证明：

1. 总架构方向正确；
2. 七实体 `World State` 方向成立；
3. 下游主线已开始承接新模型。

但从 `Step 42` 的战略目标看，当前主线仍缺少三类“赢市场”的架构对象：

1. **平台位**：机群、跨家庭学习、伴生系统对等参与者
2. **生成位**：医学理解、自我模型、主动生成能力
3. **惊奇位**：远程在场感、关系演化、未脚本化高价值时刻

因此，当前新增一个独立 `Phase 4.5` 是合理的；它不是返工，而是战略层补强。

## 3. 补强范围

### 3.1 平台位

当前需要占位的对象：

1. `FleetView`：用于表达跨家庭、跨设备、跨试点批次的共性模式和群体学习入口
2. 伴生系统对等参与者：家属 App、云服务、人工坐席、第三方履约不再只作为“外部依赖”理解，而应作为未来平台协同节点占位

当前约束：

1. 不新增第 `8` 个正式实体
2. 只允许以视图位、占位说明、候选附录形式出现
3. 不进入 `V1` 已冻结实体集

### 3.2 生成位

当前需要占位的对象：

1. 生成范式（候选）
2. `Self` 视图位
3. 第 `8` 条原则候选：具身智能体的主动生成义务

当前约束：

1. 不把生成范式写成新的已冻结运行时中心
2. 不把 `Self` 提升为正式实体
3. 不把候选第 `8` 条原则混写成已冻结原则

### 3.3 惊奇位

当前需要进入验证体系的战略对象：

1. 远程在场感
2. 关系演化
3. 惊奇时刻

这三者与工程验证并列，但不替代工程验证。

## 5. 战略验证泳道

`Phase 5` 需要新增一条与工程验证并列的战略验证泳道，固定验证以下三问：

### 5.1 远程在场感

问题：

> 异地子女是否真切感到“被陪伴着”？

要求：

1. 观察对象：子女与被照护老人
2. 数据来源：交互日志、家属 App 行为、结构化访谈
3. 最小成功信号：子女主观确认远程在场感提升，且能指出至少一个具体场景
4. 失败解释边界：若失败，需区分是交互问题、服务接力问题还是根本价值问题

### 5.2 关系演化

问题：

> 30 天后，家庭是否开始把机器人当作“家里的一员”？

要求：

1. 观察对象：老人本人、子女、保姆等长期接触者
2. 数据来源：关系阶段观测信号、访谈、使用留存、主动触发互动记录
3. 最小成功信号：出现从 `initial` 向 `familiar` 的观测式迁移证据
4. 失败解释边界：若失败，需区分是交互频率不够、可靠性不足，还是价值主张错位

### 5.3 惊奇时刻

问题：

> 在 `100` 台试点窗口内，是否能稳定找出 `>=5` 个未脚本化的高价值正向时刻？

要求：

1. 观察对象：试点家庭、交付团队、运营团队
2. 数据来源：试点复盘、事件日志、人工标注、用户反馈
3. 最小成功信号：出现 `>=5` 类可复现的“令人意外的好时刻”
4. 失败解释边界：若失败，需区分是产品没有惊喜、还是惊喜未被正确记录
