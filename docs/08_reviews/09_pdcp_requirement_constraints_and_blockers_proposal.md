## 1. 文档目的与使用方式

本文用于在 `PDCP` 评审前，给管理层快速回答三件事：

1. 当前架构设计到底受哪些“需求侧硬约束”控制。
2. 架构继续推进前，哪些问题必须先被解决。
3. 需要老板在组织、资源、决策层面拍板什么。

建议把本文件作为 `PDCP` 评审会的“前置沟通提案”，用于会前对齐与会中决策。

## 2. 当前架构所基于的全部需求侧约束（收敛版）

> 说明：以下约束按“必须满足”的口径整理，重点覆盖产品目标、系统边界、体验、安全合规、工程与商业约束。

### 2.1 产品与业务目标约束

1. **目标系统是机器人本体，但交付对象是“机器人+伴生系统+服务”的完整产品系统**，不能只做本体能力。  
2. **首发场景固定为中国大陆家庭室内养老场景**，首发人群为独居老人或子女不在身边的老两口。  
3. **首版价值优先级固定**：健康管理 > 陪伴交互 > 家庭安全巡护 > 老人看护。  (第一代产品这样排序合适，长期看陪伴交互才可能维系住机器人与用户之间的关系纽带)
4. **产品感知约束固定**：系统演进不能破坏“聪明、温暖、精致”的高端产品感，推测需支持 （1.5-2）？（2-3）？ 万元售价带。  （待决策点）

### 2.2 系统边界与能力边界约束

1. **系统边界固定为移动交互机器人，不要求物理操作（manipulation）**。  
2. **移动形态固定为轮式**，且要具备家庭低矮门槛通过能力。  
3. **交互入口固定为多模态**：语音、屏幕、光线、手势、手机 App、远程控制。  
4. **交互人设倾向固定为陪伴/社交型**，并允许有限度主动打断与主动靠近（在礼仪和授权边界内）。  

### 2.3 OODA方法与决策治理约束

1. “Observe-Orient-Decide-Act”主循环已冻结为多尺度、并发、可中断、可动态调度架构基线。 
2. **决策优先级不可更改**：安全 > 合规 > 用户指令 > 任务完成率 > 效率 > 能耗。  
3. **自主权边界固定**：用户已授权行为必须做到完全自主；未授权行为不得越权执行。  
4. **动作执行前必须经过安全/合规/授权门控**，不能因追求响应速度绕开治理链路。  

### 2.4 数据、隐私与合规约束

1. **原始视觉与语音数据必须端侧处理**，并执行匿名化与加密存储。  （可以考虑的方式：回流但做好加密）
2. **生物特征身份信息（如人脸/指纹）必须端侧处理并加密管理**，不允许裸传出端。  
3. **异常上报链路必须建立授权与审计机制**，默认优先家属链路并逐步打通社区/物业/公共应急协同。  

### 2.5 端云协同与可用性约束

1. **直接处理视觉/语音原始数据与运动安全能力必须在端侧闭环**。  
2. **联网依赖能力（通用知识检索、外部服务调用）可上云，但需有断网可解释降级策略**。  
3. **离线硬约束**：断网不得影响运动与避障安全能力。  

### 2.6 工程、成本与项目管理约束

1. **硬指标约束**：续航 >4 小时；尺寸建议不超过 50×50×120cm；整机物料成本 5-6000 元。 
2. **周期约束**：2026-03-31 争取完成产品定义与架构冻结；2026-12-31 达到量产预备；2027-01 进入 MVP 验证窗口。  
3. **组织与预算约束**：团队将由当前约 25-30 人扩展至 60-80 人；周期预算约 8000 万-1 亿元。  
4. **流程约束**：必须按 IPD 阶段门推进，并在每次转阶段前显性提交审查。  

## 3. 架构进一步推进前必须解决的问题（阻断清单）

### 3.1 P0级阻断（不解决就不应进入下一阶段）

1. **端侧算力平台主线未收敛（D1）**  
   - 影响：模型规模、实时性、热设计、成本桶 `C1/C5` 与整机体验同时受影响。  
   - 不解决后果：模块方案并行推进会出现“接口已定但算力承载失真”的系统性返工。  

2. **整机平台与关键器件基线未冻结（D6）**  
   - 影响：传感器组合、底盘能力、功耗预算、重量与成本目标联动失稳。  
   - 不解决后果：软硬件接口与验证样机版本持续漂移，NPI 前置准备失效。  

3. **伴生系统最小闭环定义未冻结（D7）**  
   - 影响：家属 App、云服务、后台人工坐席、异常升级链路无法形成可运营闭环。  
   - 不解决后果：即使本体可用，也无法支持试点、售后与规模交付。  

### 3.2 P1级关键问题（不先控住会侵蚀P0结果）

1. **成本-性能-感知三角缺乏统一决策机制**：降本动作可能破坏高端感知与核心体验。  
2. **双视角（本体实体域 vs 运行时模块）缺少持续一致性治理**：变更影响难以及时追踪。  
3. **一级接口虽已提出，但“冻结范围/演进范围/验收标准”尚未形成统一清单**。  
4. **夜间闭环与低照环境能力尚需工程实证**：若验证不足，将直接冲击安全和信任。  

### 3.3 当前最大系统性风险

当前不是“方向不清”，而是**方向清晰但关键基线未在同一时间窗内收敛**。若继续并行扩张任务，风险将从“局部技术风险”升级为“项目级时序风险”。

## 4. 与老板沟通的提案（可直接用于会前汇报）

## 4.1 需要老板拍板的3个决策

1. **决策A：端侧算力平台主线拍板**  
   - 决策内容：确定主平台与备平台，冻结一期模型规模上限与量化策略边界。  
   - 期望输出：一页纸平台决策结论 + 对 `C1/C5` 的成本与热预算承诺。  

2. **决策B：整机关键器件冻结策略拍板**  
   - 决策内容：冻结传感器主组合、底盘能力下限、电池与热设计基线。  
   - 期望输出：`E2` 工程化基线包的冻结清单与变更准入规则。  

3. **决策C：伴生系统最小闭环优先级拍板**  
   - 决策内容：明确 App/云/后台坐席/异常升级链路的一期最小可交付面。  
   - 期望输出：`E4` 最小闭环范围文档 + 试点城市运行流程草案。  

## 4.2 需要老板批准的资源与组织动作

1. **组织动作**：成立“PDCP阻断问题战情组”（架构、硬件、软件、算法、供应链、运营联合）。  
2. **节奏动作**：未来两周采用“日更风险板+三天一次决策会”机制，而不是周会节奏。  
3. **资源动作**：为算力平台、传感器方案、夜间场景验证预留专项预算与样机资源，不与常规开发争抢。  
4. **治理动作**：所有跨域变更必须附带“成本-性能-感知影响卡”，未提交不得进入评审。  

## 4.3 两周内可执行里程碑（建议）

- **T+3天**：完成 D1 候选平台技术与成本对比评审，输出推荐主线。  
- **T+7天**：完成 D6 关键器件冻结首版，发布接口冻结与演进清单。  
- **T+10天**：完成 D7 伴生系统最小闭环定义与演练脚本。  
- **T+14天**：完成一次“端到端闭环彩排评审”（健康事件、夜间异常、安全升级三条链路）。  

## 4.4 给老板的结论话术（建议）

> 当前 Kinbot 架构方向与需求侧约束是对齐的，问题不在“该不该这样做”，而在“关键冻结项是否能在同一窗口完成收敛”。建议在未来两周集中解决 D1/D6/D7 三个阻断项，并由您拍板平台主线、器件冻结策略和伴生闭环优先级。只要这三项收敛，PDCP 评审通过与后续模块并行设计将具备可执行基础。

## 5. 评审建议结论

建议将本提案作为 `PDCP` 评审会的“管理层决策附录”，并在会前完成一次 30 分钟老板预沟通，确保评审会上聚焦决策而非重复澄清。
