# 关系中心架构提案

## 1. 文档定位

本文档是基于 REQUIREMENTS.MD 前 11 个 Step 原始需求的独立架构重审报告。作者以资深架构师身份，提出一套"关系中心"的替代架构思路。

本文档不修改任何现有文档，不替代已冻结的 PDCP 基线，而是作为另一种架构视角供决策参考。

评审日期：2026-03-11
评审范围：基于 Step 1-11 原始需求的架构重构思考
架构理念：关系中心 + 能力圈

## 2. 回到本质：这是一个什么样的产品？

### 2.1 从需求中提炼的产品本质

重读 Step 1-11，剥离所有技术细节后，我看到的产品本质是：

**这是一个能够与独居老人建立长期照护关系的智能伙伴。**

不是"一个带健康管理功能的移动机器人"，不是"一个多模态交互平台"，而是**一段关系**。

这个洞察来自三个关键需求的交叉：
- Step 3: "健康管理 > 陪伴交互"的优先级排序
- Step 9: "亲切家人型"人设，每台机器人的性格都会不同
- Step 10: "聪明、温暖、精致"的高端产品感知

这三点共同指向一个事实：**用户购买的不是功能，而是一段可信赖的照护关系。**

### 2.2 关系的三个维度

一段真实的照护关系包含三个维度：

**维度 1：了解（Understanding）**
知道这个人是谁、喜欢什么、习惯如何、健康状况怎样、今天状态是否正常。

**维度 2：陪伴（Companionship）**
在需要的时候出现、用合适的方式交流、记住重要的事情、让人感到被关心。

**维度 3：照护（Caregiving）**
发现健康风险、提醒用药、协助就医、保障安全、连接家属和医疗资源。

这三个维度不是并列的功能模块，而是同一段关系的三个面向。它们相互依存：
- 没有了解，陪伴就是空洞的
- 没有陪伴，照护就是冰冷的
- 没有照护，了解和陪伴就失去了目的

### 2.3 被忽视的架构启示

如果产品的本质是"关系"，那么架构的核心应该是**关系的建立、维持和深化**，而不是功能的堆砌。

这意味着：
- 架构的中心不应该是"决策编排"或"世界状态"，而应该是"照护关系"
- 感知、运动、交互、健康管理都是**为关系服务的能力**，而不是独立的模块
- 评价架构好坏的标准不是"功能完备性"，而是"关系质量"

## 3. 对现有架构方案的反思

### 3.1 现有 PDCP 基线架构（9 个一级模块）

**做对了什么：**
- 多尺度 OODA 准确回应了 AGI 时代的架构需求
- 双视角基线正确地将机器人本体纳入架构
- 四条业务闭环识别准确
- 安全优先级清晰

**核心问题：**
功能模块均质化，缺少"关系"这个统领性概念。9 个模块看起来同等重要，但从用户视角看，它们的价值贡献差异巨大：
- `world_state_memory` 和 `decision_orchestration` 是关系的"大脑"
- `human_health_sensing` 和 `multimodal_interaction` 是关系的"感官"
- `platform_runtime` 是关系的"身体"

把它们放在同一层级，会导致团队资源平均分配，而不是聚焦在关系质量上。

### 3.2 已有替代提案（三层七域）

**做对了什么：**
- 按时间尺度分层是一个有力的组织原则
- 算力调度提升为一级能力直击样机痛点
- 安全分层降低了实时性压力
- World State 按时间尺度分治是合理的

**核心问题：**
时间尺度是技术视角，不是用户视角。用户不关心"毫秒级反应"和"小时级优化"的区别，用户关心的是"这个机器人懂不懂我"、"它是不是真的在关心我"。

三层架构可能导致：
- 开发者按层分工，但关系是跨层的
- 层间接口成为关系流转的障碍
- 关系质量的责任被分散到三个层，没有人对整体负责

## 4. 关系中心架构：核心理念

### 4.1 架构隐喻：同心圆而非分层

传统架构喜欢用"分层"隐喻：底层、中层、上层。但关系不是分层的，关系是**辐射状**的。

我提出用**同心圆**作为架构隐喻：

```
                    ┌─────────────────────┐
                    │   生态能力圈 (E)     │
                    │  家属·云服务·医疗    │
                ┌───┴─────────────────────┴───┐
                │     知识能力圈 (K)           │
                │   记忆·学习·健康档案         │
            ┌───┴─────────────────────────────┴───┐
            │        行动能力圈 (A)                 │
            │     运动·交互·执行·储物               │
        ┌───┴─────────────────────────────────────┴───┐
        │           感知能力圈 (S)                      │
        │      视觉·语音·生命体征·环境感知              │
    ┌───┴─────────────────────────────────────────────┴───┐
    │                照护关系引擎 (R)                        │
    │         关系状态·照护策略·信任建立·个性化             │
    └───────────────────────────────────────────────────────┘
              ║                                    ║
        ══════╬════════════════════════════════════╬══════
        安全基座 (Safety Foundation)    算力基座 (Compute Foundation)
```

**核心（R）：照护关系引擎**
这是整个系统的心脏。它维护"机器人与老人之间的关系状态"，决定"此刻应该如何与这个人互动"。

**第一圈（S）：感知能力圈**
为关系提供"了解"的能力。看到人、听到声音、感知健康信号、理解环境。

**第二圈（A）：行动能力圈**
为关系提供"陪伴和照护"的能力。移动、说话、展示、递送、提醒。

**第三圈（K）：知识能力圈**
为关系提供"记忆和学习"的能力。记住过去、理解习惯、维护健康档案、优化策略。

**第四圈（E）：生态能力圈**
为关系提供"扩展和连接"的能力。连接家属、调用云服务、对接医疗、引入第三方。

**基座：安全与算力**
支撑所有能力圈的基础设施。安全保证关系不会伤害用户，算力保证关系能够流畅运转。

### 4.2 关系中心架构的三个核心原则

**原则 1：关系优先于功能**

任何功能的设计都要回答：它如何增强关系质量？

例如：
- 不是"实现跌倒检测功能"，而是"让老人感到被关注"
- 不是"实现语音交互功能"，而是"让对话像和家人聊天"
- 不是"实现用药提醒功能"，而是"让老人信任机器人的健康建议"

**原则 2：能力服务于关系**

感知、行动、知识、生态都是能力圈，它们的存在价值是**让关系引擎能够更好地工作**。

这意味着：
- 能力圈不应该有独立的"业务逻辑"，它们只提供"能力"
- 所有涉及"关系判断"的逻辑都应该在关系引擎中
- 能力圈之间不应该直接调用，必须通过关系引擎协调

**原则 3：关系是有状态的**

关系不是无状态的请求-响应，关系是**有历史、有记忆、会演化**的。

这意味着：
- 关系引擎必须维护"关系状态"（不仅仅是"系统状态"）
- 每一次交互都会改变关系状态
- 关系状态决定下一次交互的方式

### 4.3 与 OODA 的关系

关系中心架构完全兼容多尺度 OODA，但给出了不同的解读：

**OODA 不是系统的组织方式，而是关系的运行方式。**

- **Observe**：通过感知能力圈了解老人当前状态
- **Orient**：在关系引擎中理解"这个状态对这段关系意味着什么"
- **Decide**：在关系引擎中决定"此刻应该如何回应"
- **Act**：通过行动能力圈执行回应

R1-R4 四类子环对应关系的不同节奏：
- R1：关系的"本能反应"（避免伤害老人）
- R2：关系的"日常互动"（对话、陪伴、提醒）
- R3：关系的"照护任务"（健康确认、用药管理、安全巡查）
- R4：关系的"长期深化"（习惯学习、策略优化、信任建立）

OODA Scale Scheduler 位于关系引擎内部，它决定"此刻关系应该以什么节奏运行"。

## 5. 五个能力圈的详细设计

### 5.1 照护关系引擎 (R - Relationship Engine)

**定位：系统的核心，唯一对"关系质量"负责的组件**

**核心职责：**

1. **关系状态维护**
   - 当前关系阶段（初识期、熟悉期、信任期、深度依赖期）
   - 信任度评分（0-100）
   - 亲密度评分（0-100）
   - 最近交互质量评估
   - 关系健康度指标

2. **照护策略决策**
   - 此刻应该主动还是被动
   - 此刻应该关注什么（健康、情绪、安全、陪伴）
   - 此刻应该用什么语气和方式
   - 此刻是否需要升级到人工服务

3. **OODA 编排**
   - OODA Scale Scheduler（决定当前主导子环）
   - 多尺度 OODA 循环的启动、切换、中断
   - 跨子环的上下文传递

4. **个性化管理**
   - 人设参数（亲切度、俏皮度、正式度）
   - 交互风格演化
   - 个性化策略生成

5. **关系事件处理**
   - 识别关系关键时刻（第一次见面、第一次求助、第一次拒绝、第一次表扬）
   - 关系修复（当老人生气、失望、不信任时）
   - 关系深化（当老人依赖、信任、喜爱时）

**不负责：**
- 具体的感知处理（交给 S 圈）
- 具体的动作执行（交给 A 圈）
- 长期记忆存储（交给 K 圈）
- 外部系统对接（交给 E 圈）

**关键接口：**

```
interface RelationshipEngine {
  // 从感知圈接收感知事实
  on_perception(fact: PerceptionFact): void

  // 从行动圈接收执行反馈
  on_action_feedback(result: ActionResult): void

  // 从知识圈接收记忆和洞察
  on_knowledge_update(insight: KnowledgeInsight): void

  // 向行动圈发出行动意图
  emit_action_intent(intent: ActionIntent): void

  // 向知识圈请求记忆查询
  query_knowledge(query: KnowledgeQuery): KnowledgeResult

  // 向生态圈请求外部服务
  request_service(request: ServiceRequest): void

  // 获取当前关系状态（供调试和监控）
  get_relationship_state(): RelationshipState
}
```

**与现有架构的对应：**
合并了 `decision_orchestration`、`world_state_memory` 的会话状态部分、`multimodal_interaction` 的人设管理部分。

### 5.2 感知能力圈 (S - Sensing Circle)

**定位：为关系提供"了解"的能力**

**核心职责：**

1. **视觉感知**
   - 人员检测、身份识别、姿态估计
   - 跌倒检测、活动状态判断
   - 空间理解、障碍物检测、深度估计
   - 物体识别、场景分类

2. **语音感知**
   - 唤醒、VAD、ASR
   - 说话人识别、情绪识别
   - 语义理解（意图识别）

3. **生命体征感知**
   - 穿戴设备数据接入（心率、血压、血氧、步数）
   - UWB 雷达信号处理（位置、活动状态、生命体征）
   - 蓝牙医疗设备接入

4. **环境感知**
   - 智能家居事件接入（烟雾、燃气、水浸、门窗）
   - 本体感知（IMU、轮速计、触觉、电池、温度）

5. **感知融合与置信度评估**
   - 多模态融合（视觉+语音+生命体征）
   - 置信度评分
   - 异常检测

**输出：PerceptionFact**
所有感知结果统一封装为 PerceptionFact，包含：
- 事实类型（人员、姿态、语音、生命体征、环境）
- 事实内容（结构化数据）
- 置信度（0-1）
- 时间戳
- 来源传感器

**不负责：**
- 决定"这个事实意味着什么"（交给关系引擎）
- 决定"应该采取什么行动"（交给关系引擎）
- 长期趋势分析（交给知识圈）

**与现有架构的对应：**
合并了 `human_health_sensing` 的信号采集和理解部分、`mobility_navigation` 的空间感知部分、`platform_runtime` 的设备驱动部分。

### 5.3 行动能力圈 (A - Action Circle)

**定位：为关系提供"陪伴和照护"的能力**

**核心职责：**

1. **运动执行**
   - 全局路径规划与语义导航
   - 局部避障与实时调整
   - 到人、跟随、巡查等导航任务
   - 充电对接

2. **交互执行**
   - 语音合成与播放
   - 屏幕内容展示
   - 灯光表达
   - 手势识别与响应

3. **照护动作**
   - 储物仓开关控制
   - 递送任务执行（到达+开仓+确认）
   - 用药提醒执行
   - 紧急响应动作

4. **安全执行**
   - 急停、防撞、防夹手（硬件级）
   - 限速、限力矩
   - 热保护、电池保护

5. **执行反馈**
   - 动作完成状态
   - 执行过程中的异常
   - 用户响应反馈

**输入：ActionIntent**
从关系引擎接收行动意图，包含：
- 意图类型（到人、对话、提醒、递送、巡查）
- 意图参数（目标位置、对话内容、提醒事项）
- 优先级
- 超时时间

**输出：ActionResult**
执行结果反馈给关系引擎，包含：
- 执行状态（成功、失败、部分完成、被中断）
- 执行过程记录
- 用户响应（如果有）
- 异常信息（如果有）

**不负责：**
- 决定"要不要执行这个动作"（由关系引擎决定）
- 决定"用什么语气说话"（由关系引擎决定）
- 记录"这次交互的质量"（由关系引擎评估）

**与现有架构的对应：**
合并了 `mobility_navigation` 的执行部分、`multimodal_interaction` 的硬件驱动部分、储物仓控制。

### 5.4 知识能力圈 (K - Knowledge Circle)

**定位：为关系提供"记忆和学习"的能力**

**核心职责：**

1. **长期记忆管理**
   - 重要纪念日（生日、结婚纪念日、退休纪念日）
   - 重要家庭事件
   - 用户习惯和偏好
   - 对话片段归档（仅限用户主动要求记忆的）

2. **健康档案维护**
   - 历史医疗档案
   - 健康基线（正常心率、血压范围）
   - 用药计划与历史
   - 健康事件历史
   - 健康趋势分析

3. **习惯学习**
   - 作息规律识别
   - 饮食偏好学习
   - 交互偏好学习
   - 异常模式识别

4. **策略优化**
   - 提醒策略优化（什么时候提醒效果最好）
   - 交互策略优化（什么方式老人最接受）
   - 主动性策略优化（什么情况下主动交互不会打扰）

5. **知识治理**
   - 记忆写入审核
   - 记忆查询权限控制
   - 记忆删除与遗忘
   - 记忆共享（与家属 App）

**输入：KnowledgeInsight**
从关系引擎接收需要记忆的洞察，包含：
- 洞察类型（习惯、偏好、健康、事件）
- 洞察内容
- 重要性评分
- 来源（对话、观察、推理）

**输出：KnowledgeResult**
响应关系引擎的查询，包含：
- 查询结果（记忆内容）
- 置信度
- 最后更新时间
- 相关记忆推荐

**不负责：**
- 实时状态维护（由关系引擎维护）
- 决定"这个记忆是否重要"（由关系引擎决定）
- 决定"何时使用这个记忆"（由关系引擎决定）

**与现有架构的对应：**
合并了 `world_state_memory` 的长期状态部分、健康档案管理、习惯学习。

### 5.5 生态能力圈 (E - Ecosystem Circle)

**定位：为关系提供"扩展和连接"的能力**

**核心职责：**

1. **家属连接**
   - 家属 App 通信
   - 事件推送（健康异常、安全告警）
   - 远程查看请求处理
   - 远程控制指令接收

2. **云服务对接**
   - 配置同步
   - OTA 更新
   - 云端大模型调用
   - 数据备份

3. **人工服务接入**
   - 客服运营坐席转接
   - 在线问诊转接
   - 人工复核服务

4. **第三方平台对接**
   - 互联网医院接入
   - 药店与配送平台
   - 智能家居平台
   - 120 接口（预留）

5. **治理与审计**
   - 隐私合规执行
   - 审计日志记录
   - 第三方准入审核
   - 数据治理

**输入：ServiceRequest**
从关系引擎接收服务请求，包含：
- 服务类型（家属通知、人工转接、第三方服务）
- 服务参数
- 紧急程度
- 授权凭证

**输出：ServiceResult**
服务执行结果，包含：
- 服务状态（已推送、已转接、已下单）
- 外部响应（如果有）
- 预计完成时间

**不负责：**
- 决定"要不要调用外部服务"（由关系引擎决定）
- 决定"何时升级到人工"（由关系引擎决定）
- 外部服务的具体实现（由第三方负责）

**与现有架构的对应：**
合并了 `companion_service_system` 和 `observability_data_governance`。

### 5.6 两个基座能力

#### 安全基座 (Safety Foundation)

**定位：贯穿所有能力圈的安全保障**

采用分层安全策略：

**L1 硬件安全（在 A 圈内实现）**
- 急停、防撞、防夹手
- 限速、限力矩
- 热保护、电池保护
- 零延迟，硬件级响应

**L2 关系安全（在 R 引擎内实现）**
- 动作意图审批（ActionIntent → 安全评估 → ActionIntent'）
- 风险等级评估（基于当前关系状态）
- 授权校验
- 合规检查

**L3 生态安全（在 E 圈内实现）**
- 隐私策略执行
- 审计与追溯
- 第三方准入审核
- 数据治理

关键设计：**安全不阻塞关系流转**
- L1 硬件安全在 A 圈内闭环，不经过 R 引擎
- L2 关系安全在 R 引擎内同步评估，延迟可控
- L3 生态安全异步执行，不影响实时交互

#### 算力基座 (Compute Foundation)

**定位：支撑所有能力圈的计算资源调度**

核心职责：
1. **感知算力分配**：根据当前关系节奏（R1-R4）动态调整感知优先级
2. **模型加载调度**：决定当前加载哪些模型、卸载哪些模型
3. **热稳态管理**：在性能需求和热预算之间平衡
4. **与关系引擎协同**：接收 OODA Scale Scheduler 的指令

关键设计：**算力服务于关系质量**
- R1 反射环：优先保障避障和急停
- R2 执行环：优先保障导航和语音交互
- R3 任务环：优先保障 VLN 和健康检测
- R4 关系环：降低到最低功耗，保留轻量监测

## 6. 照护关系引擎的核心设计

### 6.1 关系状态模型

关系引擎维护一个多维度的关系状态：

```
RelationshipState {
  // 关系阶段
  phase: 初识期 | 熟悉期 | 信任期 | 深度依赖期

  // 关系质量指标
  trust_score: 0-100        // 信任度
  intimacy_score: 0-100     // 亲密度
  dependency_score: 0-100   // 依赖度

  // 最近交互质量
  recent_interaction_quality: 0-100
  last_positive_interaction: timestamp
  last_negative_interaction: timestamp

  // 当前关注焦点
  current_focus: 健康 | 情绪 | 安全 | 陪伴 | 无

  // 当前 OODA 子环
  current_scale: R1 | R2 | R3 | R4

  // 个性化参数
  personality: {
    warmth: 0-100,      // 温暖度
    playfulness: 0-100, // 俏皮度
    formality: 0-100    // 正式度
  }
}
```

### 6.2 关系决策流程

关系引擎的核心决策流程：

```
1. 接收感知事实 (PerceptionFact)
   ↓
2. 关系理解 (Orient in Relationship Context)
   - 这个事实对当前关系意味着什么？
   - 是否触发关系关键时刻？
   - 是否需要调整关系焦点？
   ↓
3. 关系决策 (Decide in Relationship Context)
   - 此刻应该如何回应？
   - 应该用什么方式？
   - 是否需要切换 OODA 子环？
   ↓
4. 生成行动意图 (ActionIntent)
   ↓
5. 关系安全评估 (L2 Safety)
   ↓
6. 发出行动意图到 A 圈
   ↓
7. 接收执行反馈 (ActionResult)
   ↓
8. 关系状态更新
   - 更新交互质量评分
   - 更新信任度/亲密度
   - 识别关系关键时刻
   ↓
9. 知识沉淀 (KnowledgeInsight → K 圈)
```

### 6.3 关系关键时刻识别

关系引擎特别关注"关系关键时刻"，这些时刻会显著影响关系质量：

**正向关键时刻：**
- 第一次成功帮助老人解决问题
- 老人主动表达感谢或喜爱
- 老人在困难时主动求助
- 老人愿意分享私密信息

**负向关键时刻：**
- 老人明确表达不满或生气
- 老人拒绝机器人的建议
- 老人长时间不与机器人互动
- 老人对机器人的提醒感到厌烦

**修复策略：**
当识别到负向关键时刻时，关系引擎会：
1. 降低主动性（减少主动交互频率）
2. 调整交互方式（更温和、更尊重）
3. 寻找修复机会（在合适时机道歉或解释）
4. 必要时升级到人工服务

### 7.2 接口数量对比

**现有架构：** 9 个一级模块，理论上 36 对接口，实际约 13 组一级接口

**关系中心架构：** 1 个核心引擎 + 4 个能力圈 + 2 个基座

核心接口模式：**辐射状，所有能力圈都与关系引擎交互**

```
S 圈 ←→ R 引擎 ←→ A 圈
         ↕
K 圈 ←→ R 引擎 ←→ E 圈
         ↕
    安全基座 + 算力基座
```

主要接口：
- S → R: PerceptionFact 流
- R → A: ActionIntent 流
- A → R: ActionResult 流
- R ↔ K: KnowledgeQuery / KnowledgeInsight 双向
- R → E: ServiceRequest 流
- E → R: ServiceResult 流
- 安全基座：贯穿 R/A/E
- 算力基座：贯穿 S/A/R

总计约 8 组主接口，所有接口都经过关系引擎，**关系引擎是唯一的协调中心**。

### 7.3 关键接口契约的继承

关系中心架构完全继承以下已冻结的接口契约：

1. **ActionProposal / ApprovalDecision** → 重命名为 ActionIntent / SafetyApproval，归入安全基座 L2
2. **Body Capability Contract** → 归入 A 圈的运动执行部分
3. **World State Schema** → 拆分为 RelationshipState（R 引擎）+ PerceptionFact（S 圈）+ KnowledgeBase（K 圈）
4. **健康事件七段式管线** → 归入 R 引擎的健康照护决策流程
5. **A1-A7 / F1-F7 安全枚举** → 归入安全基座

## 8. 四条业务闭环在关系架构中的走法

### 8.1 健康管理闭环

```
穿戴设备 → S 圈（生命体征感知）
         ↓
    PerceptionFact: 心率异常
         ↓
    R 引擎（关系理解）
    - 查询 K 圈：老人的健康基线
    - 评估：这是否异常？严重程度？
    - 决策：需要健康确认
         ↓
    ActionIntent: 到人确认 + 问诊式补采
         ↓
    A 圈（执行）→ 导航到人 + 语音交互
         ↓
    ActionResult: 老人回应 + 补充信息
         ↓
    R 引擎（关系决策）
    - 研判：需要用药提醒
    - 决策：送药到人
         ↓
    ActionIntent: 导航 + 开仓 + 提醒服药
         ↓
    A 圈（执行）→ 递送完成
         ↓
    R 引擎（关系更新）
    - 更新信任度（成功帮助老人）
    - 生成 KnowledgeInsight → K 圈归档
    - 生成 ServiceRequest → E 圈通知家属
```

**关键差异：** 整个健康管理流程由关系引擎编排，每一步都在关系上下文中决策。不是"执行健康管理功能"，而是"在照护关系中回应健康需求"。

### 8.2 陪伴交互闭环

```
老人说话 → S 圈（语音感知）
         ↓
    PerceptionFact: 语音内容 + 情绪 + 说话人
         ↓
    R 引擎（关系理解）
    - 查询 K 圈：最近对话历史、用户偏好
    - 评估：老人的情绪状态、意图
    - 判断：这是闲聊还是求助？
         ↓
    R 引擎（关系决策）
    - 决定回复内容（基于当前关系状态）
    - 决定回复语气（基于个性化参数）
    - 决定是否需要屏幕辅助
         ↓
    ActionIntent: 语音回复 + 屏幕展示
         ↓
    A 圈（执行）→ 语音合成 + 屏幕显示
         ↓
    ActionResult: 执行完成
         ↓
    R 引擎（关系更新）
    - 评估交互质量
    - 更新亲密度
    - 识别是否为关系关键时刻
    - 生成 KnowledgeInsight → K 圈（如果重要）
```

**关键差异：** 对话不是"多模态交互功能"，而是"关系的日常互动"。回复内容和方式由关系状态决定，而不是由对话管理模块独立决定。

### 8.3 安全保障闭环

```
硬件安全（L1）：
传感器 → S 圈 → A 圈（急停）[不经过 R 引擎]

软件安全（L2）：
S 圈（环境风险感知）→ PerceptionFact: 久卧不起
         ↓
    R 引擎（关系理解）
    - 查询 K 圈：老人作息习惯
    - 评估：这是否异常？
    - 决策：需要到人确认
         ↓
    ActionIntent: 到人确认
         ↓
    安全基座 L2（审批）
    - 评估风险等级
    - 检查授权
    - 通过
         ↓
    A 圈（执行）→ 导航到人 + 语音确认
         ↓
    ActionResult: 老人无响应
         ↓
    R 引擎（关系决策）
    - 判断：紧急情况
    - 决策：升级到家属和人工服务
         ↓
    ServiceRequest → E 圈
    - 推送家属 App
    - 转接客服坐席
    - 准备 120 联动（如需要）
```

**关键差异：** 安全不是独立的"安全模块"，而是渗透在关系的每一个环节。硬件安全在 A 圈内闭环，软件安全在关系引擎内评估，生态安全在 E 圈执行。

### 8.4 服务编排闭环

```
K 圈（习惯学习）→ KnowledgeInsight: 老人药品即将用完
         ↓
    R 引擎（关系理解）
    - 评估：是否需要主动提醒买药
    - 查询关系状态：当前信任度、最近交互质量
    - 决策：主动提醒
         ↓
    ActionIntent: 语音提醒 + 询问是否需要帮助购买
         ↓
    A 圈（执行）→ 语音交互
         ↓
    ActionResult: 老人同意
         ↓
    R 引擎（关系决策）
    - 决策：调用第三方购药服务
         ↓
    ServiceRequest → E 圈
    - 第三方准入审核（安全基座 L3）
    - 调用药店平台 API
    - 下单并跟踪
         ↓
    ServiceResult: 订单已确认
         ↓
    R 引擎（关系更新）
    - 生成 ActionIntent: 告知老人订单状态
    - 更新依赖度（老人依赖机器人的服务）
    - 生成 KnowledgeInsight → K 圈归档
```

**关键差异：** 服务不是"功能调用"，而是"关系的延伸"。是否提供服务、何时提供、如何提供，都由关系状态决定。

## 9. 关键设计决策与理由

### 9.1 为什么关系引擎必须是单一的？

**决策：** 关系引擎是唯一的协调中心，所有能力圈都与它交互，能力圈之间不直接调用。

**理由：**
1. **关系的一致性**：如果多个组件都可以做"关系决策"，关系状态会分裂，用户会感到机器人"精神分裂"
2. **责任的明确性**：关系质量只有一个责任主体，出问题时知道找谁
3. **调试的可行性**：所有关系决策都经过一个点，可以完整追踪"为什么机器人这么做"

**代价：**
- 关系引擎可能成为性能瓶颈
- 关系引擎的复杂度会很高

**缓解措施：**
- 关系引擎内部可以并行处理多个 OODA 子环
- 能力圈的执行是异步的，不阻塞关系引擎

### 9.2 为什么能力圈不能有"业务逻辑"？

**决策：** 能力圈只提供"能力"，不做"决策"。例如，S 圈只负责"检测到跌倒"，不负责"决定是否需要到人确认"。

**理由：**
1. **避免决策分散**：如果 S 圈可以决定"跌倒了就自动到人"，那关系引擎就失去了对关系节奏的控制
2. **支持个性化**：同样是"跌倒检测"，对不同老人、不同关系阶段，回应方式应该不同
3. **便于测试**：能力圈只需要测试"能力是否正确"，不需要测试"决策是否合理"

**代价：**
- 接口调用次数增加（感知 → 关系引擎 → 行动，而不是感知 → 行动）
- 延迟可能增加

**缓解措施：**
- R1 硬件安全例外：急停等硬件安全可以在 A 圈内直接闭环
- 关系引擎的决策延迟控制在毫秒级

### 9.3 为什么 World State 被拆分了？

**决策：** 不再有统一的 World State，而是拆分为 RelationshipState（R 引擎）、PerceptionFact（S 圈）、KnowledgeBase（K 圈）。

**理由：**
1. **访问模式不同**：关系状态需要毫秒级读写，长期记忆可以秒级读取
2. **一致性要求不同**：关系状态需要强一致性，长期记忆可以最终一致性
3. **生命周期不同**：关系状态是会话级的，长期记忆是持久化的

**代价：**
- 没有"一个地方查到所有状态"的便利性
- 需要明确定义什么状态属于哪个部分

**缓解措施：**
- 提供统一的查询接口（虽然底层存储分散）
- 明确的状态归属规则

### 9.4 为什么算力调度是基座而不是模块？

**决策：** 算力调度不是一个独立模块，而是贯穿所有能力圈的基座能力。

**理由：**
1. **算力是稀缺资源**：在 6000-8000 元 BOM 约束下，算力比功能更稀缺
2. **算力分配影响关系质量**：如果 VLN 推理慢，导航就犹豫，用户就会觉得机器人"不聪明"
3. **算力调度需要全局视角**：不能让每个模块自己抢算力

**代价：**
- 增加了架构复杂度
- 需要与关系引擎紧密协同

**缓解措施：**
- 算力调度器接收 OODA Scale Scheduler 的指令，不需要独立决策
- 提供默认策略，大部分情况下自动运行

## 10. 风险分析与诚实评估

### 10.1 关系中心架构的优势

1. **用户视角一致性**：架构组织方式与用户感知一致（关系 + 能力）
2. **关系质量可度量**：有明确的关系状态指标，可以量化评估
3. **个性化天然支持**：每段关系都是独立的，天然支持个性化
4. **责任主体明确**：关系质量由关系引擎负责，不会扯皮
5. **接口简洁**：辐射状接口，所有能力圈都与关系引擎交互
6. **算力优先级清晰**：算力服务于关系质量，而不是功能完备性

### 10.2 关系中心架构的风险

**风险 1：关系引擎可能过于复杂**

关系引擎承担了 OODA 编排、关系决策、个性化管理、安全审批等多重职责，可能成为"上帝对象"。

**缓解措施：**
- 关系引擎内部分层：关系状态层、决策层、编排层
- 明确的内部接口和职责划分
- 充分的单元测试和集成测试

**风险 2：能力圈的"无业务逻辑"约束可能过于理想化**

实际开发中，很难完全避免能力圈内的"小决策"。例如，S 圈的"跌倒检测"本身就包含了"什么算跌倒"的判断。

**缓解措施：**
- 允许能力圈内的"技术决策"（如阈值设定），但不允许"业务决策"（如是否需要到人）
- 明确的决策边界定义

**风险 3：关系引擎成为性能瓶颈**

所有决策都经过关系引擎，如果关系引擎处理慢，整个系统都会慢。

**缓解措施：**
- 关系引擎内部并行处理
- R1 硬件安全绕过关系引擎
- 关系引擎的决策逻辑尽量轻量化，复杂计算下放到能力圈

**风险 4：与已冻结基线的兼容性**

现有 PDCP 基线已经冻结，团队可能已经按 9 个模块开始工作。关系中心架构需要重新映射。

**缓解措施：**
- 第 7 节提供了详细的映射表
- 关键接口契约全部继承
- 可以渐进式迁移：先保留 9 个模块，逐步向关系中心架构演化

### 10.3 诚实的自我评估

关系中心架构不是"最优"架构，而是"特定视角"的架构。

**它适合的场景：**
- 产品的核心价值是"关系"而不是"功能"
- 团队认同"关系质量"作为首要指标
- 愿意为关系质量牺牲一些架构灵活性

**它不适合的场景：**
- 产品是通用机器人平台，需要支持多种应用场景
- 团队按功能域分工（健康组、导航组、交互组）
- 需要快速迭代功能，不想受关系引擎约束

对于 Kinbot OODA 项目，基于 Step 1-11 的需求分析，我认为关系中心架构是**高度契合**的，因为：
- Step 3 明确"健康管理 > 陪伴交互"，但两者都是关系的体现
- Step 9 要求"亲切家人型"人设，这是关系而不是功能
- Step 10 要求"聪明、温暖、精致"，这些都是关系质量的体现
- 目标用户是独居老人，他们需要的是"陪伴"而不是"工具"

## 11. 采纳建议

### 11.1 如果完整采纳关系中心架构

**建议的迁移路径：**

**阶段 1：概念验证（2 周）**
- 实现关系引擎的核心框架（RelationshipState + 决策流程）

- 保持现有接口契约不变

**阶段 3：关系引擎完善（4 周）**
- 实现完整的关系状态模型
- 实现关系关键时刻识别
- 实现个性化策略

**阶段 4：基座能力集成（2 周）**
- 集成算力调度器
- 完善三层安全机制

**总计：12-14 周**

**风险：** 迁移周期长，可能影响 PDCP 时间线（目标 2026-03-31）

### 11.2 如果部分采纳关系中心架构的理念

即使不完整采纳架构，以下 4 个核心理念可以融入现有架构：

**理念 1：引入"关系状态"作为一级概念**

在现有的 `world_state_memory` 中增加 RelationshipState 部分，包含：
- 信任度、亲密度、依赖度评分
- 关系阶段标识
- 最近交互质量

让 `decision_orchestration` 在决策时显式考虑关系状态。

**实施成本：** 低，约 1-2 周
**收益：** 支持个性化决策，提升用户体验

**理念 2：明确"关系质量"作为架构评价指标**

在现有的架构评审和测试中，增加关系质量指标：
- 用户满意度评分
- 交互成功率
- 信任建立速度
- 长期使用留存率

让团队在开发时不仅关注"功能是否实现"，还关注"关系是否良好"。

**实施成本：** 低，主要是流程和文化
**收益：** 团队目标对齐，避免功能堆砌

**理念 3：能力圈的"无业务逻辑"原则**

在现有模块的接口设计中，明确区分"能力接口"和"决策接口"：
- 感知模块只提供 PerceptionFact，不做业务判断
- 执行模块只接收 ActionIntent，不做决策
- 所有业务决策集中在 `decision_orchestration`

**实施成本：** 中，需要重构部分接口，约 2-3 周
**收益：** 接口清晰，便于测试和维护

**理念 4：算力调度作为一级关注点**

在 `platform_runtime` 中显式实现 ComputeScheduler，与 OODA Scale Scheduler 协同。

**实施成本：** 中，约 2-3 周
**收益：** 直接回应 Step 8 样机"反应慢"的问题

### 11.3 推荐的采纳策略

基于当前时间线（距离 PDCP 目标 2026-03-31 还有 20 天）和项目风险，我推荐：

**短期（PDCP 前）：采纳 4 个核心理念**
- 理念 1：引入关系状态（1-2 周）
- 理念 4：算力调度（2-3 周）
- 理念 2：关系质量指标（流程调整，0.5 周）

总计 3.5-5.5 周，可以在 PDCP 前完成，风险可控。

**中期（PDCP 后到 MVP 前）：渐进式架构演化**
- 理念 3：能力圈无业务逻辑（2-3 周）
- 逐步将 `decision_orchestration` 重构为关系引擎
- 逐步将其他模块重构为能力圈

**长期（MVP 后到量产前）：完整迁移**
- 如果关系质量指标验证了关系中心架构的价值
- 在二代产品中完整采纳

## 12. 与其他架构提案的对比

### 12.1 三种架构视角对比

| 维度 | 现有架构（功能模块） | 三层七域（时间尺度） | 关系中心（本提案） |
| --- | --- | --- | --- |
| 组织原则 | 功能职责 | 时间尺度 | 关系 + 能力 |
| 核心概念 | 模块 | 层 + 域 | 关系引擎 + 能力圈 |
| 一级实体数量 | 9 个模块 | 3 层 7 域 | 1 引擎 4 圈 2 基座 |
| 接口模式 | 网状（36 对理论接口） | 层间接口（8 组） | 辐射状（8 组） |
| OODA 地位 | 与模块独立 | OODA 即层 | OODA 在关系引擎内 |
| 算力调度 | 隐含在 runtime | 一级能力 | 基座能力 |
| 安全机制 | 独立门控模块 | 三层安全 | 三层安全基座 |
| World State | 统一状态平面 | 三级存储 | 拆分为关系/感知/知识 |
| 适合团队 | 按功能域分工 | 按系统层次分工 | 按关系质量分工 |
| 开发视角 | 技术视角 | 技术视角 | 用户视角 |

### 12.2 三种架构的互补性

这三种架构不是互斥的，而是互补的：

**现有架构**提供了功能完备性的保障，确保所有需求都有对应的模块负责。

**三层七域**提供了时间尺度的清晰性，确保实时性要求得到满足。

**关系中心**提供了用户价值的聚焦性，确保架构服务于关系质量。

**理想的融合方案：**
- 用关系中心的理念指导架构目标（为什么）
- 用三层七域的时间尺度指导性能设计（何时）
- 用现有架构的功能模块指导实现组织（如何）

## 13. 总结

### 13.1 核心观点

1. **产品本质是关系**：Kinbot OODA 不是功能机器人，而是照护关系的载体
2. **架构应服务于关系质量**：评价架构好坏的标准是关系质量，而不是功能完备性
3. **关系引擎是核心**：应该有一个统一的组件对关系质量负责
4. **能力圈服务于关系**：感知、行动、知识、生态都是为关系服务的能力
5. **算力和安全是基座**：支撑关系运转的基础设施

### 13.2 关键差异

与现有架构的核心差异：
- 现有架构：9 个功能模块平等协作
- 关系中心：1 个关系引擎 + 4 个能力圈，关系引擎是唯一协调中心

与三层七域的核心差异：
- 三层七域：按时间尺度分层，技术视角
- 关系中心：按关系+能力组织，用户视角

### 13.3 采纳建议

**推荐策略：** 短期采纳 4 个核心理念，中长期渐进式演化

**关键理念：**
1. 引入关系状态作为一级概念
2. 明确关系质量作为架构评价指标
3. 能力圈的无业务逻辑原则
4. 算力调度作为一级关注点

### 13.4 最后的思考

架构不是目的，关系才是。

无论采用哪种架构，最终的评判标准只有一个：**老人是否感到被关心、被理解、被照护**。

如果架构能让开发者在写每一行代码时都想着"这会如何影响关系质量"，那就是好架构。
