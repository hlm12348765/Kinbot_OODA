## 1. 文档定位

本文档用于承接以下三份文档之间的差异分析与收敛建议：

1. `docs/08_reviews/16_family_co_living_intelligence_architecture_innovation_notes.md`
2. `docs/08_reviews/17_from_multi_scale_dynamic_ooda_to_family_co_living_agent.md`
3. `docs/00_governance/05_system_architecture_principles.md`

本文档的目标不是直接冻结新主线，而是回答三个问题：

1. `16` 与 `17` 的设计意图到底哪里相同、哪里不同；
2. 它们与当前 `05_system_architecture_principles.md` 的契合度如何；
3. 下一步如何把当前讨论收敛成**正式可执行**，但又**不越权提前冻结**的革新路线。

本文档当前仍属于评审输入，不直接替换现有主线文档。

## 2. 核心结论

当前判断可先压缩成 `4` 句：

1. `16` 与 `17` **不是对立提案**，而是处在不同层次的两份文档：`16` 负责上游问题重写，`17` 负责下游改造计划。
2. `16` 的强项是重新定义问题、产品北极星和原则边界；`17` 的强项是把革新翻译成仓库级文档改造动作。
3. 当前最大问题不是两份文档方向冲突，而是 `17` 把若干仍停留在 `provisional` 的判断，提前写成了执行前提。
4. 因此，正式可执行路线不应直接照搬 `17`，而应先插入一个**收敛闸门**：先冻结母命题、顶层原则、`OODA` 运行时定位和影响边界，再启动文档波次改造。

## 3. `16` 与 `17` 的设计意图异同

### 3.1 共同点

两份文档在以下方向上高度一致：

1. 都认为当前问题不是继续补强旧 `OODA` 主线，而是重写下一代家庭机器人总框架。
2. 都接受 `OODA` 从总架构中心退到运行时层，不再承担唯一上位方法论角色。
3. 都把“家庭共居智能体”作为比“更强的一代工具型机器人”更高一级的产品定义。
4. 都接受新的顶层原则群，包括生活连续性、具身在场、关系形成、主体性保护、多角色责任网络和低打扰共居。
5. 都在方向上支持把旧主线改写为“家庭共居智能体 + 多执行范式”的系统。

### 3.2 根本差异

| 维度 | `16_family_co_living...` | `17_from_multi_scale...` | 当前判断 |
| --- | --- | --- | --- |
| 主要回答的问题 | 为什么必须革新，以及革新应上抬到哪一层 | 如果革新成立，仓库中的文档该如何改 | 前者偏上游问题定义，后者偏下游执行排程 |
| 文档角色 | 讨论纪要、母命题输入、阶段性决议 | 改造计划、波次拆解、文档清单 | 两者不是替代关系，而是前后链路关系 |
| 成熟度口径 | 明确大量结论仍为 `provisional` | 默认母命题、`7` 条原则、`9 -> 7` 实体重组已可执行 | `17` 的成熟度假设明显比 `16` 更激进 |
| 对 `OODA` 的处理 | 强调降阶、跨环融合、`Orient + Decide` 融合和端到端趋势 | 强调降阶为离散决策范式，并通过执行范式协调器兼容多范式 | `16` 更强调前瞻运行时趋势，`17` 更强调可落地的协调框架 |
| 对当前主线边界的态度 | 明确不直接废弃 `PDCP / KBT-32`，只作为新增评审输入 | 直接提出重写一批主线文档 | `17` 缺少一个“何时越过当前主线边界”的阶段门 |
| 对数据 / 状态模型的态度 | 暂未冻结 `Home / Relation / Self`、顶层分解、验证指标 | 已直接推进 `World State 9 -> 7` 与 `CareRelationship / CareEvent` | `17` 把数据模型收敛放得过早 |
| 对中国现实与哲学基础 | 明确展开现代 / 后现代 / 中国分布式家庭语境 | 几乎不展开，只继承结论 | `16` 提供合法性来源，`17` 提供改造动作 |
| 对 `V1` 反推 | 明确指出还未完成 | 几乎没有展开，默认进入主线重写 | `17` 对 `V1` 收缩版支撑不足 |
| 对验证与阶段门 | 明确指出尚未定义 | 仅给出 grep 和索引级验证 | `17` 的验证仍偏文档工程，不足以支撑产品级冻结 |

### 3.3 更深一层的意图差异

如果进一步压缩：

- `16` 的设计意图，是为 Kinbot 建立**新的问题陈述**、**新的产品北极星**和**新的原则语境**。
- `17` 的设计意图，是把这个新语境翻译成**可执行的主线重写计划**。

因此，`16` 更像“架构问题重写稿”，`17` 更像“仓库迁移施工图”。

## 4. 与当前 `05_system_architecture_principles.md` 的契合度

### 4.1 当前 `05` 的角色判断

当前 `docs/00_governance/05_system_architecture_principles.md` 的特点是：

1. 它是**通用系统工程原则集**，而不是 Kinbot 专属原则集。
2. 它对复杂系统分解、歧义消解、架构决策排序、进化与健壮性有帮助。
3. 它不足以直接回答家庭共居智能体所特有的问题，例如：
   - 分布式家庭中的远程在场
   - 照护责任网络
   - 主体性与尊严保护
   - 低打扰共居
   - `OODA` 降阶后的多范式运行时

所以当前 `05` **不是错误**，但**不够**。

### 4.2 契合度对照

| 原则簇 | `16` 契合度 | `17` 契合度 | 说明 |
| --- | --- | --- | --- |
| 系统问题陈述 / 歧义 / 聚焦 / 创新 | 高 | 中 | `16` 直接重写问题陈述，符合 `05` 中“系统问题陈述原则”“歧义原则”“创新原则”；`17` 默认问题已收敛，更像在问题之后直接进入施工 |
| 价值 / 受益 / 平衡 | 高 | 中高 | `16` 把价值中心从任务成功率改为生活连续性、关系与尊严，明显强化价值-架构耦合；`17` 延续该方向，但对 `V1`、成本、阶段门的平衡不足 |
| 分解 / 表面复杂度 / 必备复杂度 / `2下1上` | 中 | 高 | `16` 尚未完成稳定分解；`17` 已有波次拆解与文档级分层，更贴近 `05` 的分解和排序方法 |
| 产品进化 / 健壮程度 / 遗留元素复用 | 中高 | 高 | `16` 明确强调 `V1 -> V2` 反推；`17` 明确保留旧 `OODA` 子环、保留 `9` 模块、保留历史文档，复用意识更强 |
| 与特定解决方案无关的功能 | 高 | 中低 | `16` 保持开放问题较多，不急于把新框架绑定到单一数据模型；`17` 过早绑定 `9 -> 7` 实体重组和特定命名，存在“先解法、后冻结”风险 |
| 架构决策排序 / 耦合整理 | 中 | 高 | `17` 的 `Wave 1-5` 序列很适合后续执行，但前提是先补一个正式收敛闸门 |
| 二元原则 / 整体原则 | 中 | 中 | 两者都承认机器人既是本体产品，又是信息系统的一部分，但都尚未把“产品实体架构视图 + 运行时功能架构视图”的继承关系写透 |

### 4.3 契合度总判断

综合判断如下：

1. `16` 更契合 `05` 中“问题陈述、歧义消解、创新与价值重写”的那一半。
2. `17` 更契合 `05` 中“分解、排序、演进和文档施工”的那一半。
3. 真正的缺口不在 `16` 或 `17` 本身，而在于当前 `05` 缺少 Kinbot 专属的原则层，导致上游问题重写与下游改造计划之间缺少一个中间承接面。

因此，当前最合理的做法不是直接让 `17` 落仓，而是先按 `17` 自己的建议，**优先改写 `05_system_architecture_principles.md`**，把 Kinbot 专属原则层补出来。

## 5. 直接按 `17` 执行的主要风险

若直接按 `17` 作为施工计划推进，当前至少有 `5` 个风险：

1. **把未冻结判断提前写成主线前提。**  
   `16` 明确还未冻结的新总框架名称、顶层分解、`V1` 反推和验证口径，在 `17` 中被部分默认成立。

2. **过早进入数据模型重组。**  
   `World State 9 -> 7`、`CareRelationship`、`CareEvent` 这些动作已经是结构级重写，理论上应晚于总框架和运行时层的冻结。

3. **对当前主线边界冲击过大。**  
   `16` 强调当前只是新增评审输入；`17` 直接进入大规模重写，容易造成 `PDCP / KBT-32` 还未完成清账就被主线重写抢跑。

4. **运行时趋势表达仍不够前瞻。**  
   `17` 已引入“多执行范式”，这是对的；但它仍较强依赖“协调器中心”的表述，对 `Orient + Decide` 融合、局部端到端化和分层趋势的表达还不够充分。

5. **执行清单存在仓库边界问题。**  
   当前 `17` 中至少有两项不能直接原样执行：
   - `Wave 4.1` 提到更新 `CLAUDE.md`，但按当前仓库协作规则，除非用户明确要求，否则不应直接修改该文件。
   - `Wave 3.5` 提到更新 `docs/03_p2_feasibility/08_post_freeze_coding_work_plan.md`，而当前仓库中并不存在这个文件，现有 `08` 文件为 `08_Kinbot_nighttime_closed_loop_plan.md`。

## 6. 正式可执行的收敛路线建议

### 6.1 总原则

当前推荐的路线不是“直接执行 `17`”，而是：

**以 `16` 作为上游问题与原则输入，以 `17` 作为下游候选施工图，中间新增一个“收敛闸门”，再进入正式改造。**

### 6.2 推荐阶段划分

#### Phase 0：革新前提收敛

目标：把 `16` 中的上位判断与 `17` 中的执行前提对齐。

本阶段必须明确冻结的只有 `4` 件事：

1. 新架构母命题是否接受；
2. 新 `7` 条顶层原则是否接受；
3. `OODA` 是否正式退到运行时层，以及是否允许跨环融合；
4. 当前对现有 `PDCP / KBT-32` 的影响边界。

本阶段**不应直接冻结**：

1. `World State 9 -> 7` 是否成立；
2. `CareRelationship / CareEvent` 是否作为正式实体进入主线；
3. 新总框架的最终命名；
4. `V1` 的最终收缩版定义。

建议产出：

- 当前这份 `18_...md` 作为差异梳理与收敛建议；
- 用户确认后，再形成正式“革新前提收敛包”。

#### Phase 1：原则层冻结

目标：把当前通用 `05_system_architecture_principles.md` 改写成 Kinbot 可执行的原则层。

建议动作：

1. 重写 `docs/00_governance/05_system_architecture_principles.md`；
2. 主体改为 Kinbot 专属母命题与 `7` 条顶层原则；
3. 将当前通用系统工程原则保留为附录或附加检查表，而不是全部丢弃。

这是最关键的一步，因为它是从“泛系统工程原则”过渡到“家庭共居智能体原则层”的正式桥梁。

#### Phase 2：总架构与运行时层重组

目标：先重写总图，再动数据模型。

建议动作：

1. 新建 `docs/02_p1_architecture/14_family_co_living_agent_paradigm.md`，承接新总框架定义；
2. 更新 `docs/02_p1_architecture/01_overall_architecture.md`，把双视角基线与新总框架对齐；
3. 更新 `docs/02_p1_architecture/03_multi_scale_dynamic_ooda_architecture_baseline.md`：
   - 可先改标题与内容，不必立刻物理改名；
   - 先完成“`OODA` 降阶为离散决策范式之一”的表达；
   - 再补连续流式、事件驱动和长周期演化三种执行范式；
4. 在运行时表达中明确写入：
   - `Orient + Decide` 融合是允许的前瞻方向；
   - 局部端到端化可以存在；
   - 但系统级安全、授权、恢复、审计边界不得被穿透。

#### Phase 3：状态模型与数据模型决策

目标：在总图稳定后，再决定 `World State` 是否重组。

建议动作：

1. 把 `World State 9 -> 7` 作为单独评审决策，而不是自动继承自母命题；
2. 只有当总架构、运行时层和 `V1` 支撑关系明确后，才决定是否：
   - 新建 `15_care_relationship_and_event_model.md`
   - 重写 `05_world_state_schema.md`
3. 若当轮评审认为 `9 -> 7` 仍过早，应允许先采用“关系 / 事件扩展层”方案，而不是强制立即重组实体主表。

#### Phase 4：下游方案与评审包同步

目标：把新总架构映射回现有主线。

建议动作：

1. 更新 `02_pdcp_system_architecture_review_package.md`
2. 更新 `04_module_layers_and_boundaries.md`
3. 更新 `06_decision_state_machine.md`
4. 更新 `03_p2_feasibility/01_overall_solution_and_module_design_baseline.md`
5. 更新 `docs/08_reviews/02_architecture_principles_alignment_check.md`
6. 与 `KBT-32 / KBT-33` 显式建立映射关系

#### Phase 5：验证口径与治理闭环

目标：把新框架从“概念成立”推进到“可验收”。

建议补入至少 `5` 类验证指标：

1. 生活连续性：异常后是否能恢复、服务是否断线；
2. 低打扰共居：是否尊重空间礼仪、时机与情绪状态；
3. 主体性保护：授权、拒绝、纠错、删除、解释是否可用；
4. 多角色责任网络：老人、子女、保姆、平台之间的权限与升级链是否闭环；
5. 关系质量与长期信任：是否能长期形成可靠、克制、可恢复的互动关系。

## 7. 对 `17` 的执行项分类建议

### 7.1 当前可直接进入执行准备的项

1. 重写 `05_system_architecture_principles.md`
2. 新建 `14_family_co_living_agent_paradigm.md`
3. 重组 `01_overall_architecture.md`
4. 重组 `03_multi_scale_dynamic_ooda_architecture_baseline.md`
5. 补做运行时跨环融合与多范式表达

### 7.2 当前应保留为 `provisional` 的项

1. `World State 9 -> 7`
2. `CareRelationship / CareEvent` 正式实体化
3. 四个业务闭环是否全部改名为“照护关系的四个面向”
4. `03_...baseline` 是否物理改名

### 7.3 当前不应直接执行的项

1. 未经用户明确授权直接修改 `CLAUDE.md`
2. 更新当前仓库中并不存在的 `docs/03_p2_feasibility/08_post_freeze_coding_work_plan.md`

## 8. 当前建议的正式判断

如果要把当前讨论压成一句正式判断，我建议写成：

**`16` 应作为本轮革新的上游问题重写与原则输入；`17` 应作为候选文档改造计划，但在进入主线前必须先经过“原则层冻结 -> 总图重组 -> 数据模型决策 -> 下游映射 -> 验证口径闭环”这五步收敛，不得把仍属 `provisional` 的判断直接写成执行前提。**

## 9. 下一步最小动作

若按当前收敛建议继续推进，下一步最小动作应是：

1. 先重写 `docs/00_governance/05_system_architecture_principles.md` 的结构方案；
2. 明确 `17` 中哪些项进入正式执行、哪些继续保持 `provisional`；
3. 再开始写 `14_family_co_living_agent_paradigm.md` 的正式骨架。

这三步完成后，革新路线才算真正从“宏观讨论”进入“正式施工准备”。
