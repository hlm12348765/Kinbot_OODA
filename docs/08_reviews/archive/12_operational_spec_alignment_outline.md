## 2. 对齐原则

运行规范后续修订建议遵循以下 6 条原则：

1. **系统边界以主线架构为准**  
   运行规范只能表达已冻结的产品系统边界，不能自行把角色模型、数据边界、传感器边界和服务边界重新定义一遍。

2. **运行规范优先表达运行概念，而不是实现参数**  
   在 `PDCP` 阶段，运行规范应优先表达角色、空间边界、授权边界、任务线程、运行模式和合规要求，而不是过早冻结器件级或控制级数值。

3. **遵守“与特定解决方案无关的功能原则”**  
   运行规范不能过早把系统问题收窄成某一组速度、距离、阈值和器件解；这些内容更适合下沉到 `S1` 工程规格与验证门。

4. **遵守“表面复杂度 / 必备复杂度原则”**  
   规范应帮助团队理解必要边界，而不是引入另一套更粗糙但同样复杂的角色模型和区域规则。

5. **主价值链必须与当前产品排序一致**  
   当前一代主价值排序已冻结为：健康管理 > 陪伴交互 > 家庭安全巡护 > 老人看护。运行规范不能把健康主链降格为轻量提醒系统。

6. **隐私与安全要按“默认端侧 + 受控例外”表达**  
   当前主线不是“绝不回流”，而是“默认端侧处理，必要时通过受控回流预留接口处理”；也不是“隐私区域完全不监测”，而是“高敏感区域更谨慎、更细粒度地监测与授权”。

## 3. 本轮审查总判断

当前《Kinbot运行规范》最大的价值在于：

1. 它已经具备“家庭机器人运行规范”应有的法律与治理语气。
2. 它对隐私、伦理、人机边界、夜间打扰、访客尊重等内容有较好的产品经理视角表达。

当前最大问题不在写作形式，而在**版本时序落后于架构主线**，表现为：

1. 角色模型仍停留在“管理员 / 使用者 / 访客”。
2. 数据边界仍停留在“敏感数据仅本地、最长 30 天、不上云”。
3. 健康功能仍停留在“监测 / 提醒 / 分析”。
4. 空间边界仍停留在“隐私区域默认排除监测”。
5. 运行规范提前冻结了大量工程数值。

因此，这份规范不适合直接作为当前一代的下发基线，但适合在本提纲指导下修订为“面向 `PDCP` 基线的运行规范版本”。

### 3.1 当前不是“内容不够”，而是文档定位失配

从当前仓库状态看，这份《Kinbot运行规范》实际上同时在尝试承担 3 种不同层级的职责：

1. 未来家庭机器人愿景级约束
2. Kinbot 产品家族级运行原则
3. Kinbot 一代产品的可执行运行规范

而它当前又没有在任何一个层级上站稳：

1. **作为未来家庭机器人的设想**  
   它还不够宏观，也不够大胆，无法承载“未来家庭机器人如何参与下一代人机共存家庭生活”的愿景表达。

2. **作为 Kinbot 产品家族级约束**  
   它没有稳定抽象出跨代际、跨 `SKU` 的共性原则，很多条款仍然带着当前一代未对齐的实现口径。

3. **作为 Kinbot 一代运行规范**  
   它又明显与当前需求输入、`PDCP` 架构基线和总体方案基线脱节，既存在主线落后问题，也混入了过早冻结的工程参数。

因此，当前版本不是一份可以直接下发执行的正式规范，更像是一份**待拆解、待重新定位的过渡稿**。

### 3.2 当前先不新建文档，而是在本提纲中拆出三层目标

基于上述定位失配，本提纲建议先在逻辑上拆出 3 层目标，用于指导下一轮修订，而不是继续把所有问题都塞回一份文档里：

1. **未来家庭机器人愿景 / 宪章层**  
   回答“未来家庭机器人将如何参与家庭生活、承担何种角色、遵守什么伦理与共存边界”，应更宏观、更大胆，不写具体工程数值。

2. **Kinbot 产品家族级运行原则层**  
   回答“无论一代、二代还是不同 `SKU`，Kinbot 家族共同遵守什么”，应沉淀跨代稳定的角色边界、数据边界、隐私与安全原则、关系边界和物理服务边界。

3. **Kinbot 一代运行规范层**  
   回答“当前一代产品到底如何运行、如何授权、如何监测、如何升级”，必须紧贴 `PDCP`、当前需求与当前技术路线。

当前这份提纲的作用，就是先把现有条款分流到这三层中的合适位置，再决定：

1. 哪些内容适合上提为“愿景 / 宪章”
2. 哪些内容适合保留为“家族原则”
3. 哪些内容应留在“一代运行规范”
4. 哪些内容应继续下沉到工程规格、验证门和模块方案

## 5. 建议保留的条款

以下内容建议继续保留，只做术语和口径对齐：

### 5.1 总则与原则

建议保留：

1. 目的与宗旨
2. 适用范围
3. 不伤害原则
4. 隐私保护原则
5. 人机和谐原则
6. 邻里和睦原则
7. 行为规范与法律约束

需要微调的地方：

1. “家庭机器人”应更明确绑定 Kinbot 当前产品边界，而不是泛泛覆盖所有家庭机器人。
2. “合理预期行为”需要和当前 `ConOps / OpsCon` 的运行线程、夜间静默、服务升级口径对齐。

### 5.2 交互原则

建议保留：

1. 可预测性
2. 个性化
3. 可理解性
4. 尊重自主
5. 情感边界
6. 主动交互避免无关联随机打扰
7. 用户可设免打扰时段
8. 不做无法兑现的情感承诺

需要微调的地方：

1. “多人对话不插话、不打断”建议改成“默认不插话、不打断；异常与安全事件按授权边界可突破”。
2. 交互方式优先级不能再用“管理员优先”表述，需要对接多角色权限基线。

### 5.3 法律与政治敏感性约束

建议保留：

1. 不监视、不骚扰、不协助违法操作
2. 敏感话题规避与拒绝响应
3. 合规更新机制

需要微调的地方：

1. “不存储与上传”应和数据治理主线一起改写，避免出现单一议题使用绝对口径、而整体数据边界又不是这个口径的割裂。

## 6. 建议改写的条款

### 6.1 角色模型与权责划分

当前问题：

1. “管理员 / 使用者 / 访客”的三类模型过粗。
2. “管理员优先”会直接覆盖老人本人最高权限、子女同权、冲突二次确认、保姆模式等已冻结基线。

建议改写为：

1. 角色集合至少明确为：
   - 老人本人
   - 子女
   - 保姆 / 照护者
   - 访客
2. 权限原则至少明确为：
   - 老人本人为最高权限主体
   - 子女可在绑定与授权后同权
   - 同权冲突必须二次确认
   - 保姆模式是独立协同模式
   - 访客仅保留基础互动权限

### 6.2 数据边界

当前问题：

1. “敏感数据仅本地存储、最长 30 天、不上云”与长期记忆、医疗档案、家属协同、受控回流预留相冲突。

建议改写为：

1. 默认端侧处理、端侧加密存储。
2. 原始敏感数据不默认回流。
3. 结构化长期记忆、健康档案、用药记录应允许在授权和治理边界内长期保存。
4. 仅在授权、脱敏、加密、用途受限、时效受限条件下，允许受控回流预留能力。

### 6.3 传感器关闭规则

当前问题：

1. 一刀切的摄像头 / 麦克风关闭规则会破坏一代纯视觉 `L1` 安全闭环。

建议改写为：

1. 区分“核心安全感知链路”和“隐私增强感知链路”。
2. 允许关闭非核心感知或非必要交互通道。
3. 不允许用户关闭会破坏夜间静默守护、近场安全和身份安全闭环的核心感知链。
4. 隐私区域的规则要改成“更谨慎、更细粒度授权”，而不是“默认不监测”。

### 6.4 空间边界与区域权限

当前问题：

1. 卧室、卫生间统一归为隐私区域，并默认排除安全监控，会直接削弱健康管理和家庭安全巡护。

建议改写为：

1. 公共区域
2. 高敏感区域
   - 卧室
   - 卫生间
3. 危险禁区

并进一步明确：

1. 卧室与床边需要更细粒度的夜间守护边界。
2. 卫生间默认高敏感、默认谨慎进入。
3. 入户门不越界。
4. 异常与授权可突破静默，但必须留审计。

### 6.5 健康与安全功能

当前问题：

1. 健康功能已经明显低于当前产品定义深度。
2. 安全功能没有充分体现夜间静默、升级链和服务协同。

建议改写为：

1. 健康功能至少拆成：
   - 信号采集
   - 健康事件候选
   - 补采与问询
   - 风险分级
   - 用药提醒 / 递送 / 确认
   - 医疗档案与在线问诊协同
2. 安全功能至少拆成：
   - 本体运动安全
   - 家庭风险发现
   - 夜间静默守护
   - 异常升级与用户取消
   - 人工与外部服务升级

## 7. 建议下沉到工程规格的条款

以下内容当前不适合留在运行规范层，建议下沉到 `S1` 工程规格、验证门或后续规格书：

### 7.1 运行安全数值

建议下沉：

1. 最大速度
2. 制动距离
3. 停止时间
4. 接触压力
5. 跌落阈值
6. 儿童模式具体速度与距离

原因：

1. 当前仍在 `PDCP` 阶段。
2. 这些数值会反向锁死底盘、控制和验证方案。
3. 它们应由 `S1` 与 `S5` 联合给出可验证指标，而不是由运行规范提前冻结。

### 7.2 环境适应参数

建议下沉：

1. 门槛高度
2. 轨道缝隙
3. 坡度
4. 地面落差
5. 夜间噪声数值
6. 充电站物理距离数值

原因：

1. 它们属于整机规格与验证门。
2. 当前主线仍在做底盘路线、头部结构和整机收敛。

### 7.3 机械与安规细节

建议下沉：

1. 倒角半径
2. 急停按键尺寸、位置等细节
3. 具体防水等级

原因：

1. 这些都属于工程规格与工业设计 / 结构 / 安规协同范畴。

## 8. 建议新增的条款

为了与当前主线架构对齐，建议未来修订运行规范时新增以下章节或条款：

1. **产品系统边界条款**
   - 明确 Kinbot 是“机器人本体 + 伴生系统 + 服务”的完整产品系统。
2. **运行模式条款**
   - 日常陪伴
   - 健康守护
   - 家庭安全
   - 夜间静默守护
   - 服务升级
3. **多角色权限条款**
   - 老人本人 / 子女 / 保姆 / 访客
4. **数据治理条款**
   - 默认端侧处理
   - 长期记忆可治理
   - 受控回流预留
5. **纯视觉边界条款**
   - 核心安全感知不得被简单关闭
   - 主动传感不作为产品 fallback
6. **Agent 增强平面条款**
   - 长期记忆
   - 技能化能力组织
   - 连接器抽象
   - 受控任务编排

## 9. 修订顺序建议

建议按以下顺序修订运行规范，而不是一次性重写：

1. **第一轮：先修正与主线冲突的高风险条款**
   - 角色模型
   - 数据边界
   - 传感器关闭规则
   - 隐私区域 / 危险区域 / 监测边界
   - 健康功能深度

2. **第二轮：把工程数值和实现细节从运行规范剥离**
   - 迁移到工程规格、模块方案和验证门

3. **第三轮：补入与当前主线一致的系统级条款**
   - `ConOps / OpsCon`
   - 多角色权限
   - 夜间静默守护
   - 服务升级
   - Agent 增强平面
