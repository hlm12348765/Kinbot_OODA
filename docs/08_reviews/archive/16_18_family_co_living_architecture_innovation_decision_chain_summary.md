## 1. 文档定位

本文档用于把 `16`、`17`、`18` 三份文档串成一条可追溯的决策链。

它不再承担主线评审职责，只回答三个问题：

1. 为什么要把 `OODA` 重新审视为运行时而不是总架构中心；
2. 为什么“家庭共居智能体”成为新的母命题；
3. 为什么收敛路线应先冻结原则与总图，再决定数据模型和迁移步骤。

## 2. 三份文档各自承担的角色

### 2.1 `16`：问题重写

`16_family_co_living_intelligence_architecture_innovation_notes.md` 的核心价值是：

1. 重新定义问题；
2. 把北极星从“工具型机器人”抬升到“家庭共居智能体”；
3. 说明 `OODA` 不能继续充当唯一总架构中心；
4. 提供现代 / 后现代 / 中国现实语境下的合法性来源。

### 2.2 `17`：施工计划

`17_from_multi_scale_dynamic_ooda_to_family_co_living_agent.md` 的核心价值是：

1. 把新的问题定义翻译成仓库级文档改造计划；
2. 把总框架、运行时和数据模型拆成可执行波次；
3. 把 `OODA` 降阶为离散决策范式，并引入多执行范式表达。

### 2.3 `18`：收敛建议

`18_family_co_living_architecture_diff_and_convergence_plan.md` 的核心价值是：

1. 比较 `16`、`17` 与当前系统原则的契合度；
2. 指出 `17` 把仍属 `provisional` 的判断写得过早；
3. 提出一个“先收敛闸门、再做施工”的顺序；
4. 明确主线不应直接跳到数据模型重组。

## 3. 这条决策链的最终结论

这三份文档合起来，最终形成的结论可以压缩成四句：

1. Kinbot 下一代方向应从“多尺度 OODA 机器人”转向“家庭共居智能体”；
2. `OODA` 应从总架构中心退到运行时层；
3. 任何数据模型重组都必须晚于原则层和总图层的冻结；
4. 主线推进应采用“问题重写 -> 原则冻结 -> 总图重组 -> 数据模型决策 -> 下游映射 -> 验证闭环”的顺序。

## 4. 为什么最终没有直接执行 `17`

`17` 之所以没有直接成为主线施工图，不是因为方向错，而是因为它默认了太多尚未冻结的前提：

1. 新总框架名称尚未完全冻结；
2. `V1` 收缩版尚未完全定义；
3. `World State 9 -> 7` 的结构性成本尚未被充分评估；
4. 验证口径尚未先于迁移动作冻结。

因此，`17` 更适合作为候选施工图，而不是直接执行的主线命令。

## 5. 归档说明

这份总结用于把历史链条保留下来，便于以后回看：

1. `16` 负责问“为什么要改”；
2. `17` 负责问“如果要改，怎么改”；
3. `18` 负责问“什么时候改、先改到哪一步最稳妥”。

归档到这里后，主线入口不再需要反复翻回这三份文档，但它们仍然为后续架构决策保留完整追溯链。
