## 1. 文档定位

本文档承接 `Step 47` 中对 `Phase 3` 的最新要求。

当前不再把路线 A 视为主推方向，而将路线 B 提升为当前目标模型方向。本文档用于回答：

1. 七实体 `World State` 到底是什么；
2. 现有 `9` 类一级实体中哪些应合并、哪些应升级、哪些应保留；
3. `V1` 在不做完整详细设计的前提下，如何只激活其中的最小子集；
4. `CareEvent` 与 `Task` 的边界如何避免再次混淆。

## 2. 当前正式方向

根据 `Step 47` 的最新更新，当前判断收敛为：

1. 路线 A 只可作为对照和风险说明，不再作为当前主方向；
2. 路线 B 才与已冻结的原则层、总架构和运行时层保持一致；
3. 当前应优先冻结正确的概念模型，再决定 `V1` 运行时只激活其中哪些子集。

## 3. 七实体目标模型

当前建议的 `7` 类一级实体为：

1. `Person`
2. `CareRelationship`
3. `Household`
4. `Place`
5. `Object`
6. `Task`
7. `CareEvent`

这 `7` 类对象分别回答不同问题：

| 实体 | 它回答的问题 |
| --- | --- |
| `Person` | 家庭里有哪些具体的人 |
| `CareRelationship` | 人与人、人与你、人与平台之间是什么照护关系、信任层级和权力传递 |
| `Household` | 这是怎样一个家庭系统、有哪些成员、有哪些外部接力网络 |
| `Place` | 事情发生在什么空间语义位置 |
| `Object` | 家庭里有哪些重要物件，包括药物与设备 |
| `Task` | 系统准备做什么、做到哪一步 |
| `CareEvent` | 已经发生了什么、被判断发生了什么、当前事件处于什么阶段 |

### 3.1 一级边界表

| 实体 | 包含什么 | 不包含什么 |
| --- | --- | --- |
| `Person` | 身份、位置、偏好、健康状态摘要、设备绑定摘要 | 关系授权、任务状态、事件生命周期 |
| `CareRelationship` | 关系角色、授权范围、二次确认、权力传递、升级优先级 | 已发生事件、执行步骤、空间状态 |
| `Household` | 家庭成员集合、`care_network`、家庭模式、应急策略、隐私策略 | 单个人的授权细节、单个事件的生命周期 |
| `Place` | 空间语义、拓扑锚点、可达性、夜间规则 | 人物关系、任务执行轨迹 |
| `Object` | 重要物件、药物与设备子类型、仓位与状态 | 权限仲裁、事件生命周期 |
| `Task` | 执行目标、执行步骤、审批状态、过程追踪 | 对已发生事实的最终判定 |
| `CareEvent` | 已发生 / 被判断已发生的照护事件、证据、推荐动作类型、关联升级链 | 具体执行步骤、话术、导航目标、重试策略 |

## 4. 从 `9` 到 `7` 的收敛规则

### 4.1 `Person <- HealthProfile`

当前判断：

- `HealthProfile` 与 `Person` 是天然 `1:1`；
- 它并不作为一个脱离人的独立本体存在。

因此推荐：

- 将 `HealthProfile` 从一级实体降为 `Person.health_state` 或同级受治理子结构；
- 病历摘要、慢病、过敏、穿戴绑定等仍保留，但不再独立占一个顶层槽位。

### 4.2 `Object <- MedicationAsset`

当前判断：

- 药物在物理世界中本质上是一类特殊物件；
- 它的差异主要在属性，而不是本体层级。

因此推荐：

- 将 `MedicationAsset` 并入 `Object`；
- 用 `object_type=medication` 或等价分类表达其特殊属性；
- 剂量、有效期、仓位、电动仓门策略等保留为对象的专属字段组。

### 4.3 `CareRelationship <- RoleBinding`

当前判断：

- `elder / child / caregiver / visitor` 本质是关系角色；
- `auth_scope / delegated_by / requires_confirmation` 本质是关系中的权限、信任与权力传递。

因此推荐：

- 用 `CareRelationship` 替代 `RoleBinding`；
- `RoleBinding` 只保留为迁移阶段或下游实现中的投影视图，不再作为目标模型中的一级实体。

### 4.4 `CareEvent <- RiskEvent`

当前判断：

- 风险事件只是照护事件的一种；
- 如果系统只记录风险，不记录陪伴、协同、修复、接力等事件，就无法评估关系与长期价值。

因此推荐：

- 用 `CareEvent` 替代 `RiskEvent`；
- `RiskEvent` 退为 `CareEvent.event_class=risk` 的特例。

## 5. `CareEvent` 与 `Task` 的边界

这是当前最容易再次犯错的地方，必须单独冻结。

### 5.1 基本边界

- `CareEvent`：**已发生 / 被判断已发生**
- `Task`：**待执行 / 正在执行**

`CareEvent` 可以触发 `Task`，但永远不是 `Task`。

### 5.2 当前约束

现有从 `RiskEvent` 继承来的 `recommended_action` 字段容易膨胀成“半个 `Task`”。

因此当前明确约束：

1. `recommended_action` 只允许写**动作类型枚举**，例如 `remind / approach / escalate_family / create_task`；
2. 不允许在该字段中写入执行细节；
3. 任何具体话术、导航目标、超时策略、重试逻辑和执行步骤，都必须落在 `Task` 中。

## 6. `V1` 最小激活方式

路线 B 并不意味着 `V1` 现在就要把所有新能力全部实现出来。

当前更合理的表达是：

**模型按 `7` 实体定义正确，`V1` 运行时只激活其中必要子集。**

推荐最小激活方式：

1. `CareRelationship` 在 `V1` 中只启用基础关系角色、授权边界、升级优先级与 `phase=initial`，不提前引入完整连续关系评分。
2. `CareEvent` 在 `V1` 中至少覆盖 `event_class=risk`、`event_class=service_handoff` 和 `event_class=medication`，`companion` 与更高阶关系事件先保留为可空位。
3. `Task` 继续承接当前陪伴、健康、递送、异常升级与人工接力执行链，但必须与 `CareEvent` 明确分层。
4. `Object` 中优先激活药物对象、穿戴对象、生命体征设备对象和关键危险物对象。
5. `Person.health_state` 只激活慢病、过敏、禁忌、紧急处理建议和设备绑定，不提前扩展成完整病历系统。

### 6.1 `V1` 最小激活子集表

| 实体 | `V1` 必须激活 | `V1` 可以延后 |
| --- | --- | --- |
| `Person` | 身份、位置、偏好、健康状态摘要 | 更细的长期行为画像 |
| `CareRelationship` | 关系角色、授权边界、升级优先级 | 连续关系质量评分 |
| `Household` | 成员、`care_network`、应急链路 | 更复杂的平台合同状态 |
| `Place` | 房间、门口、床边、卫生间、充电点、药物存放区 | 更细粒度热点语义分区 |
| `Object` | 药物、穿戴、测量设备、危险物 | 更丰富的普通家具属性 |
| `Task` | 提醒、找人、送药、上报、转人工、回充 | 更复杂的多阶段服务编排 |
| `CareEvent` | 风险、人工接力、用药事件 | 高阶陪伴关系事件 |

## 7. 当前未冻结的内容

即使路线 B 已提升为主方向，以下内容当前仍未冻结：

1. 七实体各自的完整字段表；
2. `CareRelationship` 是否需要连续型关系质量维度，还是仅保留分段等级；
3. `CareEvent` 的最终事件族全集；
4. 从当前 `9` 骨架到 `7` 目标模型的迁移步骤与 proto 发布策略。

## 8. 当前建议的推进顺序

1. 先冻结七实体目标模型的一级边界；
2. 再冻结 `CareEvent` 与 `Task` 的边界规则；
3. 再决定 `V1` 最小激活子集；
4. 最后再进入详细字段与迁移设计。

### 8.1 建议的迁移顺序

当前建议采用以下迁移顺序，而不是一次性把所有细节改完：

1. 先在架构文档层冻结 `7` 实体目标模型；
2. 再把 `RoleBinding -> CareRelationship`、`RiskEvent -> CareEvent` 的语义映射写清；
3. 再把 `HealthProfile` 与 `MedicationAsset` 分别下沉为 `Person.health_state` 与 `Object.subtype_payload`；
4. 最后再推进到 proto / 下游接口的兼容与迁移设计。

## 9. 本轮建议用户重点审阅

1. 当前这 `7` 类一级实体是否合理；
2. `Task` 保留、`RiskEvent` 升级为 `CareEvent` 的判断是否成立；
3. `recommended_action` 只保留动作类型枚举的约束是否成立；
4. 是否接受“路线 B 为目标模型，`V1` 只激活子集”的推进方式。
