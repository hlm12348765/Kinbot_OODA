## 1. 文档目的

本文档用于冻结当前 `V1` 主链中的伴生系统最小闭环。

当前只回答 5 个问题：

1. 在当前 `V1` 主链里，机器人本体之外必须同时交付什么；
2. 家属 App 与最小云分别承接哪些责任；
3. 远程确认、授权治理、记忆治理和审计如何闭环；
4. 哪些服务能力只保留接口位，不进入当前主链验收；
5. `App / 云` 异常时，机器人如何退回端侧最小闭环。

本文不展开完整互联网产品规划，也不把人工服务、在线问诊和第三方履约继续写成当前 `V1` 的默认主链。

## 2. 当前设计前提

本版本基于以下已确认条件：

- 目标系统是机器人，本体之外的手机 App、后台云服务属于伴生系统；
- 当前 `V1` 主链按“机器人本体 + 家属 App + 最小云”组织；
- 后台人工服务、在线问诊、第三方履约和智能家居继续保留为后续适配位，不进入当前主链验收；
- 家属仍是高风险异常的第一优先升级对象；
- 原始视觉、语音和生物特征数据默认端侧处理；云侧只承接最小必要的结构化结果；
- 当前开发承接只围绕离散决策与事件驱动两条主链组织；
- 任何远程确认、云侧配置或后续服务接力，一旦触及本体动作执行，仍必须回到统一审批接口；
- 长期记忆默认保存结构化结果，不保存完整原始对话；
- 老人本人和子女可以在 App 中查看、修改和删除长期记忆，修改时需要提示功能影响；
- 纪念日、提醒、主动交互频率和人设强度等配置只允许老人本人和子女调整，冲突时进入待确认；
- `120` 路线继续优先挂在手机 App / 手机能力侧；
- 后续如果重新激活人工服务或第三方服务，必须在受控适配位中承接，不能反向重写当前主链边界。

## 3. 为什么当前 `V1` 仍必须有伴生系统闭环

如果只做机器人本体，而不定义“家属 App + 最小云”的最小闭环，会直接留下 4 个问题：

1. 授权不可治理；
2. 远程确认不可闭环；
3. 异常事件不可在家庭责任网络中外化；
4. 结构化记忆不可纠错、不可审计。

因此，当前 `V1` 仍必须有伴生系统闭环，但这个闭环只优先解决：

1. 绑定与授权；
2. 远程确认；
3. 事件摘要与家属联动；
4. 记忆治理；
5. 最小审计与配置同步。

补充说明：

1. 当前 `V1` 不再把“人工坐席接住所有复杂场景”写成闭环成立前提。
2. 当前伴生系统闭环只承接分布式家庭中的远程在场与责任同步，不反向接管本体端动作执行。
3. 人工服务、在线问诊和第三方履约继续保留接口位，但只属于后续适配位。

## 4. 当前伴生系统的职责边界

### 4.1 家属 App

家属 App 的当前职责收敛为 6 类：

1. 设备绑定与家庭成员绑定；
2. 授权配置与角色治理；
3. 异常事件查看与远程确认；
4. 提醒、计划和纪念日配置；
5. 长期记忆查看、纠错和删除；
6. 当前状态总览与结果回看。

不负责：

- 绕过端侧安全链控制机器人危险动作；
- 默认保存原始音视频；
- 承担首线人工分诊职责；
- 代替后续适配位中的人工服务会话系统。

### 4.2 最小云

最小云的当前职责收敛为 6 类：

1. 设备账号与绑定关系；
2. 配置同步；
3. 结构化事件与审计留痕；
4. 远程确认结果回传；
5. 结构化记忆同步与治理结果同步；
6. 版本、策略和最小可用性治理。

不负责：

- 直接控制底盘运动或高风险执行器；
- 默认持有原始敏感数据；
- 把人工服务、在线问诊或第三方履约编排写成当前主链必交付物；
- 绕过端侧授权门控执行动作。

### 4.3 后续适配位

以下能力当前只保留接口位，不进入当前 `V1` 主链验收：

1. 首线人工服务接入；
2. 在线问诊与医疗专业主体转接；
3. 第三方履约平台网关；
4. 社区 / 物业 / 公共应急协同；
5. 智能家居和其他外部系统的高覆盖接入。

约束：

1. 后续若激活这些能力，必须显式表达 `service_availability`、授权状态和审计链。
2. 它们只能接住结构化结果，不能改写本体第一责任闭环。
3. 它们的角色边界由 [12_human_service_and_telemedicine_boundaries.md](12_human_service_and_telemedicine_boundaries.md) 承接。

## 5. 当前 `V1` 最小闭环能力包

建议把当前伴生系统最小闭环收敛为 5 个能力包：

### 5.1 绑定与授权闭环

必须支持：

- 设备与老人身份绑定；
- 子女身份绑定；
- 保姆模式授权；
- 访客默认低权限策略；
- 异常上报授权；
- 主动交互授权；
- 老人 / 子女配置冲突待确认。

### 5.2 提醒与计划闭环

必须支持：

- 用药计划配置；
- 纪念日与日程配置；
- 提醒渠道选择；
- 机器人端执行结果回传；
- 关键配置只允许老人本人和子女修改。

### 5.3 异常升级与远程确认闭环

必须支持：

- 机器人上报结构化异常事件；
- App 端查看事件摘要；
- 家属远程确认或拒绝；
- 超时无人响应时保留安全降级与重试策略；
- 远程确认结果写回机器人与审计链。

补充说明：

- 人工服务转接当前只保留为后续适配位，不作为当前闭环成立前提。

### 5.4 记忆治理闭环

必须支持：

- 查看长期记忆条目；
- 修改或删除结构化记忆；
- 标记需人工纠错的内容；
- 控制可见角色范围；
- 将治理结果同步回机器人。

### 5.5 最小同步与审计闭环

必须支持：

- 关键事件日志；
- 授权变更日志；
- 远程确认日志；
- 配置同步日志；
- 记忆治理日志。

## 6. 关键接口边界

建议先固定 6 类逻辑接口：

### 6.1 `robot_event_uplink`

作用：

- 机器人把结构化事件上传给最小云。

典型事件：

- 异常候选；
- 异常升级；
- 用药执行结果；
- 记忆条目写入；
- 远程确认请求。

补充约束：

- 事件应保留 `origin_paradigm`、`risk_level` 和 `audit_session_id`。

### 6.2 `config_sync`

作用：

- App 与最小云把授权、计划、纪念日、人设强度等配置同步给机器人。

### 6.3 `remote_confirmation`

作用：

- 家属 App 对高风险动作、异常升级和记忆治理请求做远程确认。

补充约束：

- `remote_confirmation` 产出的是远程确认结果，不是对本体执行器的直接控制指令。
- 任何需要本体执行的动作，仍必须进入 [07_safety_compliance_authorization_api.md](07_safety_compliance_authorization_api.md) 定义的统一审批门。

### 6.4 `memory_governance_api`

作用：

- 对长期记忆条目进行查看、编辑、删除、可见性变更和纠错。

### 6.5 `audit_export`

作用：

- 为售后、法务、质量与运营提供结构化审计视图。

### 6.6 `future_service_adapter_reserved`

作用：

- 为后续人工服务、在线问诊和第三方履约保留受控适配入口。

补充约束：

- 当前 `V1` 只保留接口位，不承诺默认可用性和时效；
- 任何适配结果都不能绕过端侧安全链控制本体动作。

## 7. 当前 `V1` 高价值闭环场景

建议当前 `V1` 至少打通以下 5 个端到端场景：

### 7.1 新用户初始化

流程：

1. App 绑定设备与家庭成员；
2. 配置授权和提醒；
3. 机器人同步配置；
4. 完成初始家庭运行闭环。

### 7.2 到点提醒与未响应联动

流程：

1. App / 最小云配置提醒任务；
2. 机器人执行提醒；
3. 未响应时生成结构化事件；
4. 家属 App 收到通知；
5. 家属远程确认或等待本体安全降级。

### 7.3 健康异常与家属联动

流程：

1. 机器人识别异常候选；
2. 本地确认并生成结构化事件；
3. 家属优先收到通知；
4. App 返回远程确认结果；
5. 结果写回机器人与审计链。

### 7.4 长期记忆治理

流程：

1. 机器人生成结构化记忆条目；
2. 最小云保存摘要与权限；
3. 家属或老人通过 App 查看；
4. 对条目进行修改、删除或纠错；
5. 同步回机器人。

### 7.5 断网或云异常下的最小闭环

流程：

1. 机器人继续端侧提醒、确认和事件记录；
2. App / 云同步暂缓；
3. 高风险动作继续走端侧审批；
4. 网络恢复后回补结构化日志与配置差异。

## 8. 失败回退策略

建议当前 `V1` 先固定 5 类失败回退：

1. `App 不在线`
说明：机器人继续本地执行，最小云保留事件待送达队列；必要时转为短信 / 电话等手机侧能力。

2. `最小云不可达`
说明：机器人保持端侧最小闭环，暂停远程配置同步与后续适配位调用。

3. `远程确认超时`
说明：继续按家属优先策略重试，并退回本地安全降级，不把超时直接升级为人工服务必达。

4. `记忆治理冲突`
说明：以老人本人优先，冲突条目进入待确认状态。

5. `后续适配位不可用`
说明：不影响当前 `V1` 主链；只写审计并保持 `App + 最小云` 的最小闭环。

## 9. 当前 `V1` 明确不做的事

当前 `V1` 明确不做以下 5 件事：

1. 把运营坐席写成默认首线闭环；
2. 把在线问诊、买药、配送写成当前主链的必经步骤；
3. 默认长期保存原始音视频；
4. 云侧直接遥控机器人危险动作；
5. 做成大而全的家庭管理平台或复杂社交社区。

## 10. 当前结论与后续问题

当前收口结论如下：

1. 当前 `V1` 伴生系统主链正式收缩为“家属 App + 最小云”；
2. 当前闭环只承接绑定授权、远程确认、事件摘要、记忆治理和最小审计；
3. 人工服务、在线问诊和第三方履约继续保留为后续适配位，不进入当前主链验收；
4. 任何后续外部服务扩展都不得反向改写机器人本体的第一责任闭环。

当前仍需继续澄清的重点有 4 个：

1. App 允许下发的远程控制指令清单和禁止项；
2. 最小云的一代最低可用性与容灾要求；
3. 后续适配位何时从“接口位”升级为正式交付物；
4. 当前伴生系统最小闭环如何映射到量产试点指标。
