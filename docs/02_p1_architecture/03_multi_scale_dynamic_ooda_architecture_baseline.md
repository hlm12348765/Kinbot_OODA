## 1. 文档目的

本文档用于回答 Kinbot 当前运行时层的核心问题：

**在“家庭共居智能体”总架构下，系统应如何组织多种执行范式，而不是继续把 `OODA` 当作唯一总方法论？**

当前结论是：

1. `OODA` 仍然保留；
2. 但 `OODA` 已退到运行时层，主要承担离散决策范式；
3. Kinbot 当前运行时应按多执行范式来表达，而不是只按单一 `OODA` 主环表达。

本文档当前负责：

1. 定义 `4` 类执行范式；
2. 说明离散决策范式中的 `R1-R4` 如何保留；
3. 明确执行范式协调机制、输入和切换规则；
4. 说明哪些趋势已允许进入主线表达，哪些仍受系统级硬边界约束。

## 2. 为什么经典 `OODA` 不够

经典 `OODA` 对机器人仍然有价值，但若直接按“单轮、串行、固定节拍”理解，会出现 `4` 个问题：

1. **时间尺度过于单一**  
   避障、到人提醒、家庭习惯学习和长期服务规划显然不在同一时间尺度上。

2. **空间尺度过于单一**  
   底盘周边、一个房间、整个家庭、跨房间长期空间记忆，不是同一种空间问题。

3. **串行假设过强**  
   机器人不会 Observe 完再 Orient、Orient 完再 Decide、Decide 完再 Act；真实系统里它们持续并发。

4. **对具身智能趋势容纳不足**  
   局部端到端、`VLA`、跨任务迁移、长期记忆、事件驱动和人机接力，都要求运行时从“单回合反应”升级为“多范式协同”。

## 3. 当前正式基线

在当前分支中，Kinbot 的运行时正式采用**多尺度执行范式基线**，核心基线有 `6` 条：

1. 家庭共居智能体是总架构，`OODA` 不再承担总架构角色。
2. 运行时同时容纳离散决策、连续流式、事件驱动和长周期演化 `4` 类执行范式。
3. 原 `R1-R4` 继续保留，但被重新定位为离散决策范式内部的 `4` 类执行环。
4. 所有执行范式都必须受安全、合规、授权、恢复和审计边界约束。
5. `Orient + Decide` 融合与局部端到端化是允许的前瞻方向。
6. 当前仍保留调度输入与切换规则，但它们应被理解为执行范式协调机制的一部分，而不是单一 `OODA` 总环的属性。

## 4. 四类执行范式

### 4.1 离散决策范式

离散决策范式继承旧 `OODA` 主线，用于明确感知、认知、决策、动作和监督之间的离散阶段关系。

它适用于：

1. 需要显式审批和显式状态转移的动作；
2. 需要清晰责任边界和审计链的流程；
3. 需要把 `R1-R4` 分层执行环显式建模的场景。

### 4.2 连续流式范式

连续流式范式用于承接更前瞻的具身智能路线，例如：

1. 局部感知到动作的连续策略；
2. `VLA / 局部端到端`；
3. `Orient + Decide` 的融合表达。

它不要求显式分解为固定环节，但不能绕开系统级硬边界。

### 4.3 事件驱动范式

事件驱动范式用于承接跨尺度照护事件，包括：

1. 风险事件
2. 健康事件
3. 关系影响事件
4. 服务升级事件

其关键不是持续运行主环，而是：

`事件触发 -> 状态更新 -> 响应编排 -> 回写长期状态`

### 4.4 长周期演化范式

长周期演化范式用于承接长期关系形成、习惯学习、提醒策略优化和长期记忆治理。

它强调：

1. 长期可信
2. 长期克制
3. 长期可解释
4. 长期可恢复

## 5. 离散决策范式中的四类执行环

当前确认把离散决策范式内部的执行环继续收敛为 `4` 类：

| 执行环 | 时间尺度 | 空间尺度 | 主要目标 | 典型能力 |
| --- | --- | --- | --- | --- |
| `R1 反射环` | 毫秒到百毫秒 | 机体周边、局部接触空间 | 立即规避危险 | 急停、避障、限速、夹手保护 |
| `R2 执行环` | `0.5` 秒到数秒 | 人附近、房间内局部空间 | 稳定执行当前动作 | 局部导航、到人确认、仓门执行、动作监督 |
| `R3 任务环` | 数秒到数分钟 | 房间到全屋 | 推进任务闭环 | 找人、送药、提醒升级、异常确认、回充仲裁 |
| `R4 关系与服务环` | 小时到天 | 家庭长期运行空间 | 优化长期服务 | 习惯学习、提醒策略、长期记忆治理、服务编排 |

## 6. 单轮时间与空间尺度

在 Kinbot 中，“一轮”不再对应统一时钟，而与当前问题规模绑定：

1. 在 `R1` 中，一轮对应一次局部危险确认和即时动作修正。
2. 在 `R2` 中，一轮对应一次受监督动作执行与反馈闭环。
3. 在 `R3` 中，一轮对应一次任务阶段推进或异常升级判断。
4. 在 `R4` 中，一轮对应一次长期策略更新、习惯修正或关系评价回写。

## 7. 执行范式协调机制

当前仍保留一个显式的执行范式协调机制，用于决定当前以哪个执行范式、哪类执行环为主导。

### 7.1 协调输入

当前第一版继续收敛为 `6` 类输入：

1. `risk_level`
2. `novelty_level`
3. `confidence_level`
4. `task_deadline`
5. `resource_state`
6. `human_context`

### 7.2 当前切换规则

当前继续保留 `7` 条切换规则，但其语义已改写为“执行范式 / 执行环切换规则”：

1. 检测到碰撞、跌落、夹手等高风险信号时，任何高层范式都必须被 `R1` 抢占。
2. 到人确认、开仓、递送等已批准动作进入执行时，由 `R2` 主导。
3. 找人、送药、异常升级、回充仲裁等跨步骤任务，由 `R3` 主导。
4. 纪念日、习惯学习、提醒优化、长期服务策略调整，由 `R4` 或长周期演化范式主导。
5. 夜间、卫生间、身份不确定等高敏感上下文，会压低主动交互和大范围移动的尺度。
6. 网络断开时，服务升级与长周期策略能力要收缩，尽量回退到端侧可闭环范围。
7. 关键传感器失效且已无法识别障碍物时，所有涉及运动的执行环都必须中止。

## 8. 离散决策范式中的 `OODA` 四阶段

`OODA` 在新基线下仍然保留，但被明确为离散决策范式中的四阶段语法。

### 8.1 Observe

不是被动采集，而是按当前尺度主动选择采样密度、关注区域和补采方式。

### 8.2 Orient

不只是情境理解，还负责：

1. 对感知结果进行认知和评价
2. 风险评估
3. 尺度选择
4. 时域压缩
5. 当前主导执行环切换

### 8.3 Decide

不只是出动作，还负责：

1. 当前尺度下的策略选择
2. 是否需要切环
3. 是否需要升级人工或外部系统
4. 在允许范围内与 `Orient` 融合

### 8.4 Act

不只是执行动作，还负责：

1. 执行监督
2. 失败检测
3. 恢复与回退
4. 把执行结果反馈回上层尺度

## 9. 连续流式执行范式的当前边界

当前分支允许把以下内容写入主线表达：

1. 局部端到端化是允许的；
2. `Orient + Decide` 融合是允许的；
3. 部分高频局部执行可不再显式走完整离散阶段表达。

当前分支仍不允许：

1. 以端到端为名绕开安全、授权、恢复和审计；
2. 以模型黑箱替代系统级责任边界；
3. 把连续流式范式直接写成一代默认全局实现方式。

## 10. 事件驱动执行范式的当前边界

当前事件驱动范式主要承接：

1. 跨尺度照护事件
2. 风险与健康事件升级链
3. 服务接力与责任转移
4. 关系影响与长期记忆回写

它与离散决策范式的关系不是替代，而是：

1. 事件触发可唤起某一离散执行环；
2. 事件也可直接触发服务升级；
3. 事件处理结果应回写长期状态。

## 11. 长周期演化执行范式的当前边界

当前长周期演化范式主要承接：

1. 长期关系形成
2. 家庭节奏适应
3. 提醒策略优化
4. 长期记忆治理
5. 长周期服务策略修正

它的目标不是短时最优，而是长期可信。

## 12. 与现有 Kinbot 架构的关系

这套基线不是另起炉灶，而是对现有主线的重组：

1. `01_overall_architecture.md` 负责总图与双视角基线；
2. `14_family_co_living_agent_paradigm.md` 负责家庭共居智能体总图；
3. 本文负责运行时层的多执行范式表达；
4. `R1-R4`、调度输入和切换规则继续继承；
5. `World State 9 -> 7` 与关系 / 事件实体化仍是后续单独评审项。

## 13. 当前结论与后续对齐要求

当前结论是：

1. `OODA` 继续保留，但其角色已从总架构下降为离散决策范式；
2. Kinbot 运行时已进入多执行范式表达；
3. `Orient + Decide` 融合和局部端到端化已被正式允许进入主线表达；
4. 数据模型主表重组仍未冻结。

后续涉及运行时设计的文档，至少要对齐以下 `5` 条：

1. 不再把 `OODA` 写成唯一总方法论；
2. 必须说明自己主要服务哪类执行范式；
3. 必须说明哪些地方允许融合、哪些边界不能穿透；
4. 必须保留安全、授权、恢复和审计的系统级地位；
5. 不得把 `World State 9 -> 7` 等仍属 `provisional` 的事项提前写成既成事实。
