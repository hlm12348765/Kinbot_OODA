## 1. 输入来源

本次承接对象来自 [docs/03_p2_feasibility/04_hardware_software_selection_matrix.md](../03_p2_feasibility/04_hardware_software_selection_matrix.md) 中“`5.4 分发给 VLN 专家线程的专项任务`”条目，共 6 项：

1. 冻结 `P1 / P2 / P3` 三类任务模板的真实输入定义，包括相机路数、图像分辨率、语义地图切片、位姿字段和历史上下文长度。
2. 冻结 `VLN` 高层动作输出格式，优先给出结构化动作字段或离散 action schema，避免把长自然语言生成放进实时导航链路。
3. 对 `4B / 7B / 8B + FP8` 三档模型分别给出建议的 `N_out`、目标 `TTFT`、目标稳态 `TPS` 与可接受退化边界。
4. 评估 `KV cache`、视觉 token 压缩、图像采样率、多帧复用和多速率调度对 `TTFT / TPS` 的影响，并给出优先级排序。
5. 判断 `VLN` 是否需要拆成“高层语义决策 + 低层经典规划”双速率结构，以避免端侧大模型直接卡住运动链路。
6. 最终输出一张“模型规模 × 输入规模 × `TTFT / TPS` × 是否可上线”的判定表，作为 `地瓜 S100 Pro（128 TOPS）` 与 `RK` 备线是否继续投入的共同门槛。

## 2. 承接判断

这 6 项任务虽然都与 `TTFT / TPS` 有关，但不应被错误理解为“全部由板级性能线程闭门完成”。  
`VLN` 专家线程需要主责定义的是：

- 任务模板到底是什么
- 输入规模如何冻结
- 输出格式如何结构化
- 哪些时延指标是产品上可接受的
- 哪些优化手段优先级更高
- 是否必须采用双速率结构来保护运动链路

不由 `VLN` 专家线程单独拍板的部分是：

- 板级真实跑分
- 芯片 / 工具链适配
- 热设计与功耗收敛
- `FP8` 与运行时框架的最终工程可行性

因此，`KBT-22` 的职责应被重新定义为：

- 作为 `VLN` 专家线程在本轮 P2 可行性收敛中的父 issue
- 负责冻结任务定义、输出 schema、时序边界和架构判定口径
- 为平台 / 性能线程提供可执行、可验收、可回写的统一输入

与既有 `KBT-34` 的关系应理解为：

- `KBT-22`：`VLN` 专家线程的上层责任父 issue
- `KBT-34`：历史上已拉出的量化执行议题，可继续保留为相关 issue
- 自本次承接起，`5.4` 条目下新增的专家线程子任务统一挂回 `KBT-22`

## 3. 本次拆解原则

拆解按“一条原始任务对应一个可关闭子 issue”处理，避免多个结论耦合在同一个 issue 里难以验收。

验收口径：

- 每个子 issue 都必须输出结构化结论，而不是只停留在讨论
- 每个子 issue 都必须能回写到研究文档或父 issue 评论
- 需要跨线程确认的点，明确标记 `review required`

## 5. 推荐的执行顺序

按依赖关系建议采用下面顺序：

1. 先完成输入定义冻结。
2. 再完成动作输出 schema 冻结。
3. 并行推进性能目标表和优化手段排序。
4. 在输入与输出边界稳定后，完成双速率结构判定。
5. 最后汇总为可上线判定矩阵。

原因：

- 输入没冻结，`TTFT / TPS` 就没有统一口径。
- 输出没冻结，`N_out` 就没有真实意义。
- 双速率结构不明确，很多时延目标会落在错误的责任层上。

对应 issue 依赖建议：

- `KBT-37` 依赖 `KBT-35` 与 `KBT-36`
- `KBT-38` 依赖 `KBT-35`
- `KBT-39` 依赖 `KBT-35` 与 `KBT-36`
- `KBT-40` 依赖 `KBT-35`、`KBT-36`、`KBT-37`、`KBT-38`、`KBT-39`

## 6. 计划中的 review 节点

### Review A：输入 / 输出边界冻结

触发条件：

- 输入定义子 issue 完成
- 输出 schema 子 issue 完成

review 目标：

- 确认 `VLN` 没有越权接管系统级安全链
- 确认实时导航链路不依赖长自然语言输出

### Review B：时延目标与优化优先级

触发条件：

- 模型目标表子 issue 完成
- 优化手段排序子 issue 完成

review 目标：

- 确认目标 `TTFT / TPS` 与板级现实力相容
- 确认优化优先级不会与平台线程目标冲突

### Review C：架构判定与上线门槛

触发条件：

- 双速率结构子 issue 完成
- 可上线判定矩阵子 issue 完成

review 目标：

- 确认 `VLN` 与经典规划的分层边界
- 确认主线平台和备线平台是否继续投入

## 7. 预期回写产物

本轮承接后，`KBT-22` 应至少回写三类结果：

1. 一份 `VLN` 专家线程专项任务拆解文档  
   即本文档。
2. 一组可独立关闭的 `KBT-22` 子 issue  
   对应 `5.4` 的 6 项任务。
3. 一条父 issue 维护评论  
   说明承接边界、子 issue 列表和 review 节点。

## 8. 当前结论

本轮不再把 `KBT-22` 视为“已经完成的历史规划 issue”，而是把它恢复为：

- `VLN` 专家线程在 P2 可行性收敛阶段的父 issue
- 挂载 `5.4` 六项专项任务的统一入口
- 后续 review 与回写的维护节点
