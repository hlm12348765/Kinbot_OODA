## 1. 文档目的

本文档聚焦 Kinbot 后续 `VLN -> NFM` 的模型设计方案，重点回答以下问题：

1. 在当前阶段，模型主线到底应先做什么，后做什么。
2. 第一阶段的能力范围为什么应聚焦空间理解、地图理解、语义锚点、搜索与恢复，而不直接把动作输出作为主训练目标。
3. 为什么应先用 Qwen VL 27B 左右模型做 teacher，再蒸馏出 4B 左右的主力部署模型。
4. 多视角理解、深度几何、强化学习避障、基础空间长期记忆与个性化长期记忆应如何分阶段接入。
5. Teacher和Student的输入配置如何分别设计。
6. 这套方案的主要收益、代价和工程风险分别是什么。

**重要说明**：本文档区分两套配置：

- **Research Teacher配置**：27B级模型，用于能力上限探索，可使用更丰富传感器（多视角、深度真值等）
- **PDCP Student配置**：4B级端侧模型，必须对齐量产输入契约（1组双目+2个单目，共3-5相机），参考`docs/03_p2_feasibility/04_hardware_software_selection_matrix.md`

本文档不展开算力成本细节，但默认所有设计都必须服务于后续可部署的小模型路线；同时，`VLN` 在本文中被视为 `NFM` 的能力切片，而非最终问题定义。

## 2. 总体方案

### 2.1 核心路线

Kinbot 后续导航智能主线建议收敛为：

**核心架构：27B NFM Teacher 持续迭代 + 多轮蒸馏 4B Student**

整体采用双轨制：27B 模型作为持续迭代的研究主线，每当一项新能力在 27B 上稳定后，蒸馏出新版本的 4B 部署模型。

**第一轮：NFM 基础能力建设 + 首次蒸馏（P0）**
- 用 Qwen VL 27B 级别模型训练 teacher，聚焦空间理解、地图理解、语义全局锚点与搜索恢复。
- 不输出动作，只输出结构化认知判断（房间识别、区域排序、粗粒度定位、搜索与恢复建议）。
- 能力稳定后，首次蒸馏到 4B（4B v1），确保端侧部署可行性。

**第二轮：多视角、世界状态写回与几何增强 + 再次蒸馏（P1）**
- 在 27B teacher 上继续训练多视角理解（P1-high）、观测到语义地图匹配、世界状态写回、深度估计与几何约束（P1-medium）。
- 每项能力在 27B 上稳定后，蒸馏出新版 4B（4B v2、v3）。
- 用强化学习优化避障与搜索效率（P1-low），RL 可在 4B 上直接微调。

**第三轮：个性化长期记忆 + 最终蒸馏（P2）**
- 在 27B teacher 上最后接入个性化 / 行为长期记忆能力，难度最大，排在最后。
- 基础空间长期记忆不在这一轮才开始，而是在 P0/P1 已经作为 `NFM` 主能力建设。
- 以显式记忆库为主，不将长期先验直接压入主模型参数。
- 稳定后蒸馏出 4B v5，作为完整能力的部署版本。

### 2.2 方案定位

该方案的核心定位：

**最终目标**：4B 端侧模型量产落地

**当前主策略**：27B `NFM` research teacher + dataset + distillation 先行

具体而言：

- 前阶段建设高层语义导航与空间智能认知能力，不建设端到端底盘控制模型。
- **27B teacher 是能力探索主驱动**：承担 `NFM` 能力上限探索、训练数据生成与补标、蒸馏 teacher 的职责，可使用更丰富传感器和计算资源。
- **4B student 是产品验证门**：负责实际部署，严格对齐量产输入契约和端侧约束，通过蒸馏获得 teacher 能力子集。
- 优先在 27B teacher 上固化空间理解、地图理解、语义全局锚点、目标 belief 与搜索恢复这几个最高优先级能力，再逐步接入其他增强能力。
- 每项能力在 27B 上稳定后，评估是否可蒸馏到 4B，而不是先在 4B 上定义能力边界。
- 多视角、深度几何、强化学习、基础空间长期记忆和个性化长期记忆按难度与依赖关系分阶段接入，避免同时混训导致训练目标失焦。

## 3. 能力优先级与设计原则

### 3.1 能力优先级排序

基于部署可行性、训练难度和系统收益，后续能力按以下优先级推进：

**P0 级（最高优先级，必须优先完成）**
1. 空间理解：房间识别、家具识别、当前位置判断。
2. 地图理解：视角与地图对齐、区域拓扑关系理解。
3. 基础空间长期记忆：`room graph`、`room-furniture-object relation`、`target belief map`。

**P1-high 级（高优先级，27B teacher 能力稳定后接入）**
4. 搜索与恢复：目标区域排序、搜索顺序建议、失败恢复策略。
5. 多视角理解：多相机语义一致性、遮挡确认、视角切换。
6. 粗粒度全局定位：观测到语义地图匹配、语义锚点推断、区域级重定位。

**P1-medium 级（中优先级，多视角稳定后接入）**
7. 深度估计与几何约束：可通行空间判断、门口狭窄区域识别、遮挡关系。

**P1-low 级（低优先级，基础能力全部稳定后接入）**
8. 强化学习避障优化：搜索效率优化、主动观察策略、局部避障提示。

**P2 级（最低优先级，最后处理）**
9. 个性化 / 行为长期记忆：物品位置先验、人员活动习惯、历史搜索结果回写。

### 3.2 优先级设计依据

- P0 能力是所有后续能力的基础，必须最先稳定。
- 多视角理解排在 P1-high 是因为对找人找物任务收益明显。
- 深度估计排在多视角之后，因为其依赖空间理解能力且训练难度更高。
- 强化学习排在 P1-low 是因为需要稳定的 reward 信号，而这依赖前面所有能力。
- 个性化 / 行为长期记忆排最后，是因为最容易积累脏数据，且对前置能力可靠性要求最高；但基础空间长期记忆必须前置进入 P0/P1。

### 3.3 主线能力边界

当前方案将 `VLN` 作为 `NFM` 能力切片，其主线能力边界限定为以下范围：

- 指令到目标对象、目标区域和任务约束的理解。
- 当前视角中的空间语义理解。
- 当前观测与地图区域、地图拓扑的对应理解。
- 当前观测与语义锚点、粗粒度全局帧的对应理解。
- 目标候选区域排序。
- 搜索顺序建议。
- 搜索失败后的恢复建议。

当前阶段不将以下内容设为主训练目标：

- 直接输出底盘连续控制。
- 替代现有 SLAM + 局部规划 + 安全链。
- 将多视角、深度几何、个性化长期记忆、强化学习在同一阶段统一混训。
- 在 4B 主线能力尚未稳定前推进 2B 极限压缩。

### 3.4 第一阶段能力范围的设计依据

第一阶段聚焦空间理解、地图理解，主要基于以下设计依据：

- 直接决定语义导航策略层是否能够形成稳定判断。
- 可以以区域判断、区域排序、地图对齐、恢复建议等形式沉淀为结构化监督。
- 可以直接被现有导航栈使用，从而尽快形成系统级收益。
- 相比之下，直接动作输出高度耦合底盘形态、局部规划时序和安全约束，不适合作为当前阶段的主线训练对象。

### 3.5 teacher 与 student 的职责边界

27B 左右 teacher 的职责包括：

- 作为持续迭代的研究主线，逐步学习空间语义、地图语义、多视角、深度几何等新能力。
- 形成高质量结构化判断结果。
- 承担训练数据生成、补标和 student 蒸馏教师职责。
- 每项新能力稳定后，作为该轮蒸馏的 teacher。

4B 左右 student 的职责包括：

- 通过多轮蒸馏，逐版本获得 27B teacher 的最新能力（v1 基础能力 → v2 +多视角 → v3 +深度 → ...）。
- 满足后续系统部署的时延和输入预算约束。
- 每个版本在部署环境中验证后，作为当前阶段的线上模型。

### 3.6 接入顺序原则

后续增强能力的接入顺序应遵循以下原则：

1. 在 27B teacher 上稳定基础认知层（空间理解 + 地图理解）。
2. 首次蒸馏到 4B（v1），验证部署可行性。
3. 在 27B 上继续训练多视角理解（可用更丰富相机配置），稳定后蒸馏 4B v2。
4. 在 27B 上继续引入深度与几何中间约束，稳定后蒸馏 4B v3。
5. 用强化学习优化策略效率与局部避障提示（可在 27B 或 4B 上进行，视 reward 信号而定），蒸馏 4B v4。
6. 在 27B 上最后建设长期记忆，稳定后蒸馏 4B v5。

每一轮的核心流程为：**27B 训练新能力 → 评测稳定性 → 蒸馏新版 4B → 部署验证**。

该顺序的目的，是控制训练目标复杂度、评测复杂度和部署复杂度，避免不同层级能力在同一阶段相互干扰。

### 3.7 输入配置分层原则

本方案将 Teacher 和 Student 的输入配置分别设计：

**Research Teacher 输入契约**：
- 可使用更丰富传感器配置（多视角、深度真值、更高分辨率等）
- 目标：探索能力上限，不受端侧约束限制
- 用于训练数据生成、补标和蒸馏教师

**PDCP Student 输入契约**：
- 必须对齐量产基线：1组双目+2个单目（共3-5相机）
- 参考：`docs/03_p2_feasibility/04_hardware_software_selection_matrix.md`
- 目标：满足部署时延和功耗约束

**蒸馏对齐策略**：
- Teacher 使用丰富输入训练，输出高质量判断结果
- Student 使用量产输入子集，通过蒸馏学习 Teacher 的判断能力
- 输入差异通过特征对齐和输出对齐蒸馏来弥合

### 3.8 运行频率与 token 预算前置原则

本方案将输入 token 和运行频率视为模型设计约束，但 Teacher 和 Student 的约束不同：

- **Research Teacher contract**：可使用更丰富输入（多视角、深度真值、更高分辨率）探索能力上限，token 预算相对宽松
- **PDCP Student contract**：必须严格控制 token 预算以满足端侧部署约束，输入必须对齐量产基线（1组双目+2个单目）
- 多视角输入会显著影响单次推理 token 和每秒 token 吞吐
- 不同任务级别的频率要求差异很大，必须前置纳入结构设计

因此，模型方案、输入组织方式、频率分层和 token 预算需要在同一份设计文档中一并定义。

### 3.9 Student 输入 token 降低策略

为确保 4B Student 在量产相机配置下可部署，必须采用以下 token 控制策略：

**图像输入优化**
- 每次推理只取主+辅视角各 1 张关键帧（从量产配置中选择），不取连续视频流。
- 图像分辨率控制在 336×336 或 448×448，避免过高分辨率。
- 只在需要精细判断时额外增加 1 个局部 ROI。
- 避免同时输入多张历史帧，改用结构化摘要替代。

**地图输入优化**
- 不输入完整地图图像，改用局部地图裁剪（以当前位置为中心的 5×5m 或 7×7m 区域）。
- 地图拓扑关系用文本摘要表达（如 `当前在客厅，左侧是卧室，前方是厨房`）。
- 候选区域用结构化列表表达，不重复渲染地图。

**历史信息优化**
- 不回灌完整轨迹图像，改用近 3-5 步的文本摘要。
- 摘要内容包括：已搜索区域、已排除区域、当前搜索策略。
- 跨视角一致性结果转成结构化摘要，不反复灌入原图。

**任务信息优化**
- 指令用简洁文本表达，避免冗长描述。
- 任务约束用结构化字段表达，如 `{"target": "person", "constraint": "close_approach"}`。
- 语义锚点和局部执行上下文只保留关键字段（当前房间 / 区域、朝向、相对目标距离、局部风险摘要）。

**token 预算目标（输入 + 输出）**
- 房间级导航：450-650 token/次（输入 430-630，输出 20-30）。
- 家具级找物：600-850 token/次（输入 570-820，输出 30-40）。
- 人物级找人：650-900 token/次（输入 620-870，输出 30-50）。
- 高频轻量层：120-260 token/次（输入 100-240，输出 20-30）。

**输出 token 优化原则**
- 原设计输出格式（100-180 tokens）过于冗余，包含大量解释性文本。
- 实际只需保留核心字段：当前位置、下一步目标、备选目标。
- 优化后输出：20-50 tokens（减少 60-80%）。
- 输出占总 token 的 3-7%，输入占 93-97%，因此输入优化优先级更高。

## 4. 总体能力分层

### 4.1 第一层：空间理解、地图理解与粗粒度定位

这一层解决"任务到底是什么"以及"当前环境和地图怎么看懂"。

模型需要判断：

- 用户到底要去哪里、找什么、找谁。
- 任务成功条件是什么。
- 当前约束是什么，例如是否要求靠近、是否要求持续跟随、是否需要避开某些区域。

模型需要稳定输出：

- 当前所处房间或区域判断。
- 当前视角中的家具、支撑面、门口、通道等语义判断。
- 当前观测与地图区域的对应关系。
- 目标相关房间和家具区域的优先级。

### 4.2 第二层：基础空间长期记忆、搜索与恢复

这一层解决"先搜哪里，找不到怎么办，以及当前观测该怎样落到家庭世界状态里"。RL 策略优化作为本层的增强手段，用于提升搜索效率和局部避障。

模型需要稳定输出：

- 当前观测与 `semantic_global_frame` 的匹配关系。
- `target_belief` 的更新。
- 是否需要发起粗粒度全局重定位。
- 哪些区域值得优先搜索。
- 是继续前进、先观察、还是切换区域。
- 当前搜索失败后，下一步应转向哪里。
- 当前判断的不确定性有多大。

可参考定义：

- "目标候选区域排序"：指模型认为目标更可能出现在哪些房间、家具或支撑面附近，以及这些区域的优先级。
- "下一步语义子目标"：指下一步应先去哪个房间、哪个家具区域，或者应先从哪个视角继续观察。
- "主动观察建议"：指当前是否应该转头、切换相机、靠近目标区域，或绕过遮挡后再看。
- "恢复建议"：指当前搜索失败后，下一步应切换到哪个房间、哪个家具群或哪类地图区域重新开始搜索。
- "目标存在可能性判断"：指模型对目标是否位于当前区域、相邻区域或候选区域中的相对把握程度。
- "区域排序分值"：指模型对每个候选区域给出的优先级评分，用于后续搜索顺序生成。

### 4.3 第三层：多视角理解

这一层解决"不同相机看到的信息如何统一"。

模型需要具备：

- 多视角之间的语义一致性判断。
- 目标在不同视角中的持续识别。
- 遮挡和短时丢失后的重新确认。
- 何时应该切换主要观察视角。

### 4.4 第四层：深度与几何增强

这一层解决"语义判断怎样获得几何约束"。

模型需要增强：

- 可通行空间判断。
- 门口、走廊、狭窄区域的空间关系判断。
- 目标相对位置与遮挡关系判断。
- 靠近目标时的局部安全性和可达性判断。

### 4.5 第五层：基础空间长期记忆

这一层解决"系统是否能把过去的空间观察持续写成家庭世界状态"。

模型或系统需要逐步学会：

- `room graph`
- `room-furniture-object relation`
- `target belief map`
- `person last-seen / likely-to-be`
- `shared-space hot zones`
- `map drift / map change / relocalization hints`

### 4.6 第六层：个性化 / 行为长期记忆

这一层解决"系统是否能利用家庭中的个性化习惯和历史偏好"。

模型或系统需要逐步学会：

- 家庭中物品位置偏好。
- 家庭成员活动习惯。
- 搜索失败历史和偏好抑制。
- 记忆的新鲜度和可靠性。

## 5. 分阶段详细设计

### 5.1 T0：训练对象与中间语义接口定义

T0 的目标是完成训练对象、任务分层和中间语义接口定义。

该阶段需要收敛的内容包括：

- 哪些任务属于房间级导航。
- 哪些任务属于家具级找物。
- 哪些任务属于人物级找人。
- 模型当前阶段允许输出哪些上层语义结果。
- 现有导航栈如何消费这些结果。

可参考定义：

- 当前所在区域判断：模型认为机器人当前位于哪个房间或区域，以及对应置信度。
- 目标候选区域排序：模型认为目标更可能出现在什么房间、什么家具区域或什么支撑面附近。
- 下一步语义子目标：模型认为下一步应先去哪里，或先看哪里。
- 主动观察建议：模型认为是否应切换视角、靠近、转向或重新观察。
- 搜索失败恢复建议：模型认为当前找不到目标时，下一步应切换到哪个搜索策略。
- 风险约束提示：模型对狭窄区域、遮挡区域、人员附近等场景给出的谨慎提示。

T0 的主要收益是为后续训练、蒸馏、评测和系统集成建立稳定目标；主要代价是前期需要投入额外时间完成接口定义。

### 5.2 T1：27B teacher 的基础能力形成（P0 阶段）

T1 的目标是使 Qwen VL 27B 级别 teacher 形成稳定的空间理解、地图理解、语义锚点和基础空间长期记忆能力。

#### 训练目标（按优先级排序）

**P0-high（最高优先级）**
- 房间识别与房间属性理解。
- 当前视角与地图区域对齐。
- 当前所处位置判断。
- 观测到语义地图的匹配假设生成。

**P0-medium（高优先级）**
- 家具和支撑面识别。
- 地图中的语义位置关系理解。
- 房间级目标区域排序。
- `target belief` 更新。
- `room graph / furniture relation` 写回。

**P0-low（基础优先级）**
- 家具级目标区域排序。
- 搜索顺序建议。
- 搜索失败恢复建议。
- 地图漂移提示与环境变化提示。

#### 阶段定位与输出形式

T1 以稳定认知判断为交付目标，不输出动作。模型输出采用结构化格式。

**核心输出字段：**

| 输出项 | 内容 | 是否必需 |
| --- | --- | --- |
| 当前区域判断 | 当前位于哪个房间或区域 | 必需 |
| 候选区域排序 | 目标最可能出现的区域列表及优先级 | 必需 |
| 语义锚点 / 匹配假设 | 当前观测与语义地图的对应关系 | 必需 |
| 下一步语义子目标 | 下一步应前往或观察的目标区域 | 必需 |
| 恢复建议 | 当前策略失败时的替代搜索区域 | 可选 |

**标准输出格式（用于 teacher 训练和验证，~100-150 tokens）：**

```json
{
  "current_location": {
    "room": "客厅",
    "confidence": 0.92
  },
  "target_candidates": [
    {"region": "卧室-床头柜", "priority": 0.85},
    {"region": "书房-书桌", "priority": 0.72},
    {"region": "客厅-茶几", "priority": 0.43}
  ],
  "next_action": {
    "type": "navigate_to_region",
    "target": "卧室-床头柜",
    "reason": "目标物品通常放置在床头柜上"
  },
  "recovery_suggestion": {
    "if_not_found": "切换到书房-书桌区域",
    "alternative_regions": ["书房-书桌", "客厅-茶几"]
  }
}
```

**压缩输出格式（用于 student 部署，~20-40 tokens）：**

```json
{
  "loc": "客厅",
  "conf": 0.92,
  "targets": [
    {"r": "卧室-床头柜", "p": 0.85},
    {"r": "书房-书桌", "p": 0.72}
  ],
  "next": "卧室-床头柜",
  "alt": "书房-书桌"
}
```

**极简输出格式（用于极低 TPS 场景，~15-20 tokens）：**

```json
{
  "loc": "客厅",
  "next": "卧室-床头柜",
  "alt": "书房-书桌"
}
```

**输出格式选择原则：**
- Teacher 训练阶段：使用标准格式，保留推理过程和解释文本。
- Student 部署阶段：使用压缩格式，去除冗余字段，缩短字段名。
- 极低 TPS 硬件（< 150 tokens/s）：使用极简格式，只保留核心决策信息。
- 输出优化可将 token 数量减少 60-85%，但由于输出只占总 token 的 3-7%，对总体 TPS 需求影响有限。

#### 设计收益与代价

主要收益：
- 稳定 P0 级基础能力层，为后续所有能力打好地基。
- 为 4B student 提供高质量监督信号。
- 输出可解释，便于系统集成和问题定位。

主要代价：
- 早期产出以认知判断为主，不体现端到端动作效果。
- 对训练数据质量和地图表达一致性要求高。
- teacher 训练周期较长（预计 2-3 个月）。

### 5.3 T2：首次蒸馏到 4B 主力模型（P0 阶段）

T2 的目标是完成首次蒸馏，让 4B v1 成为 P0 阶段的可部署版本。这是从研究模型到产品模型的第一次转换，后续每当 27B teacher 新增并稳定一项能力后，都会重复蒸馏流程，产出新版本 4B（v2、v3、...）。

#### student 的最低能力要求

4B student 必须保住以下核心能力（按重要性排序）：

**必须保留（能力保留率 > 90%）**
- 房间级目标定位。
- 当前位置判断。
- 视角与地图对齐。

**应当保留（能力保留率 > 80%）**
- 家具级目标区域排序。
- 搜索顺序建议。

**尽量保留（能力保留率 > 70%）**
- 搜索失败后的恢复建议。
- 关键场景下的谨慎提示。

#### 蒸馏策略设计

**1. 输出对齐蒸馏（主要策略）**

不以 teacher 的语言表述为主，而以 teacher 的判断结果为主，重点包括：
- 区域排序结果。
- 区域优先级分值。
- 下一步语义子目标。
- 主动观察建议。
- 搜索失败后的恢复建议。
- 风险提示。

```python
# 蒸馏损失设计示例
loss = (
    0.4 * region_ranking_loss +
    0.3 * location_classification_loss +
    0.2 * next_action_loss +
    0.1 * confidence_calibration_loss
)
```

**2. 中间表示蒸馏（辅助策略）**

让 student 学习 teacher 的中间表示：
- 空间特征图对齐。
- 地图-视觉注意力对齐。
- 候选区域特征对齐。

**3. 输入配置分层（关键约束）**

**Research Teacher配置**：
- 可使用更丰富传感器：多视角相机、深度真值、几何约束
- 输入灵活性：支持不同分辨率和视角组合
- 目标：探索能力上限，生成高质量蒸馏数据

**PDCP Student配置**：
- 必须对齐量产输入契约：1组双目+2个单目（共3-5相机）
- 固定输入格式：336×336或448×448分辨率
- 固定地图裁剪：5×5m局部地图
- 固定历史摘要格式

**蒸馏对齐策略**：
- Teacher在蒸馏阶段使用与Student相同的输入子集
- Teacher的额外传感器仅用于训练和能力探索
- 确保Student能复现Teacher在标准输入下的能力

**4. 难例重点蒸馏**

对以下难例场景加大蒸馏权重：
- 相似房间的区分（如两个卧室）。
- 遮挡场景下的目标判断。
- 搜索失败后的恢复决策。
- 地图对齐的边界情况。

#### 蒸馏数据组成

- 60% teacher 生成的结构化判断结果。
- 25% teacher 对难例的区域排序和恢复选择。
- 10% 真实部署闭环样本（用于校准）。
- 5% 对抗样本（用于鲁棒性）。

#### 评估标准

**能力保留率评估**
- 房间识别准确率：teacher 92% → student ≥ 85%。
- 目标区域 Top-3 召回：teacher 88% → student ≥ 75%。
- 地图对齐准确率：teacher 90% → student ≥ 80%。

**部署指标评估**
- 单次推理时延：< 400ms（在目标硬件上）。
- 内存占用：< 5GB（INT8 量化后）。
- 功耗：< 8W（持续运行）。

#### 设计收益与代价

主要收益：
- Teacher 能力探索不受端侧约束限制，可充分发挥大模型潜力。
- 为后续蒸馏提供高质量监督信号。
- 更容易暴露真实的时延和稳定性瓶颈。
- 为后续多轮蒸馏（多视角、深度增强等）验证蒸馏流程和损失设计的可复用性。

主要代价：
- 蒸馏设计本身具有额外复杂度，需要 1-2 个月调优。
- 小模型在复杂推理样本上存在能力塌缩风险（如多步推理、复杂空间关系）。
- 若 teacher 与 student 的输入组织差异过大，会导致蒸馏失败。
- 需要在能力保留和推理效率之间反复权衡。

### 5.4 T3：多视角理解接入（P1-high 阶段）

**重要说明**：本节描述Research Teacher的多视角能力探索，Student部署配置见下文。

#### Teacher多视角配置（能力上限探索）

T3在27B teacher上训练多视角理解能力，可使用更丰富的相机配置探索能力上限。

**Teacher可用输入**：
- 多视角相机：前置、侧视、后视等
- 深度真值：用于几何约束训练
- 更高分辨率：探索细节识别能力

#### Student部署配置（PDCP基线）

**PDCP量产输入契约**：
- 1组双目相机 + 2个单目相机（共3-5相机）
- 参考：`docs/03_p2_feasibility/04_hardware_software_selection_matrix.md`

**Student输入组织**：

**主视角相机（前置相机）**
- 承担当前前方场景理解。
- 主要搜索区域观察。
- 房间识别与地图对齐。
- 运行频率：与任务级别同步（0.5-3 Hz）。

**辅助视角相机（侧置或后置相机）**
- 承担补盲与遮挡后确认。
- 人物重识别（找人任务）。
- 侧后方安全确认。
- 运行频率：按需触发或降频运行（0.5-1 Hz）。

#### 输入组织策略（关键设计）

**基础输入模式（房间级、家具级任务）**
```text
主视角图像（336×336）: ~180 token
辅助视角图像（336×336）: ~180 token
地图摘要: ~120 token
任务状态: ~100 token
---
总计: ~580 token
```

**精细输入模式（找人、小尺度物品任务）**
```text
主视角图像（336×336）: ~180 token
辅助视角图像（336×336）: ~180 token
人物 ROI（224×224）: ~100 token
地图摘要: ~120 token
人物状态摘要: ~120 token
---
总计: ~700 token
```

**降频模式（连续动态任务）**
- 主策略层（1-2 Hz）：使用基础输入模式。
- 高频轻量层（4-5 Hz）：只用跟踪结果 + 局部风险摘要（~150 token）。

#### 跨视角一致性处理

**方案 1：结构化摘要（推荐）**
- 模型输出多视角的语义一致性判断。
- 转成结构化摘要：`{"main_view": "客厅-沙发区", "aux_view": "客厅-门口", "consistency": "high"}`。
- 下一帧只输入摘要，不重复输入原图。
- token 节省：从多图输入降到 ~50 token。

**方案 2：注意力融合（备选）**
- 在模型内部做跨视角特征融合。
- 优点：端到端学习。
- 缺点：训练难度更高，可解释性差。

#### 训练策略

**阶段 1：27B Teacher 单视角迁移**
- 先让 27B teacher 适应多图输入格式。
- 暂时屏蔽辅助视角，只用主视角。
- 确保已有能力不退化。

**阶段 2：27B Teacher 多视角联合训练**
- 逐步引入辅助视角（可使用更丰富相机配置）。
- 重点训练遮挡场景、找人场景。
- 监督信号：跨视角一致性、目标重识别。

**阶段 3：27B Teacher 难例强化**
- 针对主视角失败但辅助视角成功的案例。
- 针对多视角冲突的案例。
- 校准置信度。

**阶段 4：蒸馏到 4B v2**
- 27B 多视角能力稳定后，蒸馏出包含多视角能力的 4B v2。
- 蒸馏损失在 T2 基础上增加跨视角一致性对齐项。
- 验证 4B v2 在部署环境中的时延和稳定性。

#### 设计收益与代价

主要收益：
- 找人任务成功率提升 15-25%（预估）。
- 遮挡场景鲁棒性提升 20-30%（预估）。
- 降低单前视角盲区带来的误判。
- 为后续动态任务提供感知基础。

主要代价：
- 单次推理 token 会随输入视角增加而增加。
- 相机时间同步要求高（误差需 < 50ms）。
- 相机相对位置标定必须准确。
- 多视角职责不清时，增益会被噪声抵消。
- 训练数据需要同步采集多视角相机。

### 5.5 T4：深度与几何增强接入（P1-medium 阶段）

T4 的目标是在 27B teacher 上训练深度与几何约束能力，为语义判断提供几何增强，稳定后蒸馏出 4B v3。

#### 优先增强的能力

**P1-medium-high（优先接入）**
- 门口和狭窄通道判断（影响通行性决策）。
- 家具边缘和遮挡关系判断（影响目标可见性）。

**P1-medium-low（次优先接入）**
- 靠近目标时的局部可达性判断。
- 近障区域中的谨慎提示。

#### 技术方案选择

**方案 1：深度图作为额外输入（推荐）**
- 用单目或双目深度估计模型生成深度图。
- 深度图降采样后作为额外通道输入（增加 ~80-120 token）。
- 优点：模块解耦，深度模型可独立优化。
- 缺点：增加推理链路。

**方案 2：端到端深度预测（备选）**
- VLM 直接预测关键区域的深度。
- 用深度作为辅助监督信号。
- 优点：端到端，不增加推理链路。
- 缺点：训练难度大，需要大量深度标注。

#### 训练数据需求

- 真实场景深度标注（RGBD 相机采集）：~5000 场景。
- 仿真场景深度数据（完美深度）：~20000 场景。
- 关键场景强化：门口、狭窄通道、家具边缘。

#### 设计收益与代价

主要收益：
- 狭窄通道通行性判断准确率提升 20-30%。
- 近障场景误判率降低 15-20%。
- 目标遮挡关系判断更准确。

主要代价：
- 引入新的中间表示（深度图）。
- 需要深度标注或自监督训练。
- 深度估计误差会传导到下游决策。
- 单次推理 token 增加 80-120（如果用方案 1）。

### 5.6 T5：强化学习优化避障和策略效率（P1-low 阶段）

T5 将强化学习用于策略层优化，而不是直接用于底盘连续控制层。

**训练模型选择（与前几轮不同）：**

RL 是所有阶段中唯一可能需要在 4B 上直接微调的环节，原因如下：
- RL 的 reward 信号与部署时延、实际推理行为强相关，27B 上的最优策略不一定适用于 4B。
- 搜索效率、避障时机等策略决策受模型推理速度影响，4B 和 27B 的行为模式不同。
- 建议方案：先在 27B 上用 RL 探索策略上限，再在 4B 上用小学习率 RL 微调，最终产出 4B v4。

#### 适合被优化的对象

**搜索策略优化**
- 搜索顺序是否更高效（减少无效搜索）。
- 何时应该切换搜索区域。
- 何时应该放弃当前策略。

**主动观察优化**
- 何时应该主动转头观察。
- 何时应该靠近目标区域。
- 何时应该绕过遮挡后再看。

**局部避障优化**
- 何时应该绕开拥挤区域。
- 何时应该放慢速度。
- 何时应该等待（如人员通过）。

#### Reward 设计建议

```python
reward = (
    1.0 * task_success_reward +
    -0.01 * time_penalty +
    -0.1 * invalid_search_penalty +
    -0.5 * collision_penalty +
    0.2 * efficient_search_bonus +
    -0.3 * human_disturbance_penalty
)
```

#### 训练策略

- 先在 27B + 仿真环境中用 RL 探索策略上限。
- 将 27B 的 RL 策略蒸馏到 4B，作为 4B RL 微调的初始化。
- 再在 4B 上用小学习率 RL + 真实世界数据微调，适配部署时延。
- 保留监督学习的基础能力，RL 只做增量优化，避免能力退化。

#### 设计收益与代价

主要收益：
- 搜索效率提升 10-20%（预估）。
- 无效搜索次数减少 15-25%（预估）。
- 局部避障更平滑。

主要代价：
- Reward 设计难，容易出现 reward hacking。
- 真实世界数据成本高（需要数百小时机器人交互）。
- 仿真到真实迁移存在 sim-to-real gap。
- 可能引入不稳定性（RL 策略在边界情况下可能崩溃）。

### 5.7 T6：个性化 / 行为长期记忆接入（P2 阶段）

T6 处理个性化 / 行为长期记忆能力，这是难度最大、风险最高的模块。

说明：基础空间长期记忆（`room graph`、`target belief map`、地图漂移提示等）不在这一阶段才开始，而是在 `T1-T4` 已作为 `NFM` 主能力建设。

#### 个性化 / 行为长期记忆的类型

**物品位置先验**
- 记录家庭中物品的常驻位置。
- 例如：`遥控器通常在客厅茶几上`。
- 更新频率：每次成功找到物品后更新。

**人员活动区域先验**
- 记录家庭成员的活动区域偏好。
- 例如：`张先生晚上通常在书房`。
- 更新频率：每次成功找到人后更新。

**搜索失败历史**
- 记录哪些区域搜索过但没找到。
- 用于抑制重复搜索。
- 更新频率：每次搜索失败后更新。

#### 技术方案选择

**方案 1：显式记忆库 + 模型读取（推荐）**

示例：

```python
# 记忆库结构示例
memory_db = {
    "items": {
        "遥控器": {
            "frequent_locations": [
                {"location": "客厅-茶几", "confidence": 0.85, "last_seen": "2026-03-20"},
                {"location": "卧室-床头柜", "confidence": 0.62, "last_seen": "2026-03-15"}
            ]
        }
    },
    "persons": {
        "张先生": {
            "frequent_areas": [
                {"area": "书房", "time_range": "19:00-22:00", "confidence": 0.78}
            ]
        }
    },
    "search_history": {
        "2026-03-23": [
            {"target": "遥控器", "searched": ["客厅-茶几", "卧室"], "found": false}
        ]
    }
}
```

优点：
- 可解释性强，可以查看和修正记忆。
- 记忆新鲜度可控（可设置过期时间）。
- 不会污染模型参数。

缺点：
- 需要额外的记忆管理模块。
- 模型需要学会读取和利用记忆。

**方案 2：参数化记忆（不推荐）**
- 把长期先验直接训练进模型参数。
- 优点：端到端。
- 缺点：错误记忆难以修正，泛化性差。

#### 记忆更新策略

**置信度衰减**
- 记忆随时间衰减：`confidence_new = confidence_old * decay_factor^days`。
- 避免过时记忆持续影响决策。

**冲突解决**
- 如果新观察与记忆冲突，优先相信新观察。
- 但记录冲突次数，多次冲突后删除记忆。

**记忆清理**
- 定期清理低置信度记忆（< 0.3）。
- 定期清理过期记忆（> 30 天未更新）。

#### 训练策略

- 先在仿真环境中验证记忆机制。
- 再用真实家庭数据测试。
- 重点关注记忆的准确性和新鲜度。

#### 设计收益与代价

主要收益：
- 长期任务成功率提升 15-30%（预估）。
- 搜索效率提升 20-40%（预估）。
- 用户体验提升（机器人"记住"了家庭习惯）。

主要代价：
- 最容易积累脏数据，需要持续监控和清理。
- 评估周期长（需要数周到数月）。
- 隐私敏感（需要妥善处理家庭数据）。
- 依赖前面所有能力的可靠性。

## 6. 运行时设计与频率分层

这里的频率指语义导航策略层的运行频率，不是底盘控制频率。

### 6.1 频率分层原则

不同任务的目标变化速度不同，因此模型运行频率必须分层：

- 房间级导航：目标变化慢，更多依赖地图和区域级判断。
- 家具级找物：目标更细，需要更频繁刷新视觉与候选区域。
- 人物级找人：目标动态性更强，需要更高频更新。
- 连续动态任务：例如持续跟随或共享空间移动，需要更高频的局部策略调制，但不应每次都跑完整 VLM。

### 6.2 推荐频率与设计权衡

**重要说明**：本节描述的是**Research Teacher模型**的能力上限探索目标，不代表端侧4B Student的部署频率。

**Teacher模型（27B级，Research Bench）频率目标：**

| 任务级别 | Teacher目标频率 | 单次token预算 | 每秒token吞吐 | 设计目标 |
| --- | --- | --- | --- | --- |
| 房间级导航 | 0.5-1 Hz | 450-650 | 225-650 | 探索更丰富的地图理解能力上限 |
| 家具级找物 | 1-2 Hz | 600-850 | 600-1700 | 平衡候选区域刷新频率与token预算 |
| 人物级找人 | 2-3 Hz | 650-900 | 1300-2700 | 探索连续跟踪的能力上限 |
| 连续动态任务 | 4-5 Hz 轻量层 | 120-260 | 480-1300 | 双层架构：低频主策略+高频轻量层 |

**Student模型（4B级，端侧部署）稳态频率：**

| 任务级别 | Student稳态频率 | 说明 |
| --- | --- | --- |
| 房间级 | 0.25-0.40 Hz | 默认0.33Hz，语义决策层 |
| 家具级 | 0.20-0.33 Hz | 默认0.25Hz，进入候选区域后短时升频 |
| 人物级 | 0.25-0.33 Hz | 默认0.25Hz，连续动态由tracker/local planner承接 |

**关键设计原则**：
- Teacher可用更高频率探索能力上限，为蒸馏提供高质量数据
- Student在端侧只负责低频语义决策，连续动态部分交给轻量跟踪器和局部规划器
- 详细预算分析见`vln_4b_tps_and_bandwidth_budget.md`

### 6.3 运行时结构建议

建议采用"双层频率"设计，而不是让同一个模型统一承担所有频率：

- 低频主策略层：负责房间级、家具级、人物级的高层语义判断。
- 高频轻量层：负责跟踪、近障约束、局部谨慎提示、短周期更新。

这样设计的原因是，如果完整多模态主模型也按 4-5 Hz 持续跑，多视角输入会让 token 吞吐和延迟都快速失控。

### 6.4 TPS 需求分析

**关键概念区分：**
- **总 TPS**：模型每秒处理的总 token 数（输入 + 输出），影响整体吞吐量。
- **Decode TPS**：模型每秒生成的输出 token 数，影响输出生成延迟。
- **输入 token 吞吐**：每秒送入模型的输入 token 数（频率 × 单次输入）。

**4B 模型 INT8 量化的 Decode TPS 参考：**
- 移动端芯片（高通 8 Gen 3）：~400-600 tokens/s
- 边缘计算芯片（Jetson Orin NX）：~600-900 tokens/s
- 边缘计算芯片（Jetson Orin AGX）：~1000-1500 tokens/s
- 桌面 GPU（RTX 4060）：~1500-2000 tokens/s

**不同任务级别的 Decode TPS 需求（基于压缩输出格式）：**

| 任务级别 | 频率 | 输出 tokens | 时延预算 | Decode 预算 | 所需 Decode TPS |
|---------|------|------------|---------|------------|----------------|
| 房间级导航 | 0.5 Hz | 20-30 | 2000ms | 1500ms | 13-20 tokens/s |
| 家具级找物 | 1 Hz | 30-40 | 1000ms | 600ms | 50-67 tokens/s |
| 人物级找人 | 2 Hz | 30-50 | 500ms | 200ms | 150-250 tokens/s |

**关键发现：**
- 使用压缩输出格式后，所有任务级别的 Decode TPS 需求都很低（< 250 tokens/s）。
- 所有主流硬件平台都能满足 Decode TPS 需求。
- 真正的瓶颈是 Prefill 阶段处理大量输入 token（540-700 tokens）的延迟。
- 在极低 TPS 硬件（总 TPS < 100）上，必须将总 token 压缩到 200 以内才能跑 0.5 Hz。

**硬件选型建议：**

**Research Bench（研究验证阶段）：**
- Jetson Orin AGX 或 RTX 4060+：用于teacher模型训练、能力上限探索和离线蒸馏验证
- 时延裕度充足，适合快速迭代

**产品化平台（对齐PDCP基线）：**
- 主线：地瓜 S100 Pro（128 TOPS）
- 备线：RK3588 + RK18xx类协处理器
- 评估标准：在真实3-5相机与热稳态下按统一TTFT/TPS口径验证
- 参考：`docs/03_p2_feasibility/04_hardware_software_selection_matrix.md`
- 移动端部署：需要压缩输出格式 + 优化输入（单图 + 文本）。
- 极低端硬件（总 TPS < 100）：不建议部署多模态 VLN，考虑 2B 模型或纯文本方案。

## 7. 多视角输入下的 token 预算设计（Teacher/Student 分层）

这一部分按"输入组成 + 运行频率 + 每秒 token 吞吐"进行估算，**明确区分 Research Teacher 和 PDCP Student 两套契约**。

### 7.1 估算前提与分层原则

#### Research Teacher Contract（能力探索）

- **输入配置**：可使用更丰富传感器（多视角、深度真值、更高分辨率）
- **目标**：探索能力上限，不受端侧约束限制
- **token 预算**：相对宽松，优先保证能力完整性
- **平台**：研究 bench（如 RTX 4090、Jetson Orin AGX）

#### PDCP Student Contract（产品部署）

- **硬件基线**：量产配置为 1组双目+2个单目（共3-5相机）
- **单次推理输入**：从量产相机集合中动态选择 2 个视角进入模型（主视角+辅助视角各1张关键帧）
- **其余相机用途**：服务于 tracking / detection / world state update / event trigger，不等于每次都进入 VLM
- **参考**：`docs/03_p2_feasibility/04_hardware_software_selection_matrix.md`
- **目标**：在端侧约束下通过蒸馏保留核心能力
- **token 预算**：严格控制，避免同时输入多张历史帧或完整地图
- **平台**：S100 Pro 主线 / RK 备线

#### 蒸馏对齐策略

- Teacher 使用丰富输入训练，输出高质量判断
- Student 使用量产输入子集，通过蒸馏学习判断能力
- 输入差异通过特征对齐和输出对齐蒸馏弥合

### 7.2 Research Teacher 单次推理 token 组成（能力探索配置）

在多视角研究配置下，单次推理的 token 组成建议如下：

- 主视角相机 1 张关键帧：约 160-220 token
- 辅助视角相机（可多路）：每路约 160-220 token
- 深度图或几何真值（可选）：约 100-150 token
- 可选局部 ROI：约 80-120 token
- 局部地图裁剪与拓扑摘要：约 100-160 token
- 指令、任务约束、语义锚点、匹配假设、短历史摘要：约 80-140 token

**总计**：约 600-1000+ token（根据研究需求灵活配置）

### 7.3 PDCP Student 单次推理 token 组成（量产部署配置）

在量产相机配置下（1组双目+2个单目），单次推理的 token 组成建议如下：

- 主视角相机 1 张关键帧：约 160-220 token
- 辅助视角相机 1 张关键帧（从量产配置中选择）：约 160-220 token
- 可选 1 个局部 ROI：约 80-120 token
- 房间拓扑、候选区域与局部地图摘要：约 100-160 token
- 指令、任务约束、语义锚点、局部执行上下文、短历史摘要：约 80-140 token

**总计**：
- 不带 ROI 的基础单次推理：约 500-740 token
- 带 1 个 ROI 的精细推理：约 580-860 token

### 7.4 不同任务级别的单次 token 预算（Student 部署配置）

| 任务级别 | 单次输入建议 | 单次 token 目标 |
| --- | --- | --- |
| 房间级导航 | 主+辅视角关键帧 + 房间拓扑摘要 + 语义锚点 + 简短任务状态 | 450-650 |
| 家具级找物 | 主+辅视角关键帧 + 1 个 ROI + 候选家具摘要 + 语义锚点 + 近几步观察摘要 | 600-850 |
| 人物级找人 | 主+辅视角关键帧 + 1 个 ROI + 人物状态摘要 + 语义锚点 + 候选区域摘要 | 650-900 |
| 连续动态任务的轻量策略层 | 不再输入完整多图，只输入跟踪结果、局部风险摘要和短状态 | 120-260 |

### 7.5 按运行频率推算每秒 token 吞吐（Student 部署配置）

#### 房间级导航
- 单次 450-650 token。
- 频率 0.5-1 Hz。
- 每秒约 225-650 token。

#### 家具级找物
- 单次 600-850 token。
- 频率 1-2 Hz。
- 每秒约 600-1700 token。

#### 人物级找人
- 单次 650-900 token。
- 频率 2-3 Hz。
- 每秒约 1300-2700 token。

#### 连续动态任务
如果仍在 4-5 Hz 下运行完整双相机多模态输入：
- 即便按 650 token 计算。
- 每秒也会达到 2600-3250 token。

因此，更合理的做法是：
- 高频层只输入跟踪框、相对位置、局部风险和短状态摘要。
- 高频层单次压到 120-260 token。
- 在 4-5 Hz 下，每秒约 480-1300 token。

### 7.5 token 设计约束（Student 部署配置）

基于量产相机配置和当前任务频率，建议采用以下约束：

- 主力 4B 模型的常规单次输入，尽量压在 500-850 token。
- 只有少量精细场景才允许接近 900 token。
- 一旦任务需要 4-5 Hz，就必须切换到轻量策略层输入，不能继续送完整多图。
- 多视角增益应优先通过视角选择和结构化摘要获得，而不是通过无限增加图像 token 获得。

## 8. 数据建设建议

### 8.1 数据应按能力层组织

建议把数据拆成以下几类：

- 空间理解数据。
- 地图理解数据。
- 搜索与恢复数据。
- 多视角一致性数据。
- 深度几何与通行性数据。
- 长期记忆数据。

其中长期记忆数据应独立建设，不要过早混入主线训练。

### 8.2 teacher 阶段重点数据

teacher 阶段应重点覆盖：

- 房间识别和房间属性。
- 家具、支撑面和门口通道语义。
- 视角到地图的对齐关系。
- 目标候选区域排序。
- 搜索顺序和恢复策略。

### 8.3 student 蒸馏阶段重点数据

student 阶段应重点保留：

- teacher 产生的结构化判断结果。
- teacher 对难例的区域排序和恢复选择。
- 少量真实部署闭环样本。
- 双相机场景下的视角切换和确认样本。

## 9. 评测与退出条件

### 9.1 teacher 阶段重点评测

- 房间级判断是否稳定。
- 家具级判断是否稳定。
- 地图对齐是否稳定。
- 候选区域召回是否足够。
- 搜索顺序和恢复建议是否一致。

### 9.2 student 阶段重点评测

- 相对 teacher 的主线能力保留率。
- 单次推理时延。
- 房间级和家具级任务闭环稳定性。
- 找人任务的频率达成情况。
- 多视角输入下的稳定性。

### 9.3 后续阶段重点评测

- 多视角一致性是否真的提升。
- 深度几何是否真实降低误判。
- 强化学习是否提升效率而非引入不稳定性。
- 长期记忆是否提升命中率而不是积累脏先验。

### 9.4 统一退出条件

如果某一阶段引入的新能力明显破坏了上一阶段主线能力，应优先回退并修正，而不是继续叠加更多能力。

## 10. 风险控制与应对策略

### 10.1 主要风险点

**R1：语义全局帧与局部执行帧不统一**
- 风险：空间理解学不稳定，粗粒度定位和局部执行出现割裂。
- 应对：T0 阶段必须明确 `semantic_global_frame / local_metric_frame / anchor_alignment` 规范，统一坐标系、语义锚点和漂移告警机制。

**R2：teacher 到 student 蒸馏效果差**
- 风险：4B 模型能力塌缩。
- 应对：确保 teacher 和 student 输入一致，增加难例蒸馏权重。

**R3：多视角输入 token 控制失败**
- 风险：推理时延失控，无法部署。
- 应对：严格执行 token 预算，必要时降低图像分辨率或采用结构化摘要。

**R4：找人任务频率要求无法满足**
- 风险：2-3 Hz 频率下 token 吞吐过高。
- 应对：设计双层频率架构，高频层使用轻量输入。

**R5：个性化长期记忆积累脏数据**
- 风险：错误习惯先验持续影响决策。
- 应对：采用显式记忆库，设置置信度衰减和定期清理机制；基础空间长期记忆与个性化记忆分库治理。

### 10.2 退出条件

如果某一阶段出现以下情况，应考虑回退或调整方案：

- 新能力明显破坏上一阶段主线能力（性能下降 > 10%）。
- 推理时延超出约束 50% 以上。
- 训练 3 个月仍无明显进展。
- 真实场景测试失败率 > 40%。

## 11. 总结

### 11.1 方案核心要点

1. **最终目标 = 4B 产品线；当前主策略 = 27B `NFM` teacher 先行**：27B teacher 探索能力上限，4B student 通过蒸馏获得能力子集并负责部署。
2. 能力分层：P0（空间+地图+基础空间长期记忆+粗粒度定位）→ P1（多视角+几何增强+世界状态写回+RL）→ P2（个性化长期记忆）。
3. 不输出动作：聚焦语义判断、语义锚点和世界状态写回，不建设端到端控制。
4. Teacher 驱动：每项能力在 27B 上稳定后，再评估是否可蒸馏到 4B。
5. 双层频率：低频主策略层 + 高频轻量层。

### 11.2 预期收益

- P0 阶段：建立稳定的空间理解、地图理解、语义锚点和基础空间长期记忆基础。
- P1 阶段：多视角、几何增强、世界状态写回与 RL 逐步提升鲁棒性和效率。
- P2 阶段：个性化长期记忆带来用户体验质变。

### 11.3 关键成功因素

1. T0 阶段的接口定义必须清晰。
2. Teacher 能力探索是主驱动，4B 是蒸馏与产品验证门。
3. Teacher 和 Student 的输入配置分别设计：Teacher 可用更丰富传感器，Student 对齐量产基线。
4. 能力分阶段接入，避免同时混训。
5. 基础空间长期记忆必须前置进入 `P0/P1`，个性化长期记忆才排在最后，避免过早污染。
