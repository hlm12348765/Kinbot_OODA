# VLN 4B 模型 TPS、带宽与频率预算说明

文档时间：2026-03-24  
文档性质：专项预算文档  
适用范围：Kinbot 端侧 4B 级 VLN 模型的输入吞吐、输出吞吐、带宽与运行频率设计

---

## 1. 文档目的

本文档单独回答以下问题：

1. 端侧 4B 级 VLN 模型的 TPS 应该如何定义。
2. 在两路相机条件下，输入吞吐、输出吞吐和总吞吐应如何分别核算。
3. 在总 TPS 很难稳定超过两位数的前提下，房间级、家具级、人物/小尺度物品级任务的运行频率应如何设置。
4. 本地记忆库优先、按需装载上下文，是否能够有效降低主模型频率与上下文长度。
5. 带宽约束应如何理解，哪些是真正的瓶颈，哪些不是。

本文档的重点不是给出理想状态下的高频方案，而是给出一版面向端侧小尺寸模型、可落地、可收敛的稳态预算方案。

---

## 2. 术语与计算口径

### 2.1 TPS 的三种口径

为避免歧义，本文档将 TPS 分成三类：

- 输入 token 吞吐：每秒送入模型的 token 数。
- 输出 token 吞吐：每秒由模型生成的 token 数。
- 总 token 吞吐：输入 token 吞吐与输出 token 吞吐之和。

对应公式：

```text
输入 TPS = 单次输入 token × 运行频率
输出 TPS = 单次输出 token × 运行频率
总 TPS = 输入 TPS + 输出 TPS
```

### 2.2 稳态与峰值

本文档同时区分两种预算：

- 稳态预算：模型可以长时间持续运行的平均吞吐。
- 峰值预算：事件触发时，允许短时间高于稳态预算的瞬时吞吐。

对于端侧 4B 模型，真正重要的是稳态预算，而不是短时间峰值。

### 2.3 本文档的默认目标

在未锁定具体芯片前，建议按以下口径约束：

- 稳态总 TPS 目标：优先控制在 `60-90 token/s`。
- 稳态总 TPS 上限：尽量不超过 `100 token/s`。
- 峰值总 TPS：允许短时到 `120-180 token/s`，但不能持续。

如果后续板级实测证明可以稳定支撑更高吞吐，再向上调整频率；在没有证据前，不应反过来先把频率设计得过高。

---

## 3. 本代 4B 模型的运行前提

### 3.1 任务边界

当前代 4B 模型只承担低频语义策略层，不承担以下高频任务：

- 底盘控制闭环。
- 高频避障闭环。
- 连续动态跟随中的逐帧策略刷新。

连续动态任务（`> 3 Hz`）不放入当前这一代 4B 主模型主线。

### 3.2 两路相机输入原则

两路相机并不等于两路原始图像都持续送入 4B 模型。

默认原则如下：

- 主相机关键帧是主输入。
- 辅相机默认只提供摘要信号，不常开原始图像输入。
- 只有在遮挡确认、找人丢失、视角冲突时，才按需触发辅相机原图。

### 3.3 本地记忆库优先原则

系统不应在每次推理时把完整地图、完整历史、完整候选区域全部塞入上下文。

更合理的流程是：

1. 先查本地记忆库。
2. 只取与当前任务最相关的少量记忆项。
3. 将其压缩成短摘要后，再装载进 context。
4. 若本地记忆已经足够支持当前阶段决策，则主模型不需要高频重复推理。

本地记忆库中可优先维护的信息包括：

- 当前所在房间与相邻房间拓扑。
- 已搜索区域与已排除区域。
- 家具候选列表与优先级。
- 目标最近一次出现区域。
- 人员或物品的本地先验位置。
- 当前搜索阶段与失败恢复状态。

---

## 4. 本地记忆库如何降低频率

### 4.1 降频机制

本地记忆库优先，会同时降低两项压力：

- 降低单次输入 token。
- 降低主模型触发频率。

原因如下：

- 房间拓扑、候选区域、已搜索历史不会在每一秒都发生根本变化。
- 如果这些信息已经保存在本地记忆库中，就没有必要每个周期都重新让模型从头理解一遍。
- 主模型只需要在“状态发生变化”或者“记忆无法覆盖当前判断”时被唤起。

### 4.2 推荐触发方式

推荐采用“低频周期 + 事件触发”的混合策略，而不是全程固定频率：

- 低频周期触发：用于定期校正高层语义判断。
- 事件触发：用于在关键变化发生时立即补充推理。

建议的事件触发条件包括：

- 进入新房间。
- 接近门口或区域边界。
- 搜索失败，需要切换候选区域。
- 跟踪器失去目标。
- 辅相机与主相机产生明显分歧。
- 本地记忆置信度下降到阈值以下。

### 4.3 频率设计结论

本地记忆库的意义不是把所有问题都丢给缓存，而是让 4B 模型不必承担本来就没必要的高频重复判断。

因此，当前代频率设计不应先从“模型能否跑到 1-2 Hz”出发，而应先从“在什么场景下真的需要再次调用模型”出发。

## 5. 单次 token 预算重算

下表不是理想化大模型预算，而是按“本地记忆库优先 + 辅相机摘要化 + 低频语义层”后的 4B 预算重算。

### 5.1 房间级任务

房间级任务的典型输出包括：

- 当前所在房间判断。
- 下一步应进入哪个房间。
- 搜索失败后应切换到哪个房间。

建议单次 token 构成：

| 组成项 | token 估计 |
| --- | --- |
| 主相机关键帧 | 90-120 |
| 房间拓扑与局部记忆摘要 | 12-20 |
| 任务状态与位姿 | 12-18 |
| 已搜索/已排除摘要 | 8-15 |
| 输出结果 | 20-30 |
| 总计 | 145-200 |

### 5.2 家具级任务

家具级任务的典型输出包括：

- 当前房间内的候选家具排序。
- 下一步应靠近哪个家具或支撑面。
- 当前区域搜索失败后的恢复建议。

建议单次 token 构成：

| 组成项 | token 估计 |
| --- | --- |
| 主相机关键帧 | 90-120 |
| 家具候选与局部记忆摘要 | 18-30 |
| 任务状态、位姿、搜索阶段 | 15-25 |
| 局部裁剪或观察补充信息 | 20-35 |
| 输出结果 | 25-40 |
| 总计 | 170-250 |

### 5.3 人物/小尺度物品级任务

人物级和小尺度物品级任务的典型输出包括：

- 当前目标是否仍在候选区域内。
- 下一步应切向哪个观察位或候选区域。
- 是否需要触发辅相机确认或重新定位。

建议单次 token 构成：

| 组成项 | token 估计 |
| --- | --- |
| 主相机关键帧 | 90-120 |
| 人物/目标状态与本地记忆摘要 | 20-35 |
| 任务状态与位姿 | 15-25 |
| 辅相机摘要 | 10-20 |
| 重定位或小目标补充信息 | 25-45 |
| 输出结果 | 30-50 |
| 总计 | 190-300 |

### 5.4 双视角确认模式

双视角确认模式不是稳态模式，只允许短时触发：

| 组成项 | token 估计 |
| --- | --- |
| 主相机关键帧 | 90-120 |
| 辅相机关键帧 | 90-120 |
| 局部记忆与任务状态 | 25-40 |
| 输出结果 | 35-55 |
| 总计 | 240-335 |

如果再增加 ROI，则可到 `300-380 token/次`，不应作为稳态配置。

### 5.5 输入、输出与总 token 汇总

| 任务级别 | 单次输入 token | 单次输出 token | 单次总 token |
| --- | --- | --- | --- |
| 房间级 | 120-170 | 20-30 | 145-200 |
| 家具级 | 145-210 | 25-40 | 170-250 |
| 人物/小尺度物品级 | 160-245 | 30-50 | 190-300 |
| 双视角确认 | 205-280 | 35-55 | 240-335 |

## 6. 在总 TPS < 100 前提下的频率反推

### 6.1 计算公式

在总 TPS 预算给定时，可直接反推最大频率：

```text
最大频率 = 总 TPS 预算 / 单次总 token
```

### 6.2 按 `总 TPS = 80` 与 `总 TPS = 100` 反推

| 任务级别 | 单次总 token | `TPS=80` 时最大频率 | `TPS=100` 时最大频率 |
| --- | --- | --- | --- |
| 房间级 | 145-200 | 0.40-0.55 Hz | 0.50-0.69 Hz |
| 家具级 | 170-250 | 0.32-0.47 Hz | 0.40-0.59 Hz |
| 人物/小尺度物品级 | 190-300 | 0.27-0.42 Hz | 0.33-0.53 Hz |
| 双视角确认 | 240-335 | 不适合作为稳态频率 | 不适合作为稳态频率 |

这张表意味着：

- 房间级在 `总 TPS < 100` 下，`0.5 Hz` 只是上限参考，而不是默认值。
- 家具级如果坚持 `1 Hz`，则总 TPS 会显著超预算。
- 人物/小尺度物品级如果坚持 `1-2 Hz`，在 4B 小模型上基本不可持续。

### 6.3 推荐稳态频率

在端侧 4B 小模型上，建议采用以下稳态频率：

| 任务级别 | 推荐稳态频率 | 说明 |
| --- | --- | --- |
| 房间级 | 0.25-0.40 Hz | 推荐默认 `0.33 Hz`，进入新房间或失败恢复时再触发补充推理 |
| 家具级 | 0.20-0.33 Hz | 推荐默认 `0.25 Hz`，进入候选房间后才短时升频 |
| 人物/小尺度物品级 | 0.25-0.33 Hz | 推荐默认 `0.25 Hz`，中间由跟踪器和本地记忆承接 |
| 连续动态任务 | 不纳入当前代 4B 主线 | 交给轻量跟踪、规则或后续代方案 |

### 6.4 默认稳态频率下的输入/输出/总 TPS

按上一节的默认频率估算：

| 任务级别 | 默认频率 | 输入 TPS | 输出 TPS | 总 TPS |
| --- | --- | --- | --- | --- |
| 房间级 | `0.33 Hz` | 40-56 | 7-10 | 48-66 |
| 家具级 | `0.25 Hz` | 36-53 | 6-10 | 43-63 |
| 人物/小尺度物品级 | `0.25 Hz` | 40-61 | 8-13 | 48-75 |

这张表的意义是：

- 在默认稳态频率下，三类主任务都可以压在两位数 TPS 区间内。
- 真正需要额外预算的不是稳态运行，而是短时间事件触发。
- 因此频率设计要把“平均值”和“峰值”分开，而不是只盯一个瞬时数值。

### 6.5 允许的短时突发频率

可以允许短时间升频，但必须限制占空比：

- 房间级：可短时到 `0.5 Hz`。
- 家具级：可短时到 `0.5 Hz`。
- 人物/小尺度物品级：可短时到 `0.8-1.0 Hz`，但仅用于丢失重定位、遮挡确认、目标重现等时刻。

原则上，短时突发不应持续超过 `2-3 秒`，也不应频繁重复出现。

## 7. 平均频率为什么可以低于表面直觉

### 7.1 房间级任务不是高动态任务

房间级任务的变化速度本来就慢：

- 房间拓扑不会频繁变化。
- 机器人不会在 1 秒内多次跨房间跳变。
- 区域级候选排序不需要每秒重算一次。

因此，房间级任务更适合：

- `0.25-0.40 Hz` 的周期刷新。
- 叠加事件触发刷新。

而不是固定 `0.5 Hz` 或更高频率持续运行。

### 7.2 家具级任务可以依赖局部记忆维持阶段稳定

家具级任务看起来比房间级更细，但并不意味着必须每秒都重新做完整语义推理。

只要本地记忆中已经保存：

- 当前房间候选家具列表。
- 已观察过的家具区域。
- 当前最优候选支撑面。

那么模型只需要在以下时刻重新推理：

- 进入新的候选家具群。
- 当前候选失败。
- 观测到新的显著证据。

### 7.3 人物/小尺度物品任务要靠“跟踪器 + 低频语义层”配合

人物和小尺度物品任务不能简单理解成“目标在动，所以主模型就必须高频运行”。

更合理的拆分是：

- 高频部分交给检测器、跟踪器、运动估计或轻量模块。
- 低频部分交给 4B 语义模型做区域判断、重定位与恢复策略。

这样才能把 4B 主模型的平均频率压到 `0.25-0.33 Hz`，同时保留短时间的重新确认能力。

## 8. 峰值与平均值的分离设计

### 8.1 推荐计算方式

系统设计不应只看瞬时频率，而应看平均占空比。

推荐公式：

```text
平均总 TPS = Σ(单次总 token × 触发频率 × 占空比)
```

### 8.2 人物任务示例

示例：

- 稳态模式：`240 token/次 × 0.25 Hz = 60 TPS`
- 丢失重定位突发：`320 token/次 × 1.0 Hz`
- 若突发只持续 `2 秒 / 20 秒`，则平均占空比为 `10%`

则突发部分折算到平均吞吐后约为：

```text
320 × 1.0 × 10% = 32 TPS
```

因此平均总 TPS 约为：

```text
60 + 32 = 92 TPS
```

这说明：

- 即使局部时刻需要到 `1 Hz`，只要占空比受控，平均总 TPS 仍可压在 `100` 以内。
- 问题不在于是否允许短时升频，而在于是否把高频设计成了常态。

## 9. 带宽预算说明

带宽至少有三种含义，不能混为一谈。

### 9.1 相机原始数据带宽

如果两路相机按 `1280×720 RGB`、`10 fps` 采集：

```text
单帧大小 ≈ 1280 × 720 × 3 byte ≈ 2.64 MB
单路相机 ≈ 2.64 MB × 10 ≈ 26.4 MB/s
双路相机 ≈ 52.8 MB/s
```

如果到 `15 fps`，则双路可到约 `79.2 MB/s`。

结论：

- 原始相机带宽不低。
- 但这部分主要由相机采集、前处理、检测与跟踪链路承担。
- 不应把这部分原始带宽直接等同于 4B 模型的 context 带宽。

### 9.2 模型入口图像带宽

如果只向 4B 模型发送 `336×336 RGB` 关键帧：

```text
单帧大小 ≈ 336 × 336 × 3 byte ≈ 0.34 MB
```

则：

- 单主视角、`0.33 Hz`：约 `0.11 MB/s`
- 单主视角、`0.5 Hz`：约 `0.17 MB/s`
- 双视角、`1 Hz` 短时确认：约 `0.68 MB/s`

结论：

- 经过关键帧选择后，送入模型的图像带宽并不大。
- 真正限制 4B 模型的，通常不是入口图像字节带宽，而是模型自身推理过程中的内存访问与 token 处理能力。

### 9.3 本地记忆检索带宽

本地记忆库检索后装载到 context 的通常是短摘要：

- 房间拓扑摘要。
- 候选区域摘要。
- 最近一次观察结果。
- 搜索失败与恢复状态。

这类摘要通常只有数十 token 到百余 token，折算成字节后远低于图像输入。

因此：

- 本地记忆库本身不会明显增加 I/O 带宽压力。
- 反而会减少因为重复装载长历史与大地图带来的上下文冗余。

### 9.4 真正需要关注的带宽

在端侧 4B 模型场景下，更关键的其实是两类“隐性带宽”：

- 模型权重与中间状态访问带来的内存带宽压力。
- 高频调用导致的反复 prefill 开销。

由于当前尚未锁定具体芯片、推理框架和量化策略，本文档不直接给出绝对的 `GB/s` 指标，而以总 TPS 作为更稳定的先验代理指标。

这意味着：

- 在板级实测前，频率预算应优先保守。
- 先控制调用次数和单次 context 长度，再谈更高频。

## 10. 端侧 4B 模型的建议运行策略

### 10.1 运行主线

建议的主线是：

1. 本地记忆库先吸收稳定信息。
2. 4B 模型只读取少量与当前任务直接相关的摘要。
3. 采用低频主策略层。
4. 通过事件触发处理真正需要补充推理的时刻。
5. 连续动态刷新交给轻量模块，而不是强迫 4B 主模型持续高频运行。

### 10.2 推荐参数

在当前阶段，建议先按以下参数做文档主线：

| 项目 | 建议值 |
| --- | --- |
| 稳态总 TPS | `60-90 token/s` |
| 稳态上限 | `100 token/s` |
| 房间级默认频率 | `0.33 Hz` |
| 家具级默认频率 | `0.25 Hz` |
| 人物/小尺度物品级默认频率 | `0.25 Hz` |
| 人物/小尺度物品突发频率 | `0.8-1.0 Hz`，但仅限短时 |
| 双视角原图 | 仅事件触发 |
| 本地记忆装载 | 默认开启，且严格摘要化 |

### 10.3 设计原则

这一版频率设计的核心不是把模型跑快，而是先把平均吞吐压到端侧可承受范围内，再通过本地记忆、事件触发和轻量模块补足响应性。

---

## 11. 结论

1. 对端侧 4B 小模型而言，TPS 预算必须区分输入、输出与总吞吐，不能只说一个模糊的 TPS。
2. 如果总 TPS 很难稳定超过两位数，那么主模型的稳态频率就必须明显低于此前按大模型思路给出的 `0.5-2 Hz`。
3. 本地记忆库优先、按需装载 context，确实可以同时降低单次 token 和平均频率，是当前代可行性的关键前提。
4. 房间级任务推荐按 `0.25-0.40 Hz` 设计，家具级和人物/小尺度物品级推荐按 `0.20-0.33 Hz` 设计，再配合事件触发补充推理。
5. 真正要避免的是把高频设计成常态；短时间升频可以存在，但必须受控，且不能替代稳态预算设计。
