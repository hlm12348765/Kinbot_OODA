## 核查方法

本文档对照 VLN 架构文档中提出的五层能力架构，逐项核查现有代码实现情况：

- ✅ **已实现**：功能已在代码中实现
- ⚠️ **部分实现**：功能有基础实现，但不完整或不符合架构要求
- ❌ **未实现**：功能在代码中缺失
- 📝 **需补充**：功能实现存在，但需要补充或重构

## 第 1 层：语义理解层

### 架构要求（文档第 4.1 节）

**输入**：
- 用户指令
- 指代消解结果
- 身份/权限上下文
- 任务上下文

**输出**：
- `goal_type`
- `target_entity`
- `constraints`
- `success_condition`

### 现有实现核查

#### ✅ 用户指令解析
- **实现**：通过 `user_prompt` 字段接收用户指令
- **代码示例**（原型实现参考）：
  ```python
  self.user_prompt = message_data.user_prompt  # 找人场景
  self.user_prompt = message_data.params['obj_name']  # 找物场景
  ```

#### ❌ 指代消解
- **状态**：未实现
- **缺失**：没有处理"去找我的眼镜"中的"我"，或"去找她"中的"她"的逻辑

#### ❌ 身份/权限上下文
- **状态**：未实现
- **缺失**：没有身份验证或权限检查机制

#### ⚠️ 目标类型识别
- **实现**：通过不同的 Session 类隐式实现（找人/找物/跟随人等场景）
- **问题**：目标类型通过类名隐式表达，没有显式的 `goal_type` 枚举

#### ❌ 约束和成功条件
- **状态**：未实现
- **缺失**：没有显式的 `constraints` 和 `success_condition` 字段

### 小结

语义理解层**基础功能部分实现**，但缺少：
1. 指代消解机制
2. 身份/权限上下文
3. 显式的目标类型、约束和成功条件结构

## 第 2 层：语义空间记忆层

### 架构要求（文档第 4.2 节）

**输入**：
- SLAM 地图/位姿
- 房间语义
- 家具语义
- 人/物观测历史
- 高精度重建数据

**输出**：
- 目标候选区域排序
- 家具承载先验
- 人员活动先验
- 地图漂移提示
- 不确定性评分

### 现有实现核查

#### ⚠️ SLAM 地图/位姿
- **实现**：原型实现中通过数据源接口获取 ASM 地图和当前位姿
- **代码模式**：
  ```python
  # 获取目标位置和地图
  map_image, object_position, description = await data_source.get_object_position(prompt)
  # 获取当前位姿
  current_pose = await data_source.get_pose_data()
  ```
- **问题**：只获取当前位姿，没有历史轨迹记忆

#### ❌ 房间语义
- **状态**：未实现
- **缺失**：没有房间识别、房间边界、房间连通关系

#### ❌ 家具语义
- **状态**：未实现
- **缺失**：没有家具识别、家具承载关系

#### ⚠️ 人/物观测历史
- **实现**：原型实现中有基础的人员管理功能
- **代码模式**：
  ```python
  # 通过人员管理器获取目标人员
  person_to_go = person_manager.get_person(person_id)
  ```
- **问题**：只有人员 ID 管理，没有观测历史、活动先验

#### ❌ 目标候选区域排序
- **状态**：未实现
- **缺失**：没有"优先搜索哪个房间"的逻辑

#### ❌ 家具承载先验
- **状态**：未实现
- **缺失**：没有"眼镜通常在床头柜"的先验知识

#### ❌ 人员活动先验
- **状态**：未实现
- **缺失**：没有"张阿姨通常在客厅"的先验知识

#### ❌ 地图漂移提示
- **状态**：未实现
- **缺失**：没有检测定位漂移的机制

#### ❌ 不确定性评分
- **状态**：未实现
- **缺失**：没有对观测结果的置信度评估

### 小结

语义空间记忆层**严重缺失**，只有最基础的地图和位姿获取，缺少：
1. 房间/家具语义理解
2. 观测历史和活动先验
3. 候选区域排序
4. 地图漂移检测
5. 不确定性评估

这是架构文档强调的"核心壁垒"，但现有实现几乎为空。

## 第 3 层：语义导航策略层

### 架构要求（文档第 4.3 节）

**输入**：
- 当前观测
- 目标表示
- 语义空间记忆
- 局部导航状态

**输出**：
- 下一子目标
- 搜索动作
- 观察动作
- 恢复动作

### 现有实现核查

#### ⚠️ 模型推理
- **位置**：`base_navigation_session.py:124-155`, `navigator_session.py:90-115`
- **实现**：使用 Qwen3-VL 模型进行推理
- **代码示例**：
  ```python
  inputs = self.processor.apply_chat_template(chat_history, ...)
  generated_ids = self.model.generate(**inputs, max_new_tokens=1024)
  ```

#### ⚠️ 历史观测管理
- **位置**：`base_navigation_session.py:110-112`, `navigator_session.py:80-83`
- **实现**：维护 RGB、深度图、地图的历史
- **代码示例**：
  ```python
  self.history_rgb: List[Any] = []
  self.history_depth: List[Any] = []
  self.history_map: List[Any] = []  # navigator_session 中有
  ```
- **问题**：只保留最近 2-3 帧，没有长期记忆

#### ⚠️ 动作输出
- **位置**：`base_navigation_session.py:19-43`
- **实现**：输出离散动作（左转、右转、直行、停止）
- **代码示例**：
  ```python
  SYSTEM_PROMPT = '''
  输出格式必须是以下几类中的一类：
  - 左转x度（范围0-180，整数）
  - 右转x度（范围0-180，整数）
  - 直行x米（范围0.00-1.50，精确到2位小数）
  - 停止
  '''
  ```
- **问题**：
  1. **输出的是连续动作，而非语义子目标**
  2. 架构文档建议输出"房间级目标、家具级目标、拓扑节点"，但现有实现直接输出底盘动作
  3. 这与架构文档第 6.1 节的建议矛盾："让模型少做'每一步怎么踩油门'，多做'下一步去哪里看、为什么去那里看'"

#### ❌ 搜索策略
- **状态**：未实现
- **缺失**：没有"优先搜索哪个房间"的策略

#### ❌ 观察动作
- **状态**：未实现
- **缺失**：没有"先转头观察再决策"的主动观察动作

#### ❌ 恢复动作
- **状态**：未实现
- **缺失**：没有"找不到目标时如何恢复"的策略

#### ❌ 目标置信度维护
- **状态**：未实现
- **缺失**：没有维护 `target_belief` 状态

### 小结

语义导航策略层**存在严重架构偏差**：
1. **核心问题**：现有实现输出的是连续底盘动作，而非语义子目标
2. 这与架构文档第 6.1 节的建议直接矛盾
3. 缺少搜索策略、观察动作、恢复动作
4. 没有目标置信度维护

**这是最严重的架构偏差**，需要重构。

## 第 5 层：运动执行层

### 架构要求（文档第 4.5 节）

**输入**：
- 子目标位姿
- corridor / keepout / speed hints
- 当前局部障碍

**输出**：
- 实际底盘控制
- 执行状态
- 卡住原因

### 现有实现核查

#### ✅ 底盘动作执行
- **位置**：`action_exec/` 目录（未详细查看）
- **实现**：通过 `action_exec.turn_angle()` 和 `action_exec.go_forward()` 执行
- **代码示例**：
  ```python
  await action_exec.turn_angle(-angle)  # 左转
  await action_exec.go_forward(distance)  # 直行
  ```

#### ✅ 执行状态反馈
- **位置**：`action_exec/exec_status.py`
- **实现**：返回执行状态（SUCCESS, UNREACHABLE, EXECUTION_ERROR）
- **代码示例**：
  ```python
  if result.status == ExecStatus.SUCCESS:
      record_action = output_text
  elif result.status == ExecStatus.UNREACHABLE:
      record_action = f"{output_text}（实际只完成{result.actual}）"
  ```

#### ❌ 卡住原因诊断
- **状态**：未实现
- **缺失**：没有详细的卡住原因分析

### 小结

运动执行层**基础功能已实现**，但缺少详细的故障诊断。

## 总体缺口总结

### 严重程度分级

#### 🔴 P0 - 架构性偏差（阻断性问题）

1. **第 3 层输出错误**：现有实现输出连续底盘动作，而非语义子目标
   - **影响**：与架构文档第 6.1 节建议直接矛盾
   - **后果**：导致"推理慢、走得犹豫"（文档第 6.1 节提到的痛点）
   - **建议**：必须重构为子目标式输出

2. **第 2 层几乎为空**：语义空间记忆层严重缺失
   - **影响**：无法实现"优先搜索区域排序"、"家具承载先验"等核心能力
   - **后果**：无法建立"家庭语义数据闭环"（文档第 6.2 节强调的核心壁垒）
   - **建议**：这是架构文档强调的核心壁垒，必须补充

#### 🟡 P1 - 功能缺失（重要但非阻断）

3. **第 4 层完全缺失**：社交与自主移动策略层
   - **影响**：无法实现"礼让、让行、低扰动移动"
   - **建议**：根据架构文档第 9 节，这是"长期需要"，可在 P2 阶段补充

4. **指代消解缺失**：无法处理"我的眼镜"、"去找她"
   - **影响**：用户体验受限
   - **建议**：需要补充

5. **恢复策略缺失**：找不到目标时无法恢复
   - **影响**：成功率受限
   - **建议**：需要补充

#### 🟢 P2 - 优化项（可延后）

6. **身份/权限上下文**：未实现
7. **不确定性评估**：未实现
8. **卡住原因诊断**：不够详细

## 具体建议

### 建议 1：立即重构第 3 层输出方式（P0）

**当前问题**：
```python
# 现有实现：直接输出底盘动作
output = "直行0.25米"  # 或 "左转30度"
```

**架构要求**：
```python
# 应该输出语义子目标
output = {
    "navigation_skill": "search_object",
    "semantic_subgoal": "去客厅沙发边桌附近观察",
    "candidate_metric_goals": [(x1, y1, theta1), (x2, y2, theta2)],
    "explanation": "眼镜通常在沙发边桌上"
}
```

**重构路径**：
1. 修改 SYSTEM_PROMPT，要求模型输出语义子目标而非底盘动作
2. 增加子目标到底盘动作的转换层（由局部规划器负责）
3. 这样可以解决"推理慢、走得犹豫"的问题

### 建议 2：建立语义空间记忆层（P0）

**需要补充的模块**：

1. **房间识别模块**
   - 输入：RGB 图像、地图
   - 输出：当前房间类别、房间边界

2. **家具识别模块**
   - 输入：RGB 图像
   - 输出：家具类别、位置

3. **物品-家具关联先验**
   - 数据结构：`{"眼镜": ["床头柜", "沙发边桌"], "遥控器": ["茶几", "沙发"]}`

4. **人员活动先验**
   - 数据结构：`{"张阿姨": {"常驻房间": "客厅", "活动时间": "8:00-22:00"}}`

5. **候选区域排序算法**
   - 输入：目标物品、当前位置、历史观测
   - 输出：优先搜索区域列表

### 建议 3：补充接口设计（P0）

在 `base_navigation_session.py` 中增加：

```python
@dataclass
class NavigationOutput:
    navigation_skill: str  # "search_object" / "search_person" / ...
    semantic_subgoal: str  # "去客厅沙发边桌附近观察"
    candidate_metric_goals: List[Tuple[float, float, float]]
    target_belief: Optional[Dict]  # {"position": (x,y), "confidence": 0.8}
    behavior_hints: Optional[Dict]  # {"max_speed": 0.5, "keep_distance": 1.0}
    need_active_observation: bool
    need_user_confirmation: bool
    recovery_strategy: Optional[str]
    explanation: str
```

### 建议 4：分阶段实施计划

**阶段 1（2周）**：重构第 3 层输出
- 修改 SYSTEM_PROMPT
- 增加子目标解析器
- 测试验证

**阶段 2（4周）**：建立第 2 层基础
- 房间识别
- 家具识别
- 基础先验知识库

**阶段 3（4周）**：完善记忆与恢复
- 候选区域排序
- 恢复策略
- 不确定性评估

**阶段 4（延后）**：社交移动策略
- 根据架构文档第 9 节，这是"长期需要"
- 可在一代产品后期或二代补充

## 结论

现有 VLN 实现与架构文档存在**严重偏差**，主要体现在：

1. **架构性问题**：第 3 层输出方式错误，导致"推理慢、走得犹豫"
2. **核心能力缺失**：第 2 层（语义空间记忆）几乎为空，无法建立"家庭语义数据闭环"
3. **接口不符**：与文档第 5 节的接口设计严重不符

**建议优先级**：
- 🔴 P0：重构第 3 层输出 + 建立第 2 层基础（2-6 周）
- 🟡 P1：补充恢复策略、指代消解（4-6 周）
- 🟢 P2：社交移动策略（延后到一代后期或二代）

**预期收益**：
- 解决"推理慢、走得犹豫"问题
- 建立核心竞争壁垒（家庭语义记忆）
- 符合架构文档的技术方向
